以下是「OpenFang：用 Rust 構建的開源 Agent 作業系統」的語音版本。

2026 年 2 月 24 日，一個名為 OpenFang 的開源專案在 GitHub 上線，4 天內獲得超過 4,000 個星標。它的定位是 Agent 作業系統，不是聊天機器人框架，也不是大型語言模型的封裝層。整個系統用 Rust 從頭撰寫，13 萬 7 千多行程式碼，14 個模組，超過 1,700 個測試，最終編譯成一個約 32 MB 的單一執行檔。

背景與現狀問題

目前主流的 AI Agent 框架，像是 LangChain、AutoGen、CrewAI，幾乎都以 Python 為基礎。這帶來幾個共同問題。

首先是啟動時間。LangGraph 冷啟動大約要 2.5 秒，而 OpenClaw 這個同樣用 Rust 寫的框架需要將近 6 秒。對於需要定時自動運行的代理程式來說，這是實際的效能瓶頸。

其次是記憶體占用。AutoGen 在閒置狀態下大約佔用 250 MB，OpenClaw 則接近 400 MB。如果你想在一台普通的 VPS 上部署多個 Agent 實例，這個數字很快就會撐爆記憶體。

第三個問題是安全設計薄弱。多數框架只做到基本的 API 金鑰保護，對於 Prompt injection、資訊流洩漏、代理身份驗證這些 Agent 特有的威脅，幾乎沒有系統性的防護。

OpenFang 針對這些問題設計了完整的解決方案。

核心架構

整個系統由 14 個 Rust 模組組成，各自負責不同層次的功能。

核心排程器負責工作流編排和資源計量；運行時模組是 Agent 的執行核心，支援三種大型語言模型驅動模式；API 模組提供超過 140 個 REST 端點，包含 OpenAI 相容 API；通訊模組內建 40 個訊息平台適配器；記憶體模組使用 SQLite 做持久化，並支援向量嵌入。

另外還有自訂的 OFP，也就是 OpenFang 協定，一個點對點的通訊協定，讓 Agent 之間可以直接溝通，不依賴中央伺服器。

Hands 系統

OpenFang 最獨特的設計是 Hands，也就是「手」的概念。7 個 Hands 是預先打包的自主能力模組，每個都包含清單設定檔、超過 500 字的多階段系統提示、領域知識文件，以及限制越權操作的護欄。

第一個 Clip，負責下載 YouTube 影片、識別關鍵時段、生成豎版短片並自動加字幕，再發布到 Telegram 或 WhatsApp。

第二個 Lead，每天自動發現符合目標客戶條件的潛在客戶，依評分去重後輸出到 CRM 系統。

第三個 Collector，做的是 OSINT 情報收集，持續監控指定目標的變化並做情緒分析。

第四個 Predictor，是一個超級預測引擎，追蹤 Brier 分數來校準推理鏈的準確性。

第五個 Researcher，跨來源做深度研究，依照 CRAAP 標準評估資料可信度，生成 APA 格式的引用清單。

第六個 Twitter，自主管理 X 帳號，支援 7 種內容格式，敏感操作有審核佇列。

第七個 Browser，做網頁自動化，但有一個重要護欄：凡是涉及購買的操作，一定要人工批准才能執行。

這個護欄的設計思路很務實。完全自主的 Agent 在涉及金融交易時，強制加入人機協同這一關，是合理的安全邊界。

安全架構

OpenFang 最花心思的部分是安全設計，共有 16 層防護機制。

沙箱層使用 WebAssembly 雙重計量沙箱，同時追蹤燃料用量和執行時間，並有一個守護執行緒防止無限迴圈。子程序呼叫也做了環境變數隔離。

審計層使用 Merkle 哈希鏈記錄每個 Agent 動作，資料以密碼學方式連結，無法竄改。污點追蹤則標記敏感資訊從輸入到輸出的完整路徑。

身份層每個 Agent 的身份和能力集以 Ed25519 非對稱加密簽名；點對點通訊使用基於 HMAC-SHA256 的 nonce 認證；RBAC 能力閘門做細粒度的角色型存取控制。

運行時防護層包含 Prompt injection 掃描器，偵測覆蓋系統提示的嘗試；SSRF 保護封鎖私有 IP 和雲端 metadata 端點；API 金鑰在使用後自動從記憶體清零；速率限制依 IP 追蹤，並考量呼叫成本。

整合能力

LLM 支援方面，27 個提供商、超過 123 個模型，包含 Anthropic、OpenAI、Gemini、Groq、DeepSeek，以及中文模型方面的智譜 AI 和火山引擎豆包。

通訊通道方面，40 個平台適配器涵蓋 Telegram、Discord、Slack、WhatsApp、Signal、Matrix、電子郵件、Microsoft Teams、Google Chat、LINE、Facebook Messenger、Reddit、LinkedIn、IRC 等。

協定支援方面，同時支援 MCP 也就是 Model Context Protocol、Google 的 Agent 間通訊協定 A2A，以及 OpenFang 自訂的 OFP 協定。

效能比較

根據 2026 年 2 月的基準測試，OpenFang 冷啟動 180 毫秒，閒置記憶體 40 MB，安裝大小 32 MB，有 16 個安全系統和 40 個通道適配器。

相比之下，ZeroClaw 冷啟動最快只要 10 毫秒，記憶體只佔 5 MB，但只有 6 個安全系統和 15 個通道。OpenClaw 冷啟動接近 6 秒，記憶體接近 400 MB。LangGraph 冷啟動 2.5 秒，沒有通道適配器。

這些數字是 OpenFang 團隊自己測試的，尚未有獨立第三方驗證，需要保留一定程度的保留態度。

版本狀態與風險

截至 2026 年 3 月 5 日，OpenFang 已從 2 月 24 日的 0.1.0 快速迭代到 0.3.15 版，平均每天多個版本。最近的更新包括修復 UTF-8 處理問題、新增中文模型提供商、改善 Discord 整合。

Rust 的記憶體安全從語言層面消除了整類型別的 bug；單一執行檔部署大幅降低了依賴管理的複雜度；安全設計覆蓋了 AI Agent 特有的攻擊面。

但風險也很明顯。0.1.0 是首個公開版本，沒有生產環境的實際部署記錄。13 萬多行程式碼對一個小型團隊來說維護成本不低。v1.0 之前的版本可能有破壞性變更。官方文件建議生產環境要釘選到特定的 commit。

定位與方向

OpenFang 試圖解決的核心問題是：現有 Agent 框架更像函式庫，而非作業系統。你需要自己把排程、持久化、通訊、安全這些元件組合起來。OpenFang 的目標是讓 Agent 能像系統服務一樣部署，啟動後在背景自主運行，有明確的能力邊界，有完整的審計紀錄。

Hands 的設計哲學犧牲了靈活性，換取可預測的行為邊界。Rust 的選擇在效能和安全上有天然優勢，但也縮小了潛在貢獻者的範圍，這是跟 Python 生態比較時必然面對的取捨。
