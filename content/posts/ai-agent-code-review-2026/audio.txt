AI 代理程式碼審查，多代理驗證架構的技術現狀。

程式碼審查長期以來都是軟體開發流程的瓶頸，AI 輔助工具的普及讓這個問題更加明顯。舉個具體的例子：一個 250 名工程師的組織，如果每人每天合併一個 PR，一年就會產生約 65,000 個 PR，需要超過 21,000 小時的人工審查時間。根據 Pullflow 分析 4,030 萬個公開 GitHub PR 的數據，2024 到 2025 年間企業 PR 量年增 29%，但審查端的人力並沒有等比例增加。

人類審查者的生理限制使情況更複雜。研究顯示，審查者在閱讀超過 80 到 100 行程式碼後效率顯著下降，要對安全漏洞達到 95% 的信心水準，需要 12 到 14 名審查者共同參與，這在實際專案中幾乎不可能實現。Anthropic 在推出 Claude Code Review 時直接點出，code review 已成為瓶頸。根據 Pullflow 的數據，AI 代理參與 PR 的比例從 2024 年 2 月的 1.1% 成長到 2025 年 11 月的 14.9%，約一年內成長 14 倍。目前三大工具 CodeRabbit、GitHub Copilot 和 Google Gemini 佔所有 AI 參與 PR 的 72%。

目前市面上的 AI 程式碼審查工具大致分為幾類。

第一類是 PR 機器人型，直接整合 GitHub 或 GitLab，針對程式碼差異發送內嵌評論，代表工具包括 CodeRabbit、Greptile、Graphite Diamond，以及 Qodo 旗下的 PR-Agent 開源版本。

第二類是 IDE 原生型，在提交前即時回饋，包括 GitHub Copilot、Cursor 的 Bug Bot 和 Gemini Code Assist，優勢是把問題截斷在更早的開發階段。

第三類是安全專注型，像 DeepCode 和 PullRequest 這類工具結合 AI 與人工專家，適合高風險環境。

第四類是多代理平台型，Claude Code Review 是目前架構最明確的多代理實作案例，使用多個獨立代理分別負責不同面向的分析工作。

現行的 AI PR review 工具在架構上存在明顯的分歧，這些分歧決定了它們能發現什麼問題、又會產生多少噪音。

最淺的一層是 PR diff 層。GitHub Copilot 和 CodeRabbit 代表這類設計，在程式碼差異和檔案層級操作，給出行內回饋，但沒有跨儲存庫的可見性。優點是速度快、部署簡單，缺點是無法追蹤變更對其他元件的下游影響。

中間層是全 repo 層。Qodo 維護一個持久的組織上下文，索引跨儲存庫的共用程式庫、歷史決策與內部規範，可以偵測跨模組的破壞性變更。Greptile 走類似路線，專注於完整程式碼庫的語義理解，主要適合架構層級的問題偵測。

最深的一層是多代理驗證層。Claude Code Review 代表目前架構上最複雜的設計。一個 PR 打開時，系統派出多個專門代理平行分析程式碼，各自負責不同類型的問題。完成後進行一個驗證步驟，把候選問題對照實際程式碼行為過濾掉假陽性，再去重、按嚴重程度排序，最後以行內評論形式貼到 PR 對應的行號。

這是目前技術路線最核心的分歧。

單代理架構用一個大型語言模型同時處理安全性、效能、風格、架構、測試等所有面向。問題在於，當一個上下文視窗塞入太多職責，模型對每個面向的深度都被稀釋。

多代理架構讓每個代理維持在自己的專業領域，透過協調機制整合輸出。根據 diffray.ai 的數據，差異在實際漏洞偵測上相當顯著：SQL 注入的偵測率，單代理只有 27%，多代理達到 91%；認證繞過的偵測率，單代理 19%，多代理 87%；競爭條件的偵測率，單代理 23%，多代理 89%；N 加 1 查詢的偵測率，單代理只有 9%，多代理達到 94%。

驗證步驟是多代理架構中的關鍵設計決策。它存在的原因是大型語言模型在程式碼分析中的結構性問題：模型傾向於生成看起來合理的問題描述，但不一定對應真實的 bug。驗證步驟引入額外的推理環節，判斷問題是否真正存在於程式碼執行路徑中，而不只是出現在差異的文本模式裡。

Claude Code Review 的具體實作是：每個發現計算 0 到 100 的信心分數，預設閾值 80 分以上才發出評論。這種設計試圖在覆蓋率和雜訊之間取得平衡，寧可漏報，也不要用無效評論淹沒開發者。

各工具的基準測試數字差異大到讓人困惑，這個問題本身值得討論。

Greptile 自行發布的基準測試，分析 50 個真實開源 PR，結果顯示 Greptile 自己得了 82%，Cursor 58%，Copilot 54%，CodeRabbit 44%，Graphite 6%。而 Macroscope 2025 的獨立評估結果截然不同：Macroscope 48%，CodeRabbit 46%，Cursor Bug Bot 42%，Greptile 只有 24%，Graphite Diamond 18%。Greptile 從第一名掉到第四名。

這個矛盾說明一個核心問題：AI 程式碼審查工具目前沒有通用的評估標準。抓到的 bug 如何定義？是行級評論、影響分析還是修復建議？測試 PR 是怎麼挑選的？這些方法論差異直接決定排名結果。工程師在選型時，應該把任何廠商自行發布的基準數字打折扣，並優先做內部測試。

在沒有矛盾的部分，CodeRabbit 的一致性和低誤報率在多個測試中獲得認可。在一個真實測試場景中，Greptile 捕獲更多問題但有 11 個誤報，CodeRabbit 誤報只有 2 個但漏報較多，這個取捨反映了不同工具在設計哲學上的選擇。

假陽性是 AI 程式碼審查工具面臨的核心可信度問題。開發者投訴最多的是：AI 自信地標記一個根本不存在的問題，觸發大量調查工作，浪費的時間比節省的還多。

根據 diffray.ai 的研究，大多數工具的假陽性率在 5% 到 15% 之間，對中等規模的團隊每週浪費 2.5 小時工程師時間。更大的問題是幻覺帶來的信任危機：25% 的開發者估計每 5 個 AI 建議中就有 1 個包含事實性錯誤；近 20% 的套件推薦指向不存在的程式庫。信任 AI 建議準確性的開發者比例從 2024 年的 43% 下降到 2025 年的 33%，一年內掉了 10 個百分點。

這個信任問題有結構性根源：大型語言模型在訓練上被優化為自信的應試者，當無法確定答案時，傾向猜測而非承認不確定性。

目前有三個緩解方向。

第一是 RAG 整合，也就是檢索增強生成。讓代理能存取整個程式碼庫的索引，而非只看差異。Greptile 的設計重點就是完整程式碼庫索引，這個方法可減少 60% 到 80% 的幻覺。

第二是多代理交叉驗證。一個代理發現問題後，另一個代理驗證該問題是否真實存在，可提升一致性約 85.5%。

第三是靜態分析整合。將大型語言模型的語義分析與傳統靜態分析工具結合。IRIS 框架的測試顯示，這種組合能偵測 55 個漏洞，而純 CodeQL 只偵測到 27 個；綜合精確度提升約 89.5%。

三種方法組合理論上可達到 96% 的幻覺降低，但這個數字來自研究環境，實際應用仍有差距。Anthropic 聲稱 Claude Code Review 的假陽性率低於 1%，主要歸因於驗證步驟的設計。

在可量化的生產數據方面，Anthropic 內部的數據很有參考價值。在 Claude Code Review 之前，Anthropic 只有 16% 的 PR 得到深入審查，啟用後提升到 54%。大型 PR，也就是 1,000 行以上的，有 84% 會收到問題回報，平均 7.5 個問題；小型 PR 只有 31% 收到回報，平均 0.5 個。整體審查時間平均 20 分鐘。

Anthropic 內部兩個真實偵測案例值得關注：一個本地 HTTP 伺服器的 DNS Rebinding 遠端執行漏洞，以及一個憑證管理代理的 SSRF 攻擊向量。這類跨檔案上下文推理的問題，正是人工審查最容易忽略的地方。

Microsoft 的 AI 程式碼審查助手覆蓋超過 90% 的公司 PR，每月影響超過 60 萬個 PR，在 5,000 個儲存庫的早期數據中，PR 完成時間中位數改善了 10% 到 20%。

GitHub Copilot 截至 2025 年 4 月已自動審查超過 800 萬個 PR，使用 Copilot 的開發者完成任務速度快 55%，PR 週期從平均 9.6 天縮短到 2.4 天。

Claude Code Review 的定價基於 token 使用量，平均每次審查 15 到 25 美元，隨 PR 大小和複雜度線性增加。對一個每週處理 50 個 PR 的工程團隊，月度成本約 2,000 到 5,000 美元，等同於 1 到 2 個工程師每週花數小時的工時成本。

開源替代方案存在但各有限制。PR-Agent，Qodo 的開源版本，可自行部署，只需支付大型語言模型的 API 費用，高度可客製化，但需要自行維護。llm-code-reviewer 是 GitHub Action，支援 OpenAI GPT 和 Google Gemini，適合想測試但不想立即付費的團隊。

Claude Code Review 支援兩種客製化文件：CLAUDE.md 是全局指引，REVIEW.md 僅審查時讀取。REVIEW.md 可以指定始終檢查的項目，比如「所有新 API 端點必須有整合測試」，也可以指定應忽略的項目，比如「生成代碼目錄下的檔案不需要評論格式問題」，以及特定框架或語言慣例。

這個設計讓工程師可以將團隊的隱性知識編碼進審查規則，而不是每次審查都重複給出相同的背景說明。缺點是維護這些配置文件本身需要時間，過度複雜的規則會讓 AI 陷入矛盾指引。

接下來談幾個目前工具還沒有很好解決的問題。

第一是跨儲存庫影響追蹤。大多數工具在儲存庫邊界就停止分析。當一個微服務的 API 修改會破壞另一個服務的消費者時，目前只有 Qodo 聲稱能追蹤這類問題，但實際覆蓋範圍在生產環境中仍有爭議。

第二是業務邏輯理解。AI 代理擅長找出語法錯誤、型別錯誤、常見安全模式，但對於「這個函數的行為是否符合業務需求」這類問題，仍高度依賴工程師提供的上下文。

第三是審查疲勞。AI 代理增加了 PR 上的評論數量，可能讓工程師面臨更多需要處理的回饋。Graphite Agent 數據顯示其建議有 67% 的實施率，意味著 33% 的回饋被忽略。如何讓工程師對 AI 評論建立正確的閱讀習慣，而不是麻木地關閉所有通知，是個尚未解決的問題。

第四是驗證悖論。如果每個 AI 建議都需要仔細驗證，而驗證的時間等同於自己分析的時間，那效益從哪裡來？這個問題對不同規模的組織有不同答案：大型組織即使 50% 的審查可以自動化，省下的工時仍然可觀；小型團隊的工具設定和誤報處理開銷可能抵消收益。

第五是確認偏誤。當 AI 說看起來沒問題時，開發者更容易跳過深度審查。Stack Overflow 的報告指出，這種 AI 背書效應可能讓某些類型的問題更難被發現，因為人工審查者的注意力被 AI 通過的代碼所麻痺。

AI 代理程式碼審查目前處於可用但不成熟的階段。工具在特定問題類別上已經可靠，在需要全域業務邏輯理解的問題上仍然薄弱。多代理架構是明確的技術趨勢，但大多數工具的多代理是行銷術語，真正有獨立角色分工和交叉驗證機制的實作還在早期。

工具之間最重要的技術差異不是用了哪個大型語言模型，而是上下文的範圍，以及是否有驗證機制。缺乏標準化基準是整個領域的短板，工程師只能靠在真實程式碼庫上做概念驗證測試才能得到有參考價值的數字。

對於考慮部署的工程團隊，務實的起始點是：先評估目前的審查覆蓋率和瓶頸所在，選擇一個可客製化規則的工具，從少量高 PR 量的儲存庫開始，3 個月後評估假陽性率和工程師採用程度再決定是否擴大部署。AI 審查的邊際價值主要來自增加覆蓋率，它現在的最佳定位是一個永遠在線、從不疲倦的初級審查者，負責捕捉明顯問題和保持風格一致，而非替代資深工程師的架構判斷。
