以下是「Playwright Agents 與 Claude Code：三個專用子代理自動化 E2E 測試」的語音版本。

背景

Playwright 一直是 Web 端對端測試的主流框架，但寫測試的人力成本一直不低。要理解應用的使用者路徑、選擇穩定的定位器、處理非同步等待，還要在介面變更後持續維護測試。

Playwright 1.56 針對這個問題的切入點不是讓大型語言模型直接生成程式碼，而是設計了三個專用代理，分別負責測試生命週期的不同階段。這三個代理作為 Claude Code 的子代理運行，透過 Model Context Protocol 也就是 MCP 與瀏覽器互動。

三個代理的架構

Playwright Agents 的設計邏輯是把「寫一個端對端測試」拆解成三個獨立的問題，讓專門化的代理各自處理。

第一個是規劃師代理，英文叫 Planner。它的工作是在真實瀏覽器裡導覽應用程式，識別使用者路徑和邊界情況，然後把發現的內容寫成結構化的 Markdown 文件，存放在 specs 目錄下。這個過程就像資深品管工程師在寫測試計畫，先把「要測什麼」和「怎麼測」記錄清楚，不急著寫程式碼。

第二個是生成器代理，叫 Generator。它把 Markdown 計畫轉換成 TypeScript 測試程式碼，優先使用語義定位器而不是脆弱的 CSS 選擇器，並且在每個步驟執行後立即驗證選擇器和斷言是否有效。這個設計是為了防止代理憑空猜測選擇器，產生看起來合理但實際無法執行的測試程式碼。Generator 整合了超過 16 個 Playwright 工具，涵蓋點擊、文字輸入、頁面導航、截圖等各種操作。

第三個是修復師代理，叫 Healer。當測試失敗後，它重放失敗的步驟，結合 Playwright trace 檔案找出根本原因，然後在當前介面狀態中尋找等效元素，提出修補方案並驗證修復效果。如果問題無法在有限嘗試內解決，它會把測試標記為 test.fixme，並附上說明，讓人工接手，避免陷入無限重試。

與 Claude Code 的整合方式

通過一個命令就可以初始化代理：在終端執行 npx playwright init-agents，加上 --loop=claude 參數。

執行後，命令會在專案裡生成代理定義檔案和一個種子測試檔案 seed.spec.ts。代理定義本身是 Markdown 文件，這是 Claude Code 子代理架構的標準格式。你可以直接編輯這些文件，調整探索策略、程式碼風格偏好或修復策略，完全不需要改任何程式碼。

seed.spec.ts 這個種子測試在整個系統裡扮演重要角色，它提供具體的程式碼範例，讓代理理解專案的慣例，起到示例提示的作用。

Playwright MCP 與 CLI 的取捨

Playwright Agents 目前基於 MCP 運作。2026 年初 Playwright 團隊同時推出了命令列介面方案，兩者的 token 消耗差距相當明顯。

根據 Playwright 團隊的基準測試，同樣的瀏覽器自動化任務，MCP 方案需要大約十一萬四千個 token，而命令列介面方案只需要大約兩萬七千個，差了將近四倍。

原因在於架構不同。MCP 在每次工具呼叫後把完整的無障礙樹或截圖傳回語言模型的上下文視窗，而命令列方案把快照和截圖寫入磁碟，讓代理按需讀取特定檔案，大幅減少進入上下文的資料量。

Playwright 團隊的建議是：命令列方案適合 Claude Code、Cursor 等代碼代理，MCP 適合無法直接存取檔案系統的沙盒環境。對於像 Playwright Agents 這種需要持續瀏覽器上下文的長時間工作流程，MCP 的設計也有其優勢。

現有限制

根據社群的實際使用回饋，Playwright Agents 目前有幾個明確的邊界。

業務邏輯驗證方面，代理可以驗證介面的視覺狀態，但無法判斷計算結果是否正確，這需要人工定義斷言規則。複雜有狀態的流程方面，涉及多個步驟且有依賴關係的測試場景，代理的表現不穩定。測試治理方面，代理可以大量生成測試，但哪些應該進入持續整合流程這個決策目前仍需人工判斷。另外，修復師代理在複雜失敗場景下有時會給出根本原因分析有誤的修復方案。

實際應用場景

Playwright Agents 在幾個場景有明確的收益。新功能上線後，讓規劃師代理探索，快速建立基礎測試集；介面重構後，修復師代理可以批量處理失敗的測試；對 Playwright API 不熟悉的開發者，可以用自然語言描述要測試什麼，讓代理生成符合最佳實踐的程式碼。
