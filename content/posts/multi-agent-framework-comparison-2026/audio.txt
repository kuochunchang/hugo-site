以下是「Multi-Agent AI Framework 深度比較報告 2026 Q1」的語音版本。

大家好，今天我想跟各位分享一個非常令人興奮的趨勢。2025 到 2026 年是多代理人 AI 框架的爆發期。根據 Gartner 的報告，多代理人系統的諮詢量從 2024 年第一季到 2025 年第二季，暴增了 1,445%，這個成長幅度可說是非常驚人。

這波趨勢背後有幾個主要的推動力量。首先是大廠的收斂整合。Microsoft 把他們的 AutoGen 和 Semantic Kernel 這兩個框架合併成一個統一的 Microsoft Agent Framework。OpenAI 也從早期的實驗性 Swarm 專案升級成正式的 Agents SDK，這說明了這個領域已經進入了更成熟的階段。

其次是協定的標準化。有一個叫做 MCP，也就是 Model Context Protocol 的東西，現在已經成為了工具連接的事實標準，月下載量已經超過九千七百萬。同時還有一個叫做 A2A，也就是代理人之間溝通的協定，由 Google 主導、Linux Foundation 託管，這個協定主要是用來支援不同代理人之間的互操作。

第三個重要的趨勢是 CDP，也就是客戶資料平台的智能化。在 2026 年的 Gartner Magic Quadrant for CDPs 報告中，新增了一個能力維度叫做「Agentic Process Optimization」，這代表著 CDP 正在從傳統的資料平台演進成一個由代理人驅動的客戶智能平台。

對於 CDP 的團隊，我們有一個核心的建議。首先，以 LangGraph 作為你的編排層首選，因為它的圖狀流程最適合 CDP 那些複雜的資料管線。接著，搭配 MCP 來做工具的整合，最後再用 A2A 協定支援跨系統的代理人互操作。這樣的組合應該能夠幫助你們打造更強大、更靈活的多代理人系統。

今天我們來深入探討四個最主要的多代理人框架。

首先來看 CrewAI。CrewAI 的最新版本是 v1.9.3，在 GitHub 上已經累積了超過四萬四千顆星，採用 MIT 開源授權，支援 Python 語言。

CrewAI 採用的是一種叫做角色導向的多代理人架構。這個靈感來自真實世界的團隊協作模式。簡單來說，每個 Agent 都被賦予一個特定的角色、一個明確的目標，還有一個背景故事，就像一個「船員」在團隊中各司其職一樣。

這個框架是從零開始建構的，完全獨立於 LangChain。它提供兩種互補的模式。第一種是 Crews，也就是自主代理人團隊，特別適合那些需要真正自治和協作的複雜任務。第二種是 Flows，這是事件驅動的生產工作流，能夠對自動化流程提供更精細的控制。

框架的關鍵功能包括幾個方面。首先是角色導向的代理人定義，讓你可以很清楚地設定每個 Agent 的角色。然後是 Sequential 和 Hierarchical 兩種流程編排方式。它還內建了 Human-in-the-Loop 工作流，也就是讓人類可以在流程中適時介入。記憶系統方面，支援短期記憶、長期記憶和實體記憶。從 v1.8.0 版開始，它還支援原生的 async chain，也就是 A2A 協定。此外，它還整合了 MCP 的支援。

講到 CrewAI 的優勢。上手非常快，直覺的角色設定模式讓原型開發的速度最快。社群也相當龐大，四萬四千多顆 GitHub stars 說明了這一點。企業平台已經相當成熟，CrewAI 提供了一個叫 AMP Suite 的統一控制面板。框架的整合也很豐富，支援 Gmail、Slack、Salesforce 等多個應用。而且從 v1.8.0 開始，內建了 A2A 協定的支援。

不過 CrewAI 也有一些劣勢需要注意。Manager-Worker 的架構設計有一些缺陷，在階層式流程中，Manager 無法有效地協調代理人的工作。記憶的可擴展性也有限制，長期記憶依賴 SQLite3 資料庫。除錯的體驗並不是很好。生態系統相對來說還是比較小。在生產環境中還有一些已知的問題。還有，當多個代理人同時運行時，對 token 的消耗需要非常嚴格的管控。

講到定價。開源框架本身是免費的。平台的免費方案每個月提供 50 次執行機會。Professional 方案每月 25 美元可以執行 100 次。Business 方案每月 99 美元。Ultra 方案是年費十二萬美元。Enterprise 方案則提供客製化的報價。

在最近的更新方面，v1.8.0 版加入了原生的 A2A 支援，v1.9.x 系列一直在做穩定性的修正。框架的發展方向是要強化 Flow 引擎的功能，同時增加更多的企業整合。

我們來聊聊 LangGraph 這個框架。LangGraph 在 2025 年 10 月 22 日推出了 v1.0 的正式版本。在 GitHub 上的表現相當亮眼，LangGraph 本身獲得了約兩萬五千顆星，而它的母專案 LangChain 則累積了約十一萬八千顆星。這個專案採用 MIT 授權，同時支援 Python 和 TypeScript 兩種開發語言。

LangGraph 的核心設計是採用有向圖這樣的結構，把代理人的工作流程建模成由節點和邊所組成的圖。換句話說，每個工作步驟就是一個節點，步驟之間的連接就是邊。這個設計哲學的目標是給開發者最大的彈性，讓你能夠實作任何你想要的架構模式。這個想法的靈感來自 Google Pregel 和 Apache Beam 這兩個分散式計算系統，所以它內建了完整的狀態管理和持久化的能力。

在功能上，LangGraph 提供了相當豐富的特性。圖狀工作流支援條件分支，也就是說根據不同的條件走不同的路線，還支援平行執行多個任務、迴圈、以及動態路由。狀態持久化的功能讓代理人的執行狀態會自動保存下來，就算伺服器重新啟動了，你也能從之前中斷的地方繼續執行。它提供了一等的人工介入功能，也就是在需要的時候可以讓人類參與決策過程。另外還支援延遲節點執行，工具可以直接更新圖的狀態。此外還有 LangGraph Platform 這個雲端部署平台，以及 LangSmith 來進行追蹤和評估的整合。

說到優勢，LangGraph 首先提供了最大的彈性。因為是圖結構，所以理論上可以表達任何工作流模式。它的生產就緒程度也是最高的，v1.0 GA 是整個領域裡第一個穩定的大版本。很多知名的大型企業像是 Uber、LinkedIn、Klarna 和 Replit 都已經在生產環境中使用 LangGraph。在狀態管理上表現也很卓越，而且背靠著龐大的 LangChain 生態系統，基於 Pregel 和 Apache Beam 架構的可擴展性也很強。

不過當然也有一些需要考慮的地方。首先學習曲線相當陡峭，因為你需要紮實的圖論、狀態機和分散式系統的知識才能用好它。如果你的任務很簡單，使用 LangGraph 可能會感到過度工程化，而且樣板程式碼會比較多。部署的複雜度也比較高，在使用的過程中還有記憶體洩漏的風險。深度使用平台之後可能會有被鎖定的風險。

再來說說定價。LangGraph 的核心框架是完全免費的，採用 MIT 授權。Platform 則是按節點執行來計費，每個節點執行 0.001 美元。LangSmith 的 Developer 方案是免費的，提供五千次的追蹤額度，如果你想要更多功能，Plus 方案是每個用戶每個月 39 美元。順帶一提，LangChain 公司的估值已經達到了 12.5 億美元。

最後提一下最近的更新。2025 年 10 月推出了 LangGraph v1.0 GA，下半年還新增了工具直接更新圖狀態和延遲節點執行這兩個功能。

我們來聊一下 AutoGen 和微軟代理人框架的最新動態。AutoGen 目前已經更新到第 0.4 版，這是一個完全重新開發的版本。微軟的代理人框架現在正處於發布前的候選階段，在 2026 年 2 月 19 日剛剛發布，他們的目標是在 2026 年第一季末能夠正式上線。

在社群的支持方面，GitHub 上 AutoGen 獲得了大約五萬四千七百顆星，另外還有一個社群衍生版本叫 AG2，大概有四千兩百顆星。這兩個項目都採用 MIT 授權，也都支援 Python 和 .NET 也就是 C# 這兩種程式語言。

讓我們回顧一下 AutoGen 最初的設計理念。它原本是採用對話導向的多代理人架構，特別強調自然語言的互動以及動態的角色扮演。然後到了 2025 年 10 月，微軟宣布了一個重大的決定，他們要把 AutoGen 和 Semantic Kernel 這兩個框架合併，打造成一個統一的微軟代理人框架。新框架既保留了 AutoGen 那種簡潔的代理人抽象和多代理人編排的優勢，又結合了 Semantic Kernel 針對企業級應用的基礎設施，還加入了全新的圖狀工作流支援。

新框架的關鍵功能有哪些呢？首先是事件驅動的架構，然後是全新推出的圖狀工作流，用來處理複雜的流程。它還有 Session 狀態的管理，確保資料的一致性。框架本身也提供了型別安全的設計，加上中介軟體和遙測工具，方便開發者進行監控和除錯。最重要的是企業級的安全和治理功能，包括 AutoScale、Identity 身分管理和 Governance 治理，以及和 Azure AI Foundry 的整合，讓企業用戶能夠更便利地部署和管理。

現在我們來看看它的優勢在哪裡。首先，這個框架能夠原生地連接整個微軟的生態系統，包括 Azure、Microsoft 365 還有 Dynamics 365，這對於已經在使用微軟服務的企業來說非常有吸引力。框架本身是以企業級的需求來設計的，所以在可靠性和功能上都有保障。另外在程式語言支援方面，同時支援 Python 和 C# 的框架其實不多，這也是一個優勢。還有龐大的社群支持，以及微軟的長期投入承諾，而且採用 MIT 授權，完全免費。

不過它也面臨著一些挑戰。最大的問題是生態系統出現了分裂。現在同時有 AutoGen 0.2 版本、0.4 版本、AG2 社群分支，還有新的微軟代理人框架四個版本並行運作，這讓社群有點混亂，使用者也不知道該選擇哪一個。現有的 AutoGen 使用者如果想升級到新框架，需要面臨遷移的負擔。新框架現在還沒有正式上線，仍然是候選版本。原本的 AutoGen 功能已經凍結，不會再有新功能開發。AG2 這個社群分支雖然活躍，但它在生產環境中的就緒度還不夠高。還有底層的 actor 框架本身存在一些設計限制，這可能會影響某些進階應用場景。

講到價格，其實核心框架都是免費的，都採用 MIT 授權。如果你要在 Azure AI Foundry 上部署，費用會按照 Azure 雲端服務的標準計費。而 AG2 社群版本則是完全免費，沒有任何額外費用。

最後來看一下近期的發展計畫。2025 年 10 月進行了公開預覽，2026 年 2 月 19 日發布了候選版本，目標是在 2026 年第一季末能夠發布 1.0 的正式版本。至於原本的 AutoGen，現在只會進入維護模式，主要是修復漏洞和提供支援，不會再有重大更新。

今天要介紹的是 OpenHands，這是一個過去叫做 OpenDevin 的框架，最新版本已經來到 v1.4.0。它在 GitHub 上獲得了超過六萬八千顆星，是我們今天介紹所有框架中最受歡迎的。OpenHands 採用 MIT 授權，用 Python 開發。

OpenHands 的定位非常清楚，就是一個軟體開發專用的自主 AI 代理人平台，換句話說就是一個 AI 驅動的軟體工程師。有趣的是，與其他多代理人框架不同，OpenHands 刻意選擇了單代理人系統，他們認為與其分散力量，不如打造一個強大的單一代理人，搭配豐富的工具來得更有效。在 V1 版本，他們推出了 OpenHands Software Agent SDK，這是一個可組合的 Python 函式庫，可以在本地或雲端規模化地運行代理人。

那 OpenHands 到底能做什麼呢？它可以獨立完成軟體開發的整個流程，從程式碼撰寫、除錯到測試，全都可以自動化。除此之外還能進行程式碼審查、自動生成文件，甚至清理技術債。在安全性方面，每個代理人都在隔離的 Docker sandbox 中執行，這樣就能確保系統的安全。最重要的是，它與模型無關，不管你用 Claude、GPT 還是其他任何 LLM，都能無縫整合。在企業應用上，它也做得很完善，可以與 GitHub、GitLab、Bitbucket、Slack 還有 Jira 進行整合。

說到 OpenHands 的優勢，首先就是它的人氣，GitHub Stars 數最高達六萬八千多，這反映出社群的認可度。更重要的是，它在軟體開發上真的很專精，有數據顯示百分之八十七的 bug ticket 都能在同一天內解決。它的安全模型也很完善，而且作為完全開源的 MIT 授權專案，已經有一百八十八位以上的貢獻者參與開發。另外它還獲得了一千八百八十萬美元的 Series A 融資，這也顯示了投資者對它的信心。它提供的可組合 SDK 也讓開發者有更多的彈性。

當然 OpenHands 也有它的限制。首先它不是一個通用的多代理人框架，它就是專門為軟體開發任務而生，所以如果你的需求超出軟體開發的範疇，它可能就不是最適合的選擇。特別是對於 CDP 場景來說，它沒辦法直接用於資料管線或是客戶分群這類核心功能。另外 frontier 模型的成本比較高，而且 Context Window 有一定的限制，這有時候會成為瓶頸。還有一點值得注意的是，使用開放模型的話效能會有不少差距，這也是使用時需要考慮的因素。

在定價方面，開源版本是完全免費的，如果你選擇 SaaS 版本的話是按使用量計費，企業自託管則是提供客製化的報價。最後要提一下的是，OpenHands 最近在快速迭代，v1.2.1、v1.3.0、v1.4.0 連續發布，而且他們還推出了自有的代理人能力評估基準，叫做 OpenHands Index。

接下來我們要介紹一些新興的框架補充比較。

首先是 OpenAI Agents SDK。這個 SDK 在 2025 年 3 月發布，其實它取代了之前實驗性的 Swarm。在 GitHub 上已經獲得了約一萬九千顆星，採用 MIT 授權，支援 Python 和 TypeScript 兩種語言。

OpenAI Agents SDK 的設計哲學是走極簡主義路線，只有四個核心原語，分別是 Agents、Handoffs、Guardrails 還有 Sessions。它的核心功能包括 Agent 之間可以進行 Handoff 委派、支援平行執行的 Guardrails 來做輸入輸出驗證、透過 Sessions 實現持久化記憶、提供 Human-in-the-Loop 的機制、內建追蹤功能，還有 Realtime Agents 語音代理人的支援，以及對 MCP 的完整支援。

這個框架的優勢是入門門檻極低、與 OpenAI 生態無縫整合，並且支援超過一百多種 LLM。劣勢的地方是如果你要做進階編排，需要自己實作，而且整體功能相對來說比較精簡。SDK 本身免費使用，但底層模型的費用是按照 OpenAI API 計費。

接下來看 Google ADK，也就是 Agent Development Kit。Google ADK 在 2025 年 4 月於 Google Cloud NEXT 大會上發布。它在 GitHub 上有約一萬五千六百顆星，採用 Apache 2.0 授權，支援 Python、TypeScript、Go 和 Java 四種語言，這是所有框架中語言支援最廣的。

Google ADK 的設計哲學是讓代理人開發的體驗就像傳統軟體開發一樣，採用 code-first 的方法。它的功能包括可以構建模組化的多代理人系統、提供 Workflow Agents 包括 Sequential、Parallel 和 Loop 三種類型、支援 LLM 驅動的動態路由、設計上與模型無關雖然針對 Gemini 做了優化、與 Vertex AI Agent Builder 的整合，以及原生支援 A2A 協定。

Google ADK 的優勢在於與 Google 生態的原生整合、多語言支援最為廣泛、以及採用 bi-weekly 的發布節奏保持活躍。劣勢是它相對較新、社群資源目前仍在建構當中。這個框架開源免費，但如果你使用 Vertex AI 平台的話，會有額外的平台計費。

再來看 AWS Strands Agents。Strands Agents 在 2025 年 5 月開源，v1.0 版本在 7 月發布，目前最新版本是 v1.26.0。在 GitHub 上有約五千一百顆星，採用 Apache 2.0 授權，支援 Python 和 TypeScript。

Strands Agents 的設計哲學是以模型驅動，也就是讓模型本身的能力來主導代理人的行為。功能包括超簡單的 API，你只需要幾行程式碼就可以建立一個代理人、支援 Graph、Swarm 和 Workflow 多種多代理人編排方式、提供 A2A 協定的互操作、把 MCP 視為一等公民進行支援、設計上與模型無關、有 OpenTelemetry 的可觀測性，以及提供多種部署選項。

Strands Agents 的優勢是上手最簡單、與 AWS 生態深度整合、已經累積超過五百萬的下載量，而且在 AWS 內部被多個服務採用。劣勢是過度依賴模型本身的能力，在複雜場景下客製化會受到限制。這個框架開源免費，但 AWS 的基礎設施計費需要另外計算。

然後我們看 Claude Agent SDK，這是 Anthropic 推出的。Claude Agent SDK 在 2025 年 5 月以 Claude Code SDK 的名稱發布，到了 9 月才重命名為 Claude Agent SDK。它採用 MIT 授權，支援 Python 和 TypeScript。

Claude Agent SDK 的設計哲學很有趣，它認為應該給代理人一台電腦，而不是僅僅一個 prompt，這樣代理人就能獲得完整的執行環境，包括終端、檔案系統和網路的訪問。它的功能包括與 Claude Code 共用底層基礎設施、提供 Tool Use 編排迴圈、支援 Guardrails、Tracing 追蹤、Context Compaction 上下文緊縮、深度的 MCP 整合、Computer Use 能力，以及多代理人研究系統。

Claude Agent SDK 的優勢是具有 Claude 模型的原生優勢、作為 MCP 發明者提供的一等整合，以及企業級的安全能力。劣勢是它較深度地綁定於 Anthropic 的生態，而且企業版預計要到 2026 年的第二季才會推出。SDK 本身免費使用，但 Claude API 的費用是按照你所使用的模型計費。

最後，我們來看 Amazon Bedrock Agents。Bedrock Agents 的多代理人協作功能在 2025 年 3 月 10 日正式上線，它定位為框架無關的託管平台。

核心功能包括採用 Supervisor-Subagent 的架構，最多可以支援三層階層、提供智慧路由、支援平行通訊、以及整合追蹤與除錯控制台。另外還有 AgentCore 在 2025 年 10 月上線，它也是框架無關的託管平台，可以運行 LangGraph、CrewAI、Google ADK、OpenAI Agents SDK 等框架。

Amazon Bedrock Agents 的優勢是完全託管無需你自己維護、設計上框架無關、以及與 AWS 的深度整合。劣勢是它造成 AWS 的強制綁定，自定義控制的空間比較少。費用方面是按照 AWS 的使用量計費。

今天我們要討論的是兩個非常重要的跨框架協定，分別是 MCP 和 A2A。

首先來介紹 MCP，全名是 Model Context Protocol。MCP 是由 Anthropic 這間公司在 2024 年 11 月發起的一個協定標準，現在已經成為業界的事實標準，每個月的下載量超過九千七百萬次。有很多知名的科技公司都採用了 MCP，包括 OpenAI、Google、Microsoft、還有 LangChain、Hugging Face 和 Deepset。值得一提的是，在 2025 年 12 月的時候，MCP 被捐贈給了 Linux Foundation 底下的 Agentic AI Foundation 進行託管。MCP 的主要用途是什麼呢？簡單來說，它是用來標準化代理人與外部工具、資料庫以及 API 之間的連接方式。根據 Gartner 的預測，到了 2026 年，大約百分之七十五的 API gateway 廠商和百分之五十的 iPaaS 廠商都會具備 MCP 的功能。

接著我們來看 A2A，這個協定的全名是 Agent-to-Agent Protocol。A2A 是由 Google 在 2025 年 4 月發起的，目前的版本是 v0.3，已經獲得了超過一百家科技公司的支持。A2A 是由 Linux Foundation 來進行託管的。支持的公司名單相當龐大，包括 ServiceNow、UiPath、Adobe、PayPal 和 SAP 等等。從技術的角度來看，A2A 的技術基礎是建立在 HTTP、JSON-RPC 以及 SSE 之上的，而在最新的 v0.3 版本中，還新增了 gRPC 的支持。A2A 的主要目的是實現跨框架、跨廠商的代理人之間的通訊。

最後，我想強調一下 MCP 和 A2A 這兩個協定之間的關係。簡單來說，MCP 的作用是連接工具，而 A2A 的作用是連接代理人，兩者是互補的，共同構成了代理人生態系統中不可或缺的基礎設施。

我們來看一下這八個框架的綜合比較。

首先從架構模式來看，每個框架都有自己的設計理念。CrewAI 採用角色導向的方式，LangGraph 則用有向圖來組織流程，AutoGen 和微軟的 Agent Framework 走對話和事件驅動的路線，OpenHands 是單代理人加上各種工具的組合，OpenAI 的 Agents SDK 走極簡的 Handoff 設計，Google 的 ADK 是 Code-first 的模組化架構，Strands Agents 強調模型驅動，而 Claude Agent SDK 則是執行環境導向的方式。

看社群活躍度的話，按 GitHub Stars 來排名，OpenHands 最受歡迎有六萬八千多個星星，AutoGen 緊追在後五萬五千多，CrewAI 有四萬四千多，LangGraph 兩萬五千多，OpenAI Agents SDK 一萬九千多，Google ADK 一萬五千六百多，Strands Agents 五千一百多。

版本成熟度這塊，LangGraph 已經推出 1.0 GA 版本最為穩定，CrewAI 到了 1.9 版本也很穩定，Strands 1.26 版本、OpenHands 1.4 版本都已經穩定了，但 AutoGen 和微軟 Agent Framework 還在 RC 階段等著正式 GA。

語言支援方面，Google ADK 的支援範圍最廣涵蓋了 Python、TypeScript、Go 和 Java，大多數框架都支援 Python 和 TypeScript，AutoGen 特別還支援 .NET。

學習難度上，Strands Agents 和 OpenAI Agents SDK 是最好上手的，CrewAI 的學習曲線也不會太陡，但 LangGraph 相對來說需要花一點時間去理解。

講到生產環境的就緒程度，LangGraph 是最高的，OpenHands 在專用領域的表現也是最頂級的。

MCP 和 A2A 的支援這部分，所有框架都支援 MCP，A2A 方面的話，CrewAI 從 1.8 版本開始、Google ADK、還有 Strands Agents 都有原生支援。

最後如果是用在 CDP 這類場景的話，LangGraph 是最佳選擇，Google ADK、Strands Agents 和 AutoGen/MS Agent Framework 是中高等級，CrewAI 和 OpenAI Agents SDK 是中等水準，OpenHands 相對來說適配度比較低一些。

各位聽眾，今天我們要來談一個非常重要的話題，就是怎麼用多代理人系統來建構 CDP，也就是客戶數據平台。

首先我們來看 CDP 系統的核心需求有哪些。在資料整合的環節，我們需要一個 Data Ingestion Agent，它的工作是處理來自各種不同來源的資料，把它們轉換成統一的格式，然後檢測資料的品質。接著在身份解析的部分，我們需要一個 Identity Resolution Agent，它要做的事情就是比對和合併來自不同通路的客戶身份。然後是客群分類，這需要一個 Segmentation Agent 來根據客戶的行為和屬性做動態分群。旅程編排這個環節需要 Journey Orchestration Agent，負責設計和執行跨通路的個人化客戶旅程。在預測分析方面，我們需要 Predictive Analytics Agent 來做流失預警、計算客戶的終身價值，還有推薦引擎。最後在合規治理和報表洞察這兩個環節，分別需要 Compliance Agent 來做資料存取控制和隱私法合規，以及 Insight Agent 來執行自然語言查詢和自動化報表生成。

現在我們來討論一下不同框架的適配性。首先，LangGraph 是我們最推薦用於 CDP 的框架，主要有五個理由。第一個理由是它的圖狀流程架構天然就適合 CDP 的管線，從資料整合、身份解析、分群，一直到最後的啟動，這個過程本質上就是一個有向無環圖。第二個優勢是它能夠處理條件分支，這對於 CDP 這種系統來說非常重要，因為不同的資料來源和不同的客群需要走不同的處理路徑。第三個理由是它支援狀態持久化，這對於 CDP 中那些需要長時間運行的工作流非常關鍵，無論是批次處理還是跨日排程，都需要可靠的狀態管理。第四個優點是它支援 Human-in-the-Loop 的審批流程，當資料品質出現異常或遇到合規問題時，可以讓人工來介入處理。第五個優勢是它在生產環境中的就緒度最高，已經在像 Uber、Klarna 這樣的大型企業的生產環境中得到了驗證。

當然，如果你的團隊已經在 AWS 的生態系統中，你可以考慮使用 AWS Strands 配合 Bedrock AgentCore。Bedrock AgentCore 的好處是它可以運行任何框架，不會被某個特定框架鎖定。而 Strands 的優點是上手非常快，特別適合快速原型開發，而且它與 S3、Redshift、DynamoDB 這些 CDP 常用的資料儲存系統的整合都很不錯。

如果你的團隊在 Azure 生態中，Microsoft Agent Framework 是一個很好的選擇。它可以原生地整合 Azure Data Factory、Synapse、Cosmos DB 這些工具，支援 .NET 開發，而且在企業治理方面的能力很強。

還有一個框架叫 CrewAI，它特別適合快速原型驗證。它的角色模式非常直覺，開發速度也最快，不過如果你要把它用在生產環境中，還是需要謹慎一些。

最後要提醒大家的是，有些框架其實不太適合用在 CDP 上。比如說 OpenHands，它原本是為軟體開發設計的，不適合用在 CDP 的場景。還有 OpenAI 的 Agents SDK，雖然它是一個很強大的工具，但是因為設計得比較精簡，所以需要開發者自己實作很多東西，這樣反而會增加複雜度。

好的，今天我們來談談如何建構一個多代理人的客戶資料平台，以及在實施過程中需要注意的重點。

首先，在架構設計上，我們建議採用分層混合策略。想像一個金字塔的結構。最頂層是 A2A Protocol，這個協議負責跨系統的代理人互操作，也就是說當你有多個獨立的代理人系統時，它們可以透過這個協議互相溝通。

往下走到第二層，我們放置 LangGraph 作為編排層。LangGraph 的主要工作是什麼呢？它負責建構圖狀的工作流，管理整個系統的狀態，處理條件分支，以及最重要的，實現 Human-in-the-Loop，也就是人類可以在適當的時機介入流程。

再往下是各個功能專門的代理人。比如說，Data Agent 負責資料的整合和處理，Identity Agent 負責身份的解析和驗證，Segmentation Agent 負責把客戶分門別類進行分群。

最底層則是 MCP Layer，MCP 是 Model Context Protocol 的縮寫，它的工作是負責工具的連接。你的資料庫、API、CRM 系統、行銷平台等等，都透過 MCP Layer 來建立標準化的連接。

那麼在具體實施上，我們有幾項建議。首先，編排層一定要用 LangGraph。為什麼呢？因為 LangGraph 能用圖結構來建模整個 CDP 管線，而且 v1.0 的正式版本已經提供了生產環境的穩定性保證，這對企業應用來說非常重要。

其次，工具的整合要使用 MCP。MCP 提供了標準化的方式來連接資料庫、CRM、行銷平台等各種外部系統，這樣可以大大降低整合的複雜性。

第三，如果你需要跨系統的互操作，比如說你的代理人需要和其他部門或外部合作夥伴的代理人系統溝通，那就使用 A2A Protocol。

第四點是關於快速驗證。如果你想快速驗證一下代理人角色的定義是否合理，流程是否正確，可以用 CrewAI 或 Strands 這樣的工具來做原型。不過這是驗證階段，正式上線還是要用更穩定的架構。

在監控方面，一定要使用 LangSmith 或 OpenTelemetry 這樣的工具，確保在生產環境中能夠看清楚系統的每一步運行情況。

最後一點非常重要，要保持框架無關的代理人邏輯。簡單說就是，你的核心業務邏輯不要和具體的框架綁在一起。這樣的好處是什麼呢？如果將來框架選型改變，或者有更好的框架出現，你的核心邏輯可以比較容易地遷移過去，不用全部重新寫。

不過我們也需要提醒你幾個風險。首先，多代理人系統的成本會比較高，因為每次對話都會產生 token 消耗，這個成本需要嚴格管控。我們的建議是從第一天就設定成本上限和監控機制，不要等到成本爆炸了才想辦法。

第二個風險是，不要把框架選型等同於架構設計。框架只是工具，真正決定 CDP 能否成功的，是你對業務領域的理解，以及你的資料治理做得好不好。

最後，這個領域現在還在快速演進中。技術進步很快，所以我們建議你每個季度重新評估一下框架的選型，看看有沒有更好的選擇出現。
