2026 年 2 月 24 日，GitHub 上出現了一個叫 OpenFang 的專案，短短 4 天內就累積了超過 4,000 顆星。這個數字本身不算特別稀奇，但有趣的地方在於它的定位。作者沒有把它叫做「agent 框架」，反而稱它為「Agent Operating System」，這個名字背後想表達的是一個更大的野心。

OpenFang 是完全用 Rust 從頭寫的，包含了 13 萬多行的程式碼，分散在 14 個不同的模組裡，配合超過 1,700 個測試來確保品質。最後編譯出來是一個大約 32 MB 的單一執行檔。這個設計背後的核心邏輯很有意思：它認為 Agent 應該要能夠自主地在排程上運行，而不是被動地等待使用者去觸發它。

過去兩年，各式各樣的 AI Agent 框架如雨後春筍般涌現。LangChain、LangGraph、CrewAI、AutoGen 都各有各的擁護者，但它們基本上都有一個共同的假設，那就是 Agent 被人呼喚出來，完成任務，然後就停止了。

這個模式對於互動式的任務還不錯，但如果你想要的是讓 Agent 持續監控某件事，或者每天自動做研究然後把結果整合成報告，這類需要長期運行的場景，現有的框架處理起來就會有點笨拙。你得自己去搭排程系統，管理狀態，處理失敗恢復的邏輯。而且這些以 Python 為主的框架還會帶來幾個實實在在的問題。

首先是啟動時間的問題。LangGraph 冷啟動大概需要 2.5 秒，另一個 Rust 框架 OpenClaw 更是需要接近 6 秒。對於那些需要定時排程運行的自主 Agent 來說，這個啟動時間就會變成實際的成本問題。

再來是記憶體占用。AutoGen 在閒置的狀態下就要吃掉大約 250MB 的記憶體，OpenClaw 甚至需要接近 400MB。如果你想在資源比較受限的環境，比如說 VPS 或者邊緣節點上部署多個 Agent 實例，這就會變成很大的限制。

另外安全設計的部分也有點薄弱。大多數框架的安全機制還停留在比較基本的 API 金鑰保護，對於 Agent 特有的那些威脅，像是提示詞注入、資訊流的洩漏、代理身份驗證，這些都缺乏對應的防護措施。

還有一個問題是通訊通道的支援有限。CrewAI、AutoGen 這些框架通常不會內建通訊通道的適配器，如果你需要的話就得另行整合。

OpenFang 的切入點就在這裡。它要做的就是讓 Agent 更像作業系統裡的背景程序，而不是那種等著被人下指令的命令列工具。

整個系統由 14 個 Rust 模組組成，我來給各位逐一介紹它們的作用。

首先是最核心的 openfang-kernel，它負責整個工作流的編排，還有工作計量、角色存取控制、任務排程，以及預算追蹤這些系統層級的功能。

然後是 openfang-runtime，這是 Agent 真正運作的引擎。它驅動了 Agent 的循環邏輯，支援 3 種不同的大語言模型驅動模式，還內建了 53 個工具供 Agent 使用，最重要的是它有個 WASM 沙箱環境，確保 Agent 執行時的安全性。

在 API 層面，openfang-api 提供了超過 140 個的 REST、WebSocket 和伺服器推送事件的端點，關鍵是這些 API 都相容於 OpenAI 的標準，所以既有的應用可以直接對接。

通訊方面，openfang-channels 整合了 40 個訊息適配器，讓系統可以跨越各種不同的通訊平台。數據持久化由 openfang-memory 處理，它使用 SQLite 來儲存數據，同時支援向量嵌入功能。

openfang-types 定義了整個系統的核心資料型別，還包含了污點追蹤機制和 Ed25519 簽名驗證。openfang-skills 內建了 60 個技能，同時還整合了 FangHub 市場，讓使用者可以獲取更多額外的技能。

openfang-hands 負責管理 7 個自主的 Hands，以及它們的整個生命週期。openfang-extensions 提供了 25 個 MCP 範本和憑證保管庫功能。

在網路通訊協定上，openfang-wire 定義了 OFP 這個自訂的點對點協定，使用 HMAC-SHA256 進行身份認證，這樣 Agent 之間可以直接通訊，完全不需要依賴中央伺服器。

工具方面，openfang-cli 是命令列介面和守護程序的管理工具，openfang-desktop 是基於 Tauri 2.0 的原生桌面應用程式。openfang-migrate 則是從 OpenClaw 或 LangChain 這些系統遷移過來的轉換引擎。最後 xtask 是負責整個專案的建構自動化。

值得一提的是，每個 Agent 可以在三種執行模式之間切換。Full 模式是完全自主的，Agent 可以獨立決策並執行。Assist 模式是建議模式，Agent 會提出建議但需要人類來確認才會執行。Observe 模式則是純監控，系統只觀察但不實際採取任何行動。

Hands 是 OpenFang 最有特色的設計，它的概念就是把 agent 從被動的工具變成主動的背景程序。OpenFang 預先打包了 7 個自主能力模組，我們叫它們 Hands。

每個 Hand 的結構是這樣的。首先有一個叫 HAND.toml 的清單檔，裡面定義了這個 Hand 需要哪些依賴、它的執行排程，還有它能執行哪些能力。然後是多階段的系統提示詞，字數超過五百字，用來定義 agent 的行為邊界，告訴它能做什麼、不能做什麼。每個 Hand 還有一份叫 SKILL.md 的知識文件，讓 agent 深入理解特定任務的背景。最後很重要的是護欄機制，也就是 Guardrails，這是一套安全機制，防止 agent 越權操作。

現在讓我介紹這 7 個 Hand 分別是什麼。

第一個是 Clip，它的功能是從 YouTube 影片自動剪輯垂直短片，還會自動加上字幕，然後發布到 Telegram 或 WhatsApp。

第二個是 Lead，它每天自動發現潛在的客戶，根據你定義的理想客戶畫像來評分，還會自動去重避免重複，最後把結果輸出到你的 CRM 系統。

第三個是 Researcher，它能從多個來源進行深度研究，會按照 CRAAP 標準來評估信息的可信度，生成的引用格式是 APA 標準。

第四個是 Browser，這個 Hand 做網頁自動化操作。不過有個很重要的護欄，就是涉及購買行為時一定要人工批准才能進行。

第五個是 Twitter，它能自主管理你的 X 或 Twitter 帳號，支援 7 種不同的內容格式，發布前還有審核佇列。

第六個是 Collector，它做開源情報收集，也就是 OSINT，能持續監控你關注的目標有什麼變化，還會做情緒分析。

第七個是 Predictor，這是個超級預測引擎，它會追蹤 Brier 分數來校準推理鏈，最後輸出的預測結果會包含信賴區間。

這些 Hands 都是在排程上運行，不會傻傻地等著用戶輸入，而是持續地建立知識圖譜，把結果推送到 dashboard 上。整個設計哲學很簡單：人設定目標，agent 就持續執行，只有碰到例外情況的時候才通知人。

其中 Browser Hand 那個「購買前必須人工批准」的護欄，其實是一個很務實的安全設計。因為完全自主的 agent 一旦涉及金融交易，就需要有明確的人機協同機制，不能放任它自己決定。

安全架構是 OpenFang 花最多力氣的地方。相比起其他系統，CrewAI 只有一個安全系統，ZeroClaw 有六個，但 OpenFang 實現了十六層的防禦。

首先是沙箱層的防護。OpenFang 使用了 WASM 雙計量沙箱，這裡面有三個關鍵技術。一個是 Fuel metering，用來限制 Agent 能夠執行的計算量；一個是 Epoch interruption，可以在特定的時間點中斷執行；還有 Watchdog thread，像是一個監視線程，防止 Agent 進入無限循環或者耗盡系統資源。除此之外，還有子程序沙箱，透過清空環境變數，然後只有選擇性地傳遞特定的環境變數，來隔離外部程序的呼叫。

再來是審計與追蹤層。OpenFang 使用 Merkle 哈希鏈來記錄審計紀錄，每一個 Agent 的動作都以密碼學的方式連結起來，無法竄改。它還有污點追蹤技術，可以追蹤敏感資訊從輸入到輸出的完整路徑。另外還有 SHA256 工具呼叫迴圈偵測，可以自動熔斷重複的工具呼叫模式。

身份與存取層提供了三種保護機制。第一個是 Ed25519 簽名清單，每個 Agent 的身份與能力集都以非對稱加密簽名。第二個是 OFP 相互認證，P2P 通訊使用 HMAC-SHA256 的 nonce 驗證。第三個是 RBAC 能力閘門，提供細粒度的角色型存取控制。

運行時防護層有四個防禦機制。Prompt Injection 掃描器可以偵測系統提示覆蓋的嘗試和 Shell injection 的模式。SSRF 保護會封鎖私有 IP 範圍和雲端 metadata 端點。秘密清零技術確保 API key 使用後會自動從記憶體清除。還有 GCRA 速率限制，這是一個成本感知的 Token bucket 演算法，可以根據 IP 位址進行追蹤。

最後是會話與路徑層的防護。OpenFang 有七階段的訊息歷史驗證與修復機制。它實現了路徑穿越防止，包含對 Symlink 逃逸的防護。另外還配置了安全的 HTTP 標頭，包括內容安全策略、X-Frame-Options，以及 HSTS 等等。

在整合能力方面，先看大語言模型的支援。目前原生支援 Anthropic、Google Gemini 和 OpenAI 這三個主要驅動。透過 OpenAI 相容的接口，你可以接入包括 Groq、DeepSeek、Together、Ollama、Mistral、Moonshot、智譜 AI，還有火山引擎的 Doubao 等等，加起來超過 27 個以上的提供商。而在模型這個層面呢，整個系統內建了 123 個以上的模型定義，另外還有 23 個不同的別名。特別值得一提的是，針對中文語言模型的支援相對來說滿完整的。

通訊通道方面，這個系統支援 Telegram、Discord、Slack、WhatsApp、Signal、Matrix、Email、Microsoft Teams、Google Chat、LINE、Facebook Messenger、Reddit、LinkedIn、IRC、XMPP，加上另外 25 個以上的其他平台，總共 40 個通訊通道。這個廣度在同類框架中其實相對少見，因為大多數框架預設通常只支援 2 到 3 個頻道而已。

協定支援方面，首先有 MCP，也就是 Model Context Protocol，這個系統既支援作為伺服器的角色，也支援作為用戶端的角色，雙向都可以。還有 Google 的 A2A 協定，是 Agent 之間的通訊標準。另外還有一個叫做 OFP 的自訂 P2P 協定，它帶有 HMAC-SHA256 認證機制。最後，這個系統提供 OpenAI 相容的 API，裡面包含 140 個以上的端點，現在那些已經在使用 OpenAI SDK 的工具可以直接無縫切換過來。

現在讓我們來看看效能的比較。根據 2026 年 2 月的測試數據，我們對幾個主要的框架做了評估。

首先是冷啟動的時間。OpenFang 需要 180 毫秒，OpenClaw 最慢要將近 6 秒，ZeroClaw 則最快只要 10 毫秒，LangGraph 需要 2.5 秒，CrewAI 大約 1.2 秒。

在記憶體的使用上，OpenFang 需要 40 兆，OpenClaw 要 394 兆，ZeroClaw 最省只需要 5 兆，而 LangGraph 和 CrewAI 這兩個 Python 的框架沒有特別的記憶體數據。安裝包的大小呢，OpenFang 是 32 兆，OpenClaw 最大要 500 兆，ZeroClaw 只要 8.8 兆，LangGraph 是 150 兆。

在安全系統的層數上，OpenFang 內建了 16 個，OpenClaw 是 3 個，ZeroClaw 有 6 個，而 LangGraph 和 CrewAI 就各自 1 到 2 個。通訊通道的適配器部分，OpenFang 支援了 40 個，OpenClaw 13 個，ZeroClaw 15 個，但 LangGraph 和 CrewAI 都沒有內建，需要自己去開發。講到自主排程的功能，OpenFang 內建了一套叫 Hands 的排程自主執行架構，其他幾個框架都需要自己去實作。

綜合來看，ZeroClaw 在冷啟動和記憶體使用上表現得最好，但它的安全層數和通訊通道支援得比較少。LangGraph 的強項在於它提供了圖形化的狀態管理，還有對生產環境的穩定支援。CrewAI 則是因為它的角色式 agent 設計直觀好用而受到歡迎。那 OpenFang 呢，它的特色就在於零依賴的部署方式、廣泛的通訊整合能力，加上那套 Hands 的自主排程架構。

需要說明的是，這些都是各家自己報告的測試數據，目前還沒有獨立的第三方機構來驗證這些結果。

快速開始的部分。安裝非常簡單，只需要一行指令，用 curl 下載安裝腳本就可以了。完成初始化以後，web dashboard 會在 localhost:4200 啟動。系統同時還提供了相容 OpenAI 的 API，這樣你就可以輕鬆接入現有的工具鏈。

新增一個 agent 的基本流程其實很直觀。首先你要先建立一個 agent，然後啟用你想要的功能，比如說 Researcher 這個 Hand，啟用以後就可以開始對話或查看執行狀態了。如果你之前是使用 OpenClaw 的使用者也別擔心，提供了一行指令就能完成遷移。

最後來聊聊版本狀態跟風險評估。截到今天三月五號，OpenFang 從二月二十四號的 0.1.0 版本快速迭代到了 0.3.15。開發速度確實很快，但這速度本身也帶來了風險。

先說說優點。因為是用 Rust 開發的，Rust 語言本身對記憶體安全有強硬的保證，這從根本上就避免了一整類的程式錯誤。再加上採用單一二進位檔案的部署方式，大幅度簡化了依賴管理的複雜度。從安全設計來看，它也針對 Agent 特有的攻擊面做了防護。

但風險呢？首先，0.1.0 是第一個公開版本，目前還沒有真實的生產環境部署記錄。再來，13 萬 7 千行程式碼配上 1700 多個測試，測試覆蓋率並不算特別高，有些邊界情況可能還沒被發現。整個專案有 15 個 crates 加上 40 個通訊頻道適配器要維護，這是一個不小的維護量，長期能不能持續更新還要看後續的發展。還有，在 1.0 版本推出之前，有破壞性的變更是預期內的事，所以不要太驚訝。

什麼時候該考慮用 OpenFang 呢？如果你需要把 agent 部署到網路限制比較嚴格的環境，沒辦法裝 Python runtime，那 OpenFang 會是個選項。或是你的使用場景需要接入很多不同的通訊頻道，又或者你本身對 Rust 生態系有偏好。專案的文件建議在生產環境裡，最好是釘選到特定的 commit 版本，而不是用浮動的版本號。

OpenFang 想解答的核心問題是：怎樣才能讓 agent 真正在背景執行，而不是一直等著被喚醒。Hands 有個有趣的設計哲學，把它定位為「能力包」而不是「可配置的 Agent」。這個取捨犧牲了一些靈活性，但換來的是行為邊界更清楚、更可預測，這在生產環境裡往往是更重要的特性。

OpenFang 有三個很明確的技術特點。第一個是零依賴的單一二進位部署方式，第二個是相對完整的安全架構，第三個是 Rust 語言帶來的效能和安全性優勢。當然，選擇 Rust 也縮小了潛在貢獻者的範圍，畢竟 Python 的開發者數量要大得多，這是個需要長期觀察的權衡。

現在的 0.3.15 版本還不適合直接用在關鍵的生產環境。不過，如果你在評估下一代 agent 基礎設施的可能選項，OpenFang 的架構決策還是很值得研究的，無論最後你決定採不採用它。好，今天就聊到這裡。
