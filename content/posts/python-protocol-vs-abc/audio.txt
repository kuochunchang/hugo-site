Python 有兩種定義介面的主要工具：abc.ABC，也就是抽象基底類別，和 typing.Protocol。兩者的目標相似，都是約束型別、描述一個物件應該具備哪些方法，但底層設計哲學截然不同，選錯會讓程式碼在型別安全性和靈活性之間失衡。

要理解兩者的差異，先得弄清楚兩種型別系統的概念。

名義型別，就是型別相容性由明確的繼承聲明決定。你寫了 class Dog(Animal)，Dog 就是 Animal 的子型別，不管兩者的實際結構如何。沒有繼承關係，即使兩個類別有完全相同的方法，型別就不相容。

結構型別則相反，型別相容性由物件的實際結構決定，也就是它有哪些方法和屬性，而非繼承關係。Python 的鴨子類型本來就是結構型別的執行期版本，Protocol 是這個精神在靜態型別層的延伸。

ABC 走名義路徑，Protocol 走結構路徑。這個根本差異決定了兩者在使用上幾乎所有的具體差異。

ABC 這個模組在 Python 2.6 就引入了，目的是為鴨子類型提供正式語義，讓開發者能定義必須被實作的方法，並在違反時提早報錯。

用法很直觀：繼承 abc.ABC，用 @abstractmethod 標記必須實作的方法。如果子類沒有完整實作所有抽象方法，在實例化時就會拋出 TypeError。這個檢查發生在執行期，跟靜態型別工具無關。

ABC 還提供了兩個處理第三方類別的機制。register() 讓你把一個外部類別虛擬地宣告為某個 ABC 的子類，不需要修改那個類別本身。__subclasshook__ 讓 ABC 自訂 issubclass 的判斷邏輯，collections.abc 裡的許多類別就是這樣實作的。

不過這兩個機制都有個共同問題：register 只告訴型別系統「這個類別算是子類別」，但不驗證方法是否真的存在，執行時才會出錯。

ABC 最有用的場景其實是模板方法模式，也就是基類提供共用的邏輯骨架，子類只實作核心部分。比如一個機器學習的 BaseModel，fit 和 predict 方法裡包含了驗證、狀態檢查等共用邏輯，子類只需要實作 _fit 和 _predict。這種模式 Protocol 無法直接複製，因為 Protocol 的隱式實作不繼承任何基類行為。

typing.Protocol 在 Python 3.8 引入，讓鴨子類型能被靜態型別檢查器理解，而不需要修改既有類別的繼承結構。

Protocol 最顯著的特性是隱式實作。你定義了一個 Drawable Protocol，任何有相同方法簽名的類別都自動符合，不需要繼承聲明。Circle 和 Sprite 對 Drawable 一無所知，mypy 仍然接受它們。這種解耦在整合第三方程式庫時特別有用。

Protocol 預設只在靜態分析時生效。加上 @runtime_checkable 可以啟用執行期的 isinstance 檢查，但這裡有個重要限制：執行期檢查只驗證方法是否存在，不驗證型別簽名。就算一個方法的返回型別完全不對，isinstance 仍然回傳 True。mypy 的靜態分析會抓到這種問題，但執行期不會。

Protocol 也支援泛型，可以描述更精確的型別關係。繼承 Protocol 時要特別注意：子 Protocol 必須在基底類別清單中保留 Protocol，否則 mypy 會把它當成普通的名義繼承，失去結構匹配的效果。

Python 標準函式庫在 typing 和 collections.abc 裡預先定義了大量 Protocol，比如 Iterable、Iterator、Sequence、Mapping、Callable 等。這些設計讓函式可以接受任何符合結構的物件，而不是強迫使用者繼承特定基底類別。

兩者的差異整理一下：ABC 是名義型別，需要繼承；Protocol 是結構型別，隱式相容。ABC 在實例化時就強制驗證，Protocol 預設只做靜態分析。ABC 的 isinstance 直接支援且準確，Protocol 需要 @runtime_checkable 且不驗證簽名。ABC 支援透過繼承分享實作，Protocol 不行。整合第三方類別時，ABC 需要用 register，Protocol 只要結構符合就自動相容。

選 Protocol 的情況：整合第三方程式碼，你無法修改那些類別的繼承結構。約束函式參數的行為，但不打算建立繼承體系。測試用的 mock 物件，不需要繼承被模擬的基類。

選 ABC 的情況：有共用邏輯需要透過繼承分享。希望在類別定義階段就強制驗證介面實作，而非等到靜態型別檢查。領域模型中「是一個 X」這個關係有明確語義，名義繼承傳達了更清晰的設計意圖。

Protocol 和 ABC 不互斥，一個常見的組合方式是：用 Protocol 定義外部 API 的型別約束，供呼叫方靜態分析；用 ABC 定義內部繼承體系，讓實作方獲得共用邏輯。比如定義一個 Repository Protocol 作為外部介面，再定義一個 BaseRepository ABC 提供查詢和驗證的共用邏輯。呼叫方只看到 Protocol，實作方繼承 BaseRepository 獲得便利，兩者各司其職。

使用上有幾個常見陷阱值得注意。

Protocol 的意外滿足：某些無關的類別可能剛好符合你的 Protocol，不是因為它們實作了你的概念，只是湊巧有一樣名字的方法。要用更具體的方法名稱來縮小範圍。

Protocol 的執行期限制：@runtime_checkable 的 isinstance 只確認方法存在，不驗證簽名，實際呼叫時才會發現問題。需要嚴格執行期驗證的場合，ABC 更可靠。

子 Protocol 繼承時忘記保留 Protocol 基類：這樣會退化成普通的名義繼承，失去結構匹配效果。

ABC 的 register 帶來的虛假安全感：只告訴型別系統「這個類別算是子類別」，不做任何方法驗證。

ABC 和 Protocol 解決的是不同層面的問題。ABC 是繼承框架，Protocol 是介面描述工具。選擇的核心問題是：你需要的是型別約束，還是行為繼承？前者選 Protocol，後者選 ABC，都需要就都用。

在實際專案中，公開 API 的型別標注用 Protocol，內部實作的基底類別用 ABC，兩者往往各司其職。這也是 Python 型別系統演進的方向：讓靜態型別分析更貼近 Python 本來的鴨子類型哲學，而不是取代 ABC。
