Alistair Cockburn 在二零零五年提出六邊形架構，原名叫做 Ports and Adapters，也就是端口和適配器模式。它的核心主張很直接：業務邏輯不應該依賴任何外部技術。Eric Evans 在二零零三年出版的《領域驅動設計》則主張軟體的結構應當反映業務領域本身。這兩個工具的關注點不同，六邊形架構解決的是技術如何與業務隔離，DDD 解決的是業務如何建模和組織。正因為關注點不同，它們組合起來比單獨使用更有力。

傳統的分層架構從上到下分別是展示層、應用層、領域層、基礎設施層。表面上看這個結構很合理，但有一個結構性缺陷：領域層依賴基礎設施層。資料庫換了、外部服務改了，領域邏輯就得跟著改動。更麻煩的是業務規則會散落在各層之間，要找「訂單取消」這個業務邏輯，可能需要橫跨 Controller、Service、DAO 三個地方才能拼湊出全貌。六邊形架構的出發點就是翻轉這個依賴方向：讓基礎設施去依賴領域，而不是反過來。DDD 提供的則是領域層內部應該長什麼樣子的設計語彙。

六邊形架構的核心結構分成三個區域。最外層是 Adapter，中間是應用服務層，最內部是領域核心。六邊形這個圖形本身沒有特殊意義，Cockburn 選用它只是為了強調應用程式有多個面，可以面向 HTTP、面向資料庫、面向訊息佇列等，而不是傳統分層那種單一的上下方向。核心規則只有一條：所有依賴箭頭指向內部。基礎設施知道領域的存在，但領域完全不知道基礎設施。

Port 是應用程式核心定義的介面，分兩類。Primary Port 也叫驅動端，是外部世界呼叫應用程式的入口，通常對應使用案例或應用服務，比如訂單服務介面，定義了下單和取消訂單等操作。Secondary Port 也叫被驅動端，是應用程式呼叫外部世界的出口，通常對應資料存取或外部服務，比如訂單倉儲介面，定義了儲存和查詢操作。Port 的關鍵特點是使用純領域語言定義，不包含任何框架標注或技術術語。

Adapter 是 Port 的具體實作，負責在業務語言和技術語言之間轉換。Primary Adapter 包括 HTTP Controller、命令列處理器、訊息佇列消費者等，負責將外部請求轉換為對 Primary Port 的呼叫。Secondary Adapter 則包括 JPA 倉儲實作、REST 客戶端等，將 Port 呼叫轉換為具體的技術操作。替換資料庫技術只需換掉 Adapter，領域邏輯完全不受影響。測試時使用記憶體版本的 Adapter，不需要啟動資料庫。

DDD 分為策略設計和戰術設計兩個層次。策略設計解決的是系統應該切成哪些塊。其中最核心的工具是限界上下文，也就是 Bounded Context。同一個詞在不同業務脈絡下含義可能完全不同，「客戶」在訂單系統是購買者，在支援系統是服務對象，在財務系統可能只是計費實體。限界上下文用明確邊界隔離這些不同的語意，讓每個上下文內部的通用語言保持一致和精確。Context Map 則描述多個上下文之間的關係，包括如何整合、哪個團隊擁有哪個上下文、資料流向如何。

戰術設計解決的是在一個限界上下文內部，業務邏輯如何組織。Entity 是有唯一識別碼、生命週期可變的物件，同一張訂單可以改狀態，但它仍然是同一張訂單。Value Object 是沒有識別碼、用屬性值定義相等性的不可變物件，比如金額和地址，兩個金額相同的物件就是相等的。Aggregate 是一群緊密相關的 Entity 和 Value Object 的集合，對外只暴露一個 Aggregate Root，定義了一個一致性邊界，外部程式碼只能操作這個根物件。Domain Service 用來承載那些不自然屬於任何 Entity 的業務操作，比如跨帳戶轉帳。Repository 是 Aggregate 的持久化抽象，對領域層來說就像一個記憶體中的集合。Domain Event 表示業務上發生了什麼事，用來解耦不同 Aggregate 之間的副作用。

六邊形的核心內部正是 DDD 戰術模式的主場。Aggregate 是業務規則的歸宿，以訂單為例，新增商品時的訂單上限檢查、確認訂單時的狀態變更，這些業務規則都放在 Aggregate 的方法內部，而不是放在 Service 裡。Value Object 封裝業務驗證，確保物件本身始終處於有效狀態，比如金額不能為負數這個驗證就放在 Money 類別的建構子裡。Aggregate 也負責記錄發生的 Domain Event，由應用層統一在操作完成後發布出去。

Application Service 是 Primary Port 的實作，負責編排業務流程，但不包含業務邏輯本身。它的典型流程是：從 Repository 取得或建立 Aggregate，呼叫 Domain 方法執行業務邏輯，再透過 Repository 持久化，最後發布 Domain Event。業務規則全在 Aggregate 內部，Application Service 只做流程編排。一旦 Application Service 開始做判斷、做計算，就是業務邏輯外漏的警訊。

在戰略設計層面，每個限界上下文可以實作為一個獨立的六邊形，擁有自己的 Port 和 Adapter。上下文之間的通訊策略直接映射到 Adapter 的實作方式。防腐層是當兩個上下文的模型差異很大時，用 Adapter 作為翻譯層，防止外部模型污染內部模型。Open Host Service 是上下文提供的 API，以穩定的協議暴露功能。Shared Kernel 則是共用的 Value Object 或 Domain Event 定義，通常放在單獨的模組。Port 的方法命名也應該直接使用業務詞彙，這個細節影響代碼是否能作為業務文檔來閱讀。

六邊形架構最直接的好處是讓測試策略變得清晰。領域層的 Aggregate 和 Value Object 沒有任何外部依賴，可以直接實例化進行測試，不需要任何 Mock。應用層測試使用記憶體版本的 Adapter 替換真實的資料庫和外部服務，測試速度快且不依賴環境。針對 Port 介面還可以寫契約測試，記憶體版本和真實的資料庫版本都必須通過相同的測試套件，確保替換時行為一致。

六邊形架構、洋蔥架構和 Clean Architecture 這三種架構的核心原則高度相似：業務邏輯在中心，依賴方向指向中心，外部技術不能影響中心。洋蔥架構可以看作六邊形架構的細化版本，引入了 DDD 的分層術語來組織六邊形的內部。Clean Architecture 則明確定義了使用案例層，讓業務場景更加顯式。實務中，三種架構的特徵往往同時出現在同一個系統中。與傳統分層架構相比，最關鍵的差異就是依賴方向的翻轉：從領域依賴基礎設施，變成基礎設施依賴領域。

這個組合在實際落地時有幾個常見問題。第一是貧血模型，把業務邏輯全放在 Application Service，領域物件只是資料容器，結果 Application Service 變成龐大的事務腳本，領域層名存實亡。第二是 Port 洩漏技術概念，Port 介面中出現分頁物件、事務標注等框架相關的內容，違反了 Port 應技術無關的原則。第三是 Aggregate 設計過大，把太多 Entity 放進同一個 Aggregate，每次操作都要載入大量資料。第四是領域物件直接依賴框架，比如在 Domain Entity 上加了 JPA 的標注，正確做法是分離領域實體和持久化實體，由 Adapter 負責兩者的轉換。第五是 Port 設計過於細碎，為每個外部調用都建一個 Port，導致介面爆炸。第六是跨限界上下文直接引用領域模型，不同上下文應通過資料傳輸物件或 Domain Event 通訊，而不是直接 import 彼此的 Entity。

這個組合有明顯的前期成本，適合業務規則複雜且會持續演進的系統，適合技術棧可能替換的場景，適合需要支援多個輸入通道的情況，以及需要長期維護的系統。相對地，以 CRUD 為主的系統、短生命週期的工具、業務邏輯非常薄的 API，通常不需要這個組合，用簡單的分層就夠了。

六邊形架構和 DDD 的組合核心紀律只有一條：依賴方向永遠從技術指向業務，而不是反過來。六邊形架構定義邊界如何建立、依賴如何流向，DDD 定義邊界內部的業務模型如何設計、不同邊界之間如何協商。維持這條紀律，代碼的演化成本就能保持在可控範圍內，業務邏輯也就成為系統中最穩定、最可測試的部分。
