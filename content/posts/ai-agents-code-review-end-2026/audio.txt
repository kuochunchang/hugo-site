以下是「代理工程的終點：程式碼審查能被消滅嗎？」的語音版本。

AI 代理讓工程師寫程式碼的速度提升了一個數量級，但程式碼審查卻沒有因此消失，反而變得更重。這個矛盾是當前代理工程最核心的困境。

速度陷阱：越快越多，審查越累

2025 年的 DORA 報告記錄了一個讓人不安的現象：組織的 AI 採用率提升九成，同期程式碼審查時間增加了九成一，PR 大小則膨脹了一倍半以上。工程師用 AI 生成程式碼的速度快了，但驗證這些程式碼是否正確的工作量更大了。

LinearB 的數據更直接：AI 生成的 PR 有將近七成被拒絕，手動撰寫的 PR 拒絕率只有一成半。換句話說，代理的輸出品質還遠不足以信任。一篇分析四百七十個 PR 的研究顯示，AI 生成的程式碼比人工撰寫的多將近兩倍的缺陷，而且約有四成五的 AI 生成程式碼含有安全漏洞。

這製造了一個弔詭：AI 把「寫程式碼」的門檻幾乎拉平，卻把「判斷程式碼是否值得存在於主要分支」的成本推高了。

代理 PR 為何被拒：七種失效模式

一篇分析三萬三千五百個 PR 的研究，來自 Claude、Copilot、Codex、Devin、Cursor 這五個主要代理，識別出七種只在代理 PR 中出現的拒絕模式。

第一，對 AI 程式碼的不信任，審查者直接因為「這是 AI 生成的程式碼」而拒絕。第二，過大的貢獻，代理提交的 PR 過於龐大複雜，讓有效審查根本無法進行。第三，實驗性提交，PR 的目的只是評估某個代理是否好用，而非改善專案。第四，情境限制，代理無法取得完成變更所需的必要資訊。第五，決策延遲，改動需要進一步調查。第六，驗證觸發，提交的唯一目的是觸發自動化檢查。第七，不必要的複雜度，解法比問題本身更複雜。

其中值得特別提的是：有六成八的被拒代理 PR 沒有任何明確的拒絕理由，這讓代理很難從失敗中學習改進。

三條通往「消滅審查」的路

儘管現狀如此，確實有幾條技術路徑指向讓傳統程式碼審查失去必要性的未來。

第一條路徑是多智能體審查流水線。最直接的方向是以代理審查代理的產出。一個代理寫程式碼，另一個批評，另一個執行測試，另一個驗證合規性。Qodo 已部署超過十五個專門審查代理，在人工審查介入之前完成初步過濾。Steve Yegge 的「Gas Town」系統是這個方向的具體實作，用市長、野貓、精煉廠這三種角色組成分層架構，精心編排代理的協作流程。

第二條路徑是自動合併代理。GitLab 在 2025 年 11 月推出 AI Merge Agent，讓衝突解決和 PR 測試自動化。早期採用者的數據顯示：自動合併成功率八成五，CI/CD 週期縮短三成。Microsoft 的 AI PR 審查系統在五千個儲存庫中創造了一到兩成的 PR 完成時間改善。但即便是最激進的自動化，也傾向保留人工的最終確認，而非完全移除它。

第三條路徑也是最根本的，來自研究者 Martin Kleppmann 的論述：形式驗證取代人工審查。與其讓人類審查 AI 生成的程式碼，不如讓 AI 在生成程式碼的同時，生成這段程式碼正確性的數學證明。形式驗證的問題歷來是成本太高，比如 seL4 微核心，八千七百行 C 程式碼需要二十人年才能完成驗證。但 Kleppmann 的論點是：LLM 正在讓撰寫證明腳本的成本大幅下降。關鍵在於，即使 LLM 在生成證明時出現錯誤，證明檢查器會拒絕所有無效的證明，迫使代理重試。如果這條路走通了，工程師的角色就從「看程式碼對不對」轉向「寫規格說系統應該滿足什麼性質」，傳統的程式碼審查就此消失。

為什麼現在還沒有消滅

幾個根本性的障礙讓上述路徑尚未到達終點。

首先，問責無法自動化。無論 AI 貢獻了多少，都必須有人類承擔責任。程式碼審查存在的部分原因是確保知識轉移和機構問責，這些功能不是任何工具能完全取代的。

其次，安全性需要威脅建模。觸及身份驗證、支付或不受信任輸入的程式碼，需要的不只是模式比對，而是人類對攻擊面的想像力。

第三，規格本身需要人類判斷。形式驗證路徑的瓶頸在於：你必須先知道「這個系統應該做什麼」，才能寫規格，才能驗證。把需求轉成精確的形式規格，目前仍高度依賴人類判斷。

第四，上下文限制。代理看不到它需要看的東西。程式碼審查中很多關鍵判斷依賴於「這段程式碼在整個系統裡意味著什麼」，這種跨越時間和倉庫邊界的上下文理解，代理目前做得還不夠好。

審查的形態轉變

比較務實的預測是：程式碼審查不會消失，但它的形態會變。多智能體工作流會負責機械性的格式、安全模式、測試覆蓋率檢查；人類工程師的角色則向前移動，從審查已寫好的程式碼，轉向在程式碼生成之前就把規格寫清楚。

測試數據支持這個轉變的可能性。追蹤到的代理 PR 中，測試覆蓋率從三成一增長到五成二。代理愈來愈能自動產生測試，如果測試充分，它本身就是一種審查機制，能在人工看到程式碼之前就攔截大多數錯誤。

消滅程式碼審查的真正路徑，可能不是某個技術直接取代它，而是技術讓「審查時需要人類判斷的比例」越來越小，直到傳統的 PR review 成為只處理架構和業務邏輯這類高層問題的會議，而所有機械性的工作都已被代理吸收。
