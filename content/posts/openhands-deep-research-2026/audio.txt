以下是「OpenHands 深度研究報告：開源自主 AI 編程代理平台全面解析」的語音版本。

OpenHands 是當前最領先的開源自主 AI 編程代理平台，由 All Hands AI 開發維護。簡單來說，它能像人類開發者一樣工作——寫程式碼、執行終端指令、瀏覽網頁，而且還可以從單一個代理擴展到數千個並行代理，真正實現大規模的自動化開發。

提到 OpenHands 的規模，我們來看一些關鍵數據。它在 GitHub 上已經獲得超過 6 萬 8 千顆星，累計下載次數超過 500 萬次，吸引了超過 400 位貢獻者參與開發。融資方面，All Hands AI 已經籌集了 2,380 萬美元的資金。在 SWE-Bench 這個編程代理的性能評測平台上，OpenHands 的驗證分數已經達到 77.6%，這是一個相當亮眼的成績。它同時支援超過 100 個不同的大語言模型供應商，整個項目採用 MIT 授權協議，核心部分完全開源。

OpenHands 的定位是作為商業產品 Devin 的開源替代方案，但它的野心已經遠遠超越了這個簡單的定位。All Hands AI 正在建構一個完整的 AI 軟體開發代理生態系統，涵蓋 SDK、雲端平台、企業部署、還有基準測試排行榜等多個層面，真正把 AI 輔助開發變成了一個完整的平台和生態。

OpenHands 這個專案有個很有趣的演變故事。它的前身叫做 OpenDevin，最初是在 2024 年初由一群學術研究者在 GitHub 上啟動的。他們當時的目標很清楚，就是想創造一個開源版本的 Devin AI。隨著專案越來越受歡迎，發展得越來越快，核心團隊就決定成立了一間叫 All Hands AI 的公司，並把專案正式改名為 OpenHands。

說到創始團隊，他們這邊集合了很多業界和學術界的人才。首先是 CEO Robert Brennan，主要負責公司的策略和營運工作。然後是首席科學家 Graham Neubig，他在自然語言處理和人工智能代理方面特別有研究。首席 AI 官 Xingyao Wang 是伊利諾大學香檳分校的博士候選人，之前還在 Google 的多模態預訓練團隊工作過。還有首席架構師 Ray Myers，專門負責系統架構和技術設計。到目前為止，整個團隊已經成長到 25 人以上，成員來自工程、研究、營運和商業發展等各個領域。

如果看一下專案的發展歷程，2024 年 1 月 OpenDevin 在 GitHub 正式啟動。到了同年 7 月，他們發表了一篇學術論文在 arXiv 上，這篇論文有 24 位作者。9 月的時候，專案獲得了 500 萬美元的種子輪融資，由 Menlo Ventures 領投。進入 2025 年，8 月份他們發表了 V1 架構的重新設計。11 月是個重要的月份，先是獲得了 1,880 萬美元的 A 輪融資由 Madrona 領投，同時也在 SWE-Bench Verified 這個評測上達到了目前最先進的成績。12 月 OpenHands v1.0.0 正式發布，同時推出了 Software Agent SDK。到了 2026 年 1 月，OpenHands Index 的基準測試排行榜上線，2 月又發布了 v1.4.0 版本，這也是本研究進行時的最新版本。

整體來看融資情況，種子輪融資了 500 萬美元，時間是 2024 年 9 月，由 Menlo Ventures 領投，還有 Pillar VC、Betaworks 和 Rebellion 等機構參與。A 輪融資則是 1,880 萬美元，在 2025 年 11 月進行，由 Madrona 領投，Menlo Ventures、Obvious Ventures 和 Fujitsu Ventures 也參與其中。加起來一共融資了 2,380 萬美元。除了這些機構投資人，還有一些很知名的天使投資人支持這個專案，包括 Hugging Face 的共同創辦人 Thom Wolf、Cloudera 的共同創辦人 Jeff Hammerbacher，還有 PyTorch 的創造者 Soumith Chintala。

OpenHands 的 AI 代理具備多項強大的核心能力。首先是程式碼生成與編輯，它可以跨越多個檔案和倉庫來撰寫、修改和重構程式碼。其次是終端指令執行，可以執行 shell 指令、安裝套件和運行測試。OpenHands 還內建瀏覽器功能，讓你可以研究文件、API 和各種外部資源。除此之外，它還能自動化進行程式碼審查並應用反饋。在測試方面，OpenHands 可以生成和維護單元測試與整合測試，幫助提高測試覆蓋率。它也支援文件自動化，能夠生成發行說明和專案文件。從安全角度來看，OpenHands 可以掃描並修復跨倉庫的安全漏洞。最後，它還有一個非常強大的特點就是支援大規模並行任務，可以同時運行數千個代理來處理不同的任務。

OpenHands 提供多種不同的產品形態，讓你可以根據自己的使用場景來選擇。第一個是 Software Agent SDK，這是一個可組合的 Python 函式庫，專門為開發者設計，讓他們可以建構自訂代理。第二個是命令列工具，也就是 CLI，它類似於 Claude Code 或 Codex，可以在終端直接使用。如果你喜歡圖形界面，OpenHands 還提供了一個 Local GUI，這是一個基於 React 開發的桌面應用，使用體驗類似於 Devin 或 Jules。對於團隊協作或企業使用，OpenHands 提供了一個雲端平台，託管在 app.all-hands.dev。最後，對於有特殊安全需求的企業，OpenHands 還提供企業版本，可以在私有 VPC 中使用 Kubernetes 部署，確保安全性和合規性。

OpenHands 最大的優勢在於它的十個關鍵差異化特色。首先，OpenHands 完全開源，採用 MIT 授權，這意味著核心程式碼、代理邏輯甚至 Docker 映像檔都是開源的。第二，它的設計具有模型無關性，透過一個叫做 LiteLLM 的技術，它可以支援超過一百個不同的模型供應商，包括本地模型。第三是雲原生擴展，OpenHands 可以無縫地從運行單一代理擴展到運行數千個並行代理。第四，OpenHands 提供了生產就緒的伺服器，內建了 FastAPI REST 和 WebSocket 伺服器。第五，它支援原生的遠端執行，同一份代理程式碼可以在本地環境或容器化的遠端環境中執行。第六，OpenHands 採用事件溯源架構，提供完整的確定性重播、暫停或恢復功能，以及對話歷史管理。第七，OpenHands 對 MCP 的整合是一等公民級別的，這意味著外部工具伺服器與原生工具受到同等對待。第八，安全優先設計是 OpenHands 的另一個特色，它包含多層安全機制，如風險評估、使用者審批流程和秘密管理。第九，OpenHands 支援子代理委派，這是一種階層式協調機制，可以處理複雜的多步驟任務。最後第十個特色是 OpenHands Index，OpenHands 自建了一個全面的基準測試排行榜，可以持續追蹤各個模型的表現。

好的，讓我們進入技術架構的深度解析。

首先介紹一下 V1 架構設計。這是二零二五年八月推出的新架構，相比之前的單體 V0 架構進行了根本性的重新設計。這個新設計基於四個核心原則。

第一個原則是可選隔離。代理程式預設會在你的本地電腦上執行，不過當你需要更高的安全性時，可以無縫地切換到沙盒環境中運行。

第二個原則是預設無狀態。這意味著系統中大部分的元件都是不可變的，只有一個例外，就是對話狀態這個特殊物件，它是唯一可以改變的。

第三個原則是嚴格的關注點分離。代理程式的核心邏輯與具體的應用完全解耦，無論你是用命令列、網頁介面還是 GitHub App，都共用同一個核心函式庫。

第四個原則是兩層的可組合性。系統提供獨立部署的套件，以及型別化的元件，這樣就能實現安全的功能擴展。

接下來看看模組化的四套件架構。整個系統分為四層。最上面是 openhands.agent_server，這是一個基於 FastAPI 的 REST 和 WebSocket 伺服器。中間層是 openhands.sdk，包含了代理程式、對話、大語言模型、工具和 MCP 這些核心抽象。下面一層是 openhands.tools 和 openhands.workspace 並列，前者提供具體的工具實作，像是 bash 命令、檔案編輯和瀏覽器控制等等，後者則負責提供不同的執行環境，可以是本地、Docker 容器或雲端的託管 API。

再往下看事件溯源狀態管理。系統中的所有互動都被建模為不可變的事件，這些事件會被附加到一個事件日誌裡。基礎的事件分為三種。首先是行動事件，記錄代理程式呼叫工具和它的推理過程。其次是觀察事件，記錄工具執行後的結果。還有壓縮事件，用來存儲對話歷史的摘要。而對話狀態是系統中唯一可以改變的物件，它包含可變的元資料，像是代理程式狀態和統計數據等等，還有一個只增不減的事件日誌，使用 FIFO 鎖來保證執行緒安全。

然後是執行時期的架構設計。系統採用分散式的客戶端伺服器架構。工作流程是這樣的：首先使用者提供一個基礎的 Docker 映像。OpenHands 會在這個映像基礎上建構一個新的映像，裡面包含了 Runtime Client。然後啟動這個容器，並初始化行動執行器。後端伺服器透過 REST API 把要執行的動作傳送到容器裡面。容器內部執行完這些動作後，會返回結構化的觀察結果給後端。

關於沙盒類型，系統支援四種不同的配置。首先是 Docker 沙盒，使用隔離的 Docker 容器，適合本地開發使用，這也是最推薦的方式。其次是進程沙盒，它在本地進程中運行，沒有容器隔離，適合快速測試。還有遠端沙盒，用在雲端部署的場景。最後是 Apptainer 沙盒，這是一種無需 root 權限的容器方案，適合在高性能運算環境或其他受限環境中使用。

Docker 映像的標籤策略採用三層設計來優化建構效能。第一層是版本標籤，這是最通用的方式。第二層是鎖定標籤，基於依賴項的鎖定雜湊值。第三層是源代碼標籤，這是最精確的方式。在進行建構時，系統會依序檢查這三層：先看有沒有現成的可以直接用，省去重建；如果沒有，就進行快速重建；如果還是不行，就進行標準重建；最後才進行完整重建。

還有一個重要的機制叫做上下文視窗管理，也就是 Condenser 系統。隨著對話越來越長，上下文視窗會不斷增長，這樣成本就會上升。為了解決這個問題，LLMSummarizingCondenser 會把舊的事件用 AI 生成的摘要來替換。這些壓縮事件會被儲存在事件日誌中，並在呼叫大語言模型之前自動應用。這套機制可以實現成本降低二倍，同時不會影響效能。

最後是安全架構，系統採用多層的安全機制。首先是風險評估層，由 SecurityAnalyzer 對每個動作進行風險等級評估。其次是審批流程層，使用 ConfirmationPolicy 根據風險等級決定是否需要使用者的手動審批。還有秘密管理層，使用 SecretRegistry 來進行延遲綁定憑證，並對輸出自動進行遮罩。系統還能自動偵測模式，將秘密注入為環境變數。最後在容器隔離層，不管你用的是 Docker 還是 Kubernetes，所有的操作都是以非 root 的使用者身份來執行的。

最後再看一下整體的技術堆疊。後端主要使用 Python，大約佔了百分之七十五點六。前端使用 TypeScript，佔了百分之二十二點三。Web 框架採用 FastAPI。前端框架使用 React SPA。大語言模型的整合方面，系統支援超過一百個供應商，透過 LiteLLM 來統一管理。容器化方面則採用 Docker 和 Kubernetes。

我們來看看怎麼使用 OpenHands 以及各種部署的方式。

首先談快速安裝。安裝 OpenHands 有三種主要的方法。第一種方法是使用 uv 工具，這是目前最推薦的做法。你需要用 uv 來安裝 openhands，指定 Python 版本為 3.12，然後執行 openhands serve 這個指令就能啟動服務了。

第二種方法是用傳統的 pip 來安裝。你只需要安裝 openhands-ai 這個套件，然後同樣執行 openhands serve 就可以了。

第三種方法是使用 Docker。你可以用 Docker 來拉取官方的映像檔，然後啟動容器，讓它對外暴露 3000 這個埠號。

不管用哪一種方法安裝，啟動之後就可以直接在瀏覽器裡面打上 localhost:3000，就能夠使用 OpenHands 的網頁介面了。

在談系統需求的部分。首先你需要一個現代化的處理器，至少要有 4GB 的記憶體。如果要使用沙盒執行的話，Docker Desktop 是必需的。如果你是 Windows 的使用者，你需要安裝 WSL 2 並搭配 Ubuntu。如果你用 macOS，你需要在 Docker Desktop 裡面啟用「Allow the default Docker socket to be used」這個選項。

在 CLI 的部分，有幾個常用的選項。你可以用 --gpu 這個選項來啟用 GPU 的支援。用 --mount-cwd 可以把你目前的工作目錄掛載到容器裡面。還有一個環境變數 OH_SANDBOX_USE_HOST_NETWORK，設定為 true 的話就能夠啟用主機的網路模式。

OpenHands 還提供了 SDK 的程式化使用方式。開發者可以用 Python 的 SDK 來程式化地建立 LLM、Agent 和對話這些物件，然後傳送訊息給代理，讓它自動去執行各種任務。

除此之外，OpenHands 也提供了雲端的 REST API。你可以透過 HTTP 的請求來建立對話、指定程式碼倉庫，然後讓代理在雲端遠端工作。

關於部署選項，我們有三種主要的方式可以選擇。第一種是在本地執行 CLI 或者 GUI，這個版本採用 MIT 授權，完全免費，最適合個人開發者在自己的工作站上面使用。第二種是使用 OpenHands Cloud 這個託管服務，它架設在 app.all-hands.dev 這個網址，是一個 SaaS 的模式，比較適合小型的團隊。第三種是自行部署的企業版本，可以在 Kubernetes 和 VPC 裡面部署，採用商業授權，適合大型企業使用。

最後，OpenHands 還支援一種叫做 headless 的模式，也就是沒有使用者介面的運行方式。這種模式特別適合在 CI/CD 的管線、自動化流程，或者是那些不需要人互動的執行場景裡面使用。

各位聽眾，我們現在來談談 OpenHands 所支援的大語言模型和供應商的選擇。

首先是雲端供應商的部分。如果你想自帶金鑰的話，也就是 BYOK，目前有很多知名的供應商可以選擇。Anthropic 提供的有 Claude 4.5 Opus、Claude Sonnet 4.5 和 Claude Sonnet 4。OpenAI 這邊有 GPT-5、GPT-5.2 Codex、GPT-4o，還有 o4-mini。Google 的 Gemini 系列包括 Gemini 3 Pro、Gemini 3 Flash 和 Gemini 2.5 Pro。DeepSeek 提供 v3.2 和 v3.2 Reasoner。Alibaba 的 Qwen3-Coder 有多種尺寸可選。Mistral 則有 Devstral Small，這是 24B 的參數量，還有 Devstral Medium。

如果你覺得直接跟這些供應商打交道有點麻煩的話，OpenHands 其實提供了一個很貼心的方案，就是透過 OpenHands 的 LLM 供應商來存取這些頂級模型。最棒的是，你可以用成本價、零加價的方式來使用它們，而且不需要為每個供應商單獨設定 API 金鑰，這樣就簡單多了。

那除了雲端服務之外，OpenHands 也支援在本地運行模型。目前完整支援的工具包括 Ollama、LM Studio 和 llama.cpp。如果你是用 AMD 處理器，AMD Lemonade Server 可以在 AMD Ryzen AI 個人電腦上做本地推理。另外，只要是任何與 OpenAI 相容的 API，OpenHands 都可以支援自訂端點。我們特別推薦的本地模型有 Qwen3-Coder-30B-A3B-Instruct 和 Devstral Small，就是那個 24B 的版本。

OpenHands 還有自己特別微調的專屬模型。第一個是 OpenHands LM 32B v0.1，這是一個專門為 OpenHands 微調的開源編程模型。第二個是 OpenHands Critic 32B，這是用來做解決方案選擇的評判模型，它是用 TD 學習的方式訓練的。

最後要提醒大家一點，效能表現最好的還是 GPT-4 以上等級和 Claude 3 以上等級的模型。如果你用比較小的本地模型，像是 llama3.1、codegemma，或者小型的 deepseek-coder，效果通常不會那麼理想。所以在選擇模型的時候，記得要根據你的需求和預算來權衡一下。

現在我們來看看效能表現和基準測試的成果。

首先是 SWE-Bench Verified 的評分結果。Claude Sonnet 4.5 用單次推理就達到了百分之七十二點八，這是一個非常亮眼的成績。相比之下，GPT-5 在推理模式設定為高的情況下，達到了百分之六十八點八，而 Qwen3 Coder 480B 則是百分之六十五點二。如果用 Claude 搭配 Critic 模型，透過推理時間擴展的方式，進行五次嘗試，可以拉到百分之六十六點四。最好的組合配置能夠達到百分之七十七點六的成績。

在 GAIA 基準測試這個評測上，Claude Sonnet 4.5 也是領先的，達到了百分之六十七點九，GPT-5 在同樣的推理設定下是百分之六十二點四。

接下來我要介紹一個全新的評測指標，叫做 OpenHands Index，這個評測平台在二〇二六年一月份才上線，它是業界首個廣覆蓋、持續更新的大語言模型軟體工程評估排行榜。它涵蓋了五大任務類別。第一個是問題修復，用的是 SWE-Bench Verified 這個基準，在這個類別上領先的是 Claude 4.5 Opus。第二個是全新開發，用的是 commit0 這個評測工具，在這裡 GPT 5.2 Codex 表現最好。第三個類別是前端開發，用 SWE-Bench Multimodal 來評測，Claude 4.5 Opus 又是領先者。第四個是軟體測試，基準是 SWT-Bench，Claude 4.5 Opus 依然領先。第五個類別是資訊收集，用 GAIA 來評測，Claude 4.5 Opus 也是冠軍。

從這些成績我們可以看出幾個關鍵的發現。首先，Claude 4.5 Opus 在四個類別中都領先，可以說是目前最全面的強大模型。GPT 5.2 Codex 雖然在某些特定領域表現出色，特別是在那些長期的全新開發任務上，它的成功率明顯更高。另外，DeepSeek v3.2 是目前最強的開源權重模型。值得一提的是，Claude Opus 完成任務的速度是最快的，這得益於它強大的並行工具呼叫能力。還有一個有趣的發現，就是 Gemini 3 Flash 這個相對較小的模型，意外地表現得比更大的 Gemini 3 Pro 還要好。

最後我們來談談推理時間擴展這個技術。OpenHands 他們在實驗中用到了推理時間擴展的技術來達到最佳成績。這個方法的流程是這樣的：首先生成多個代理的解決方案，然後使用一個專門的 Critic 模型，也就是用 Qwen 2.5 Coder Instruct 32B 這個模型，經過 veRL 微調過的版本，來幫忙選擇出最佳的方案。透過這種方式，他們的成績從單次推理的百分之六十點六，提升到了五次嘗試後的百分之六十六點四，這代表著一種對數線性的效能提升趨勢。

接下來我們來看競品比較分析。首先是綜合比較，我列舉了七個主要的 AI 開發平台。在授權方面，OpenHands 和 Aider 以及 Roo Code 都是開源的，而 Devin AI、Claude Code 和 Cursor 則是商業專有產品。講到模型支援，OpenHands 能夠相容一百多種模型，其他的產品要麼就是用多種模型，要麼就像 Claude Code 一樣只支援 Claude。執行環境上，OpenHands 用的是 Docker 或 Kubernetes 沙盒環境，Devin AI 用託管沙盒，而 Claude Code 和 Aider 是在本地終端執行，Cursor 和 Roo Code 分別整合在 IDE 和 VS Code 內。在擴展性上，OpenHands 最強，能夠從單個代理擴展到數千個，其他平台都是單一代理。網頁瀏覽功能只有 OpenHands 和 Devin AI 內建提供。自託管的能力方面，OpenHands、Aider 和 Roo Code 都支援自託管，Devin AI 和 Cursor 則不支援。企業功能上，OpenHands 提供 VPC、RBAC 和稽核等企業級功能，Devin AI 有企業方案，Cursor 有團隊版本。在自主程度上，OpenHands 和 Devin AI 都是完全自主的，Claude Code 和 Cursor 則是互動式的，Aider 是人機協作模式。介面方面，OpenHands 提供 CLI、GUI 和 SDK 三種方式，Devin AI 是 Web IDE，Claude Code 是 CLI，Cursor 是 IDE 外掛，Aider 是 CLI，Roo Code 是 VS Code。

現在來看 OpenHands 跟 Devin AI 的深度對比。在資料主權方面，OpenHands 能夠完全掌控，可以部署在本地或私有雲，而 Devin 的程式碼會傳送到 Cognition 的伺服器。模型靈活性上，OpenHands 可以隨意切換任何大語言模型，Devin 則是被鎖定在 Cognition 的專有模型。成本方面，OpenHands 是免費的自託管加上 API 費用，Devin 則需要支付高額的訂閱費。使用者介面的完善度，Devin 提供了開箱即用的精緻體驗，OpenHands 的功能很完整但看起來比較樸素。在設定複雜度上，OpenHands 需要一些配置和維護工作，Devin 的設定最少。供應商鎖定方面，OpenHands 沒有，Devin 的鎖定程度很高。總的來說，OpenHands 是注重資料主權和靈活性的企業首選，而 Devin 更適合快速原型設計和小型團隊。

比較 Claude Code 和 Cursor 的時候，OpenHands 相比 Claude Code 更加自主，能夠擴展至並行執行多個任務，而 Claude Code 是一個更互動式的開發者工具，但它被鎖定在 Claude 模型。跟 Cursor 相比，Cursor 是一個 AI 增強的 IDE，每月要二十美金，強調的是協作編輯體驗，而 OpenHands 是一個獨立平台，強調的是端到端的自主任務完成和雲端擴展能力。

最後看一下穩定性的評比。在獨立比較中，OpenHands 被評為開源代理中最穩定的系統，獲得了五顆星中的三顆，而 Goose 和 Cline 等競品只獲得了兩顆星。特別是在有明確規格和測試驅動的任務中，OpenHands 的表現最優。

我們來談談 OpenHands 的定價模式。它提供了幾個不同的方案來滿足各種使用者的需求。首先是本地開源版本，這是完全免費的，適合單一使用者，而且對話次數沒有限制。這個版本包含了網頁界面、命令行工具，以及 Git 整合功能。

如果你想要使用雲端服務，OpenHands 也有免費的個人方案。這個方案同樣只支援單一使用者，但是每天有 10 次對話的限制。不過它提供了雲端存取的便利性，還能整合 Jira 和 Slack，而且你可以使用 API。

對於需要更多功能的使用者，Cloud Pro 方案每月 20 美元，也是單一使用者。這個版本給你額外的功能和更多的對話額度。如果你是一個團隊或企業，Cloud Growth 方案每月 500 美元，可以支援無限數量的使用者，對話次數也沒有限制。這個方案還包含了角色權限控制，也就是 RBAC，以及集中的計費管理。

最後是自我託管的企業版本，定價是根據你的具體需求來客製化的。這個版本支援無限使用者和無限對話，還包含了 SAML 和 SSO 單點登入功能、VPC 專用網路，以及大型程式碼庫 SDK 的支援。新使用者現在有一個特別優惠，可以獲得 20 美元的免費額度，這是限時的。另外值得一提的是，OpenHands 與 LLM 的供應商合作，以成本價提供模型存取，也就是說他們不會在模型費用上加價。

現在讓我們看看 OpenHands 在社群中的表現。根據最新的數據，OpenHands 在 GitHub 上已經獲得了超過 68,300 顆星，還有 8,500 多個分支。有 400 多位貢獻者參與這個項目的開發，總下載量已經超過 500 萬次。如果你看主分支的提交記錄，已經累積了 6,119 次提交，發布了 99 個版本。平均每週大約有 60 次新的提交，這顯示了項目的活躍度。最令人印象深刻的是，有 24 位論文作者貢獻了學術研究，而且 OpenHands 已經被頂級機器學習會議 ICLR 2025 接收。

這個項目已經吸引了很多知名企業的關注。蘋果、Google、亞馬遜、Netflix、抖音和 NVIDIA 等科技巨頭的工程師都已經克隆或 fork 了 OpenHands。除此之外，還有一些策略合作夥伴。AMD 與 OpenHands 建立了策略合作關係，特別關注在 AMD Ryzen AI 個人電腦上進行本地部署。C3 AI 則稱 OpenHands 是「遠端自主代理擴展的唯一解決方案」。

在實際應用方面，我們看到一些很有說服力的案例。Flextract 報告說他們 87% 的 bug 工單都在同一天被代理自動解決。美國的行動電信公司 US Mobile 的資深軟體工程師報告說，OpenHands 完成了他們 80% 的工作，原本需要一週才能完成的任務，用 OpenHands 只要大約 5 分鐘就能搞定。

在整合方面，OpenHands 支援了很多常用的開發工具。版本控制方面，它支援 GitHub、GitLab、Bitbucket 和 Azure DevOps。對於專案管理，可以整合 Jira 和 Linear。通訊工具方面有 Slack 的支援。它還能與通用的 CI/CD 系統整合，支援 MCP（也就是 Model Context Protocol）工具協定。在開發環境上，VS Code 可以通過容器內的外掛來使用，計算環境則支援 Jupyter 和 VNC 桌面。

最後，OpenHands 在學術領域也產生了相當的影響力。除了我們剛剛提到的 ICLR 2025 論文接收之外，還有多篇研究論文發表，包括講述 SDK 架構的論文。OpenHands 還與卡內基美隆大學合作開發了一個叫做 Hodoscope 的軌跡分析工具。此外，他們在 OpenHands/benchmarks 這個資源庫公開了基準測試的結果，讓大家可以參考和比較。

接下來談談 OpenHands 已知的限制與挑戰。

首先是任務方面的限制。當開放式的功能設計缺乏明確的規格時，就會導致規劃迴圈和所謂的「上下文盲視」問題。另外，在長期且跨越多個代碼倉庫的變更過程中，因為模型的長期記憶能力有限，代理就傾向於「忘記」最初的指令。現在還有一個重要的模型依賴性問題——那些七到十二B 參數的小型模型效果並不理想，真正能夠可靠運作還是需要四百B 參數以上的前沿模型。

複雜任務的多步推理過程可能會消耗數百萬個 tokens，這直接關係到成本考量。此外，不穩定的測試環境、緩慢的軟體安裝過程或者複雜的服務編排，都可能讓代理偏離軌道。還有一類問題是非顯性的缺陷，意思是代理可能會引入一些通過本地推理就能檢查通過，但在更廣泛的系統背景下就會失敗的細微 bug。最後，在創意設計方面有受限的表現——這個系統在明確規格和測試驅動的工作上表現出色，但面對創意架構決策時就明顯力不從心。

再看技術層面的限制。Docker 映像大約十GB 的體積，安裝時間相當長。還有很多的傳遞依賴項需要處理，同時需要 Docker Desktop 才能使用沙盒功能。Git 憑證處理方面也經常出現一些問題。

根據社群的實際反饋，試用和除錯的過程約消耗二十五美元的 API 費用，一旦系統穩定運行以後，每個 Pull Request 的成本大概在三美元左右，這是以微服務升級為參考場景。

轉向未來潛力。在技術路線圖上，第一個目標是要完成 V1 架構的完善，實現從單體 V0 到模組化 SDK V1 的完全遷移。第二是要優化 Docker 映像，減少那十GB 的體積和安裝時間。第三是要精簡依賴，消除不必要的傳遞依賴項。第四是擴展 OpenHands Index，這是一個持續更新的排行榜，要覆蓋更多的任務類別。企業功能也要加強，擴展 VPC 部署、治理和合規能力。最後還要深化與 AMD 的合作，推進在本地工作站上的代理效能。

潛在的應用場景相當廣泛。大規模的代碼庫遷移可以自動化現代化遺留系統。持續的安全修補能夠進行自主的漏洞偵測和修復。在測試基礎設施上能進行大規模的自動化測試生成和維護。DevOps 自動化可以用於基礎設施即程式碼的管理和部署。還可以進行跨倉庫的協調變更，同時跨越數百個倉庫進行修改。對於隱私敏感的組織，本地 AI 開發功能讓他們能完全在本地環境執行。

從市場潛力來看，OpenHands 正處在 AI 輔助軟體開發的核心增長軌道上。根據 Gartner 的數據，多代理人系統的諮詢量從 2024 年第一季到 2025 年第二季暴增了一千四百四十五個百分點。AI 編程工具的市場規模預計在 2026 到 2028 年間能達到數百億美元。OpenHands 的差異化優勢在於它的開源特性、模型無關的架構、企業級的安全保障，以及大規模的並行能力，這些都形成了獨特的競爭壁壘。商業模式上也很成熟——免費的開源核心加上付費的雲端和企業方案，提供了清晰的可持續商業化路徑。

現在總結一下整體的評價。OpenHands 已經確立為開源自主 AI 編程代理的領導者。它成功地橋接了學術研究，特別是在 ICLR 2025 會議上的成果，和產業應用之間的鴻溝。在技術深度上，它擁有事件溯源架構、模組化 SDK、多層的安全機制。生態的廣度包括超過一百個模型的支援、多平台的整合、活躍的社群。商業可行性方面有明確的免費到付費的階梯定價，還有二千三百八十萬美元融資的背書。實戰方面已經通過了驗證——在 SWE-Bench 上達到了業界最優水平，有企業案例支撐，並且已經積累了超過五百萬次的下載。

根據不同的用戶身份，我們給出幾個建議。如果你是個人開發者，建議使用本地的 CLI 或 GUI，搭配 Claude Sonnet 或 GPT-4o 來處理日常的 bug 修復和測試生成。小型團隊可以考慮使用 Cloud Pro 方案，整合 GitHub 和 Slack，自動化代碼審查和安全掃描。企業團隊應該評估自主託管的企業版本，確保資料主權，並利用大規模的並行能力來處理技術債務。AI 研究者可以基於 SDK 來建構自訂的代理，並參與 OpenHands Index 的基準測試研究。開源貢獻者可以加入超過四百位貢獻者的行列，這個專案積極維護，社群也相當活躍。

不過也有幾個風險值得提醒大家注意。前沿模型的 API 費用可能會累積得相當可觀。複雜或模糊的任務仍然需要人類的監督。Docker 映像的大小和設定門檻可能會影響到初次使用的體驗。目前這個系統主要依賴外部的 LLM 供應商，自有模型的能力仍然在發展中。
