以下是「Harness Engineering 三大核心：Context Engineering、架構約束與垃圾回收的深度解析」的語音版本。

2026 年 2 月，OpenAI 發表了一篇關於 Harness Engineering 的重要文章，揭示了一個令人震驚的工程成就：一支三到七人的工程師團隊，在五個月內以近乎零手寫程式碼的方式，構建出一個超過一百萬行的生產級軟體系統。他們的秘訣，就是 Harness Engineering。

Martin Fowler 的技術部落格將 Harness 定義為三個核心組成：Context Engineering 情境工程、Architectural Constraints 架構約束，以及 Garbage Collection 垃圾回收。這三個組成共同構成了讓 AI 代理能夠高效、可靠地大規模工作的基礎設施。

在深入三大核心之前，我們先來理解 Harness Engineering 與相關概念的層次關係。最底層是 Prompt Engineering，優化的是給大型語言模型的指令文字。再上一層是 Context Engineering，管理所有輸入給模型的資訊，包括文件、工具、記憶體等等。而最上層的 Harness Engineering，則是在模型之外，管理整個系統的行為約束、回饋機制與持續改進循環。簡而言之：情境幫助 AI 代理思考，Harness 則防止系統偏離正軌。

第一核心：Context Engineering 情境工程

Context Engineering 解決的問題是：AI 代理需要在正確的時間獲得正確的資訊。

OpenAI 的關鍵洞見是：不要把一份 AGENTS.md 文件當作大百科全書。他們的做法是將其控制在約一百行，作為一份目錄，指向分散在 docs 目錄中更深層的知識來源。這種漸進式揭露的模式有其根本原因：情境視窗是稀缺資源。一份龐大的指令文件會擠壓任務描述、程式碼和相關文件的空間，導致 AI 代理要麼遺漏關鍵約束，要麼開始針對錯誤的目標進行優化。

靜態文件只是 Context Engineering 的一部分。OpenAI 讓 AI 代理能夠存取動態情境，包括透過 Chrome DevTools Protocol 直接檢查 UI 的渲染狀態、使用 LogQL 讀取應用程式日誌進行自主除錯、使用 PromQL 存取系統指標，以及透過分散式追蹤分析跨服務的請求鏈路。這意味著 AI 代理不只依賴靜態知識，還能像人類工程師一樣，透過觀察系統的實際執行狀態來做決策。

一個關鍵原則是：當 AI 代理卡住時，不是要求它更努力，而是要問：「缺少什麼能力？如何讓這個能力對代理既清晰又可強制執行？」

第二核心：Architectural Constraints 架構約束

架構約束解決的根本問題是：文件中寫下的規則，AI 代理終究會違反。

OpenAI 的洞見是：如果一個規則無法被機械式強制執行，AI 代理就會偏離它。因此，他們將架構決策轉化為自動化的、決定性的強制機制，而非只是文件說明。

OpenAI 為系統定義了一個嚴格的六層架構，依賴只能單向流動：從 Types 類型定義層，到 Config 設定層，到 Repo 資料存取層，到 Service 業務邏輯層，到 Runtime 執行期管理層，最後到 UI 前端展示層。低層可以被高層依賴，但高層絕不能反向依賴低層。這個規則由自動化工具機械式強制：當 PR 提交時，CI 系統會執行架構 linter，一旦發現違規，PR 就會被阻擋，同時附上如何修復的具體指引。

OpenAI 特別聰明的設計是：linter 的錯誤訊息本身就是修復指引。傳統的 linter 只告訴你「哪裡錯了」，而他們的 linter 還告訴 AI 代理「如何修正」，並附上相關文件的連結。工具本身就在教導 AI 代理。

除了層級依賴，OpenAI 還靜態強制執行一系列規則，例如：禁止使用 console.log、所有日誌必須使用結構化格式；Schema 和 Type 的命名必須符合約定；單一檔案不得超過五百行；在所有系統邊界必須使用 Zod 進行嚴格的資料驗證。

第三核心：Garbage Collection 垃圾回收

即便有了完善的情境工程和架構約束，系統的熵增是不可避免的。文件會過時，邊緣情況會突破約束，技術債會悄悄累積。

OpenAI 借鑒了電腦科學中「垃圾回收」的概念：定期讓專門的 AI 代理掃描系統，找出不一致性和約束違規，並自動修復。

垃圾回收代理的工作流程是這樣的：每天定時觸發，代理啟動後掃描文件一致性、架構違規、死程式碼，以及命名慣例偏差。對於找到的問題，它會生成修復方案。低風險的修復自動提交 PR 並合併，高風險的修復則建立詳細的 Issue 等待人工安排。

OpenAI 對技術債的態度是：視其為高利貸，幾乎總是值得立即償還的。在 AI 代理大規模生成程式碼的環境下，若一個壞的模式出現在程式碼庫中，代理會複製它，讓問題以指數速度擴散。垃圾回收的目標是：將人類的「品味」捕捉一次，然後在每一行程式碼上持續強制執行。

三核心的協同效應

這三個組成並非獨立運作，而是相互強化。情境工程提供知識與動態情境，讓 AI 代理理解「應該做什麼」。架構約束機械式確保代理「按正確的方式做」，並透過智慧型錯誤訊息不斷教導它。垃圾回收則定期清理「已經做錯的」，並將發現的問題回饋到情境工程和架構約束中，形成持續改進的閉環。這個循環讓系統具備自我修復能力，不需要大量人工介入。

量化成果

OpenAI 的實驗結果令人印象深刻。超過一百萬行的程式碼，約一千五百個 PR 開啟並合併，團隊從三人增長到七人，平均每位工程師每天完成三點五個 PR。

其他研究也驗證了 Harness 的重要性。一項研究顯示，透過 Harness 優化，同一個模型的成功率從六點七百分比提升至六十八點三百分比，提升了超過十倍。LangChain 基準測試中，透過 Harness 改進，排名從第三十名躍升至第五名。

挑戰與局限性

Harness Engineering 並非萬靈丹。它面臨幾個值得注意的挑戰：前期投入高，OpenAI 花費整整五個月構建 Harness 基礎設施；架構約束能防止結構偏離，但無法保證業務邏輯的正確性；Harness 本身也需要持續維護和演進；另外，若垃圾回收代理的決策出現系統性偏差，可能造成大規模難以追蹤的問題。

結論與展望

Harness Engineering 代表了一種軟體工程哲學的根本轉變：從「工程師寫程式」到「工程師設計讓 AI 代理能夠寫好程式的系統」。

三大核心的本質是：情境工程讓代理知道「做什麼」和「為什麼這樣做」；架構約束機械式確保代理「按正確的方式做」；垃圾回收自動化維護確保系統「持續正確」。

隨著 AI 程式設計代理能力持續提升，Harness Engineering 的重要性只會增加。未來，能夠設計和維護高效 Harness 的工程師，將擁有遠超傳統程式設計師的生產力槓桿。正如 OpenAI 所揭示的：在代理優先的世界中，人類工程師的稀缺資源不再是寫程式的時間，而是思考、判斷和設計系統的能力。
