Claude Code 的 Skills 與 Plugins 機制，為工程團隊進行軟體架構設計提供了一個可程式化的工具層。本文整理這個生態系的技術原理、現有的架構相關工具、實際採用狀況，以及有效運用這些工具的方法。

技術機制：Skills 的運作原理

Skill 是放在特定目錄下的資料夾，核心是一個 SKILL.md 檔案，由 YAML frontmatter 加上 Markdown 指令組成。本質上，Skill 是一種基於提示注入的 meta-tool 系統，而非可執行程式碼。Claude Code 中有一個名為 Skill 的 meta-tool，整合了所有可用的 skill 描述；Claude 透過語言理解選擇合適的 skill，而非依賴嵌入向量或分類器。

Skills 採用漸進式揭露策略：Claude 啟動時只載入各 skill 的 metadata，大約一百個 token；當 Claude 判斷某 skill 與當前任務相關，才載入完整的 SKILL.md，通常不超過五千個 token；附帶的腳本或參考資料則按需載入。這個設計讓數十個 skill 同時安裝而不顯著佔用 context window。Skill 的執行效果是對 Claude 進行暫時性行為修改，注入情境指令，讓 Claude 暫時成為特定領域的專家代理。

Skill 執行有兩種模式。第一種是 Inline 模式，也就是預設模式，在當前 session context 中執行，可存取對話歷史。第二種是 Forked subagent 模式，透過設定 context fork 啟動隔離的 subagent，從全新 context 開始，適合需要長時間、獨立執行的任務。

Plugin 是 Skill 的打包分發形式，適合跨專案、跨團隊共享。Plugin 的 skill 名稱帶有命名空間前綴，例如 superpowers:brainstorming，避免不同 plugin 之間的命名衝突。Skill 有四個存放層級：企業層級適用於組織全體用戶；個人層級適用於個人所有專案；專案層級適用於單一專案；Plugin 層級適用於 Plugin 啟用範圍。

用於架構設計的現有工具

首先來看官方發布的 feature-dev plugin。這是 Anthropic 官方提供的最完整架構設計工具之一，提供七個階段的結構化功能開發流程，核心是三個專用子代理：code-explorer 負責讀取程式碼庫，建立現有架構的理解圖譜；code-architect 負責分析現有模式、抽象層與模組邊界，設計新功能的完整架構方案；code-reviewer 則從架構層面審視實作品質。執行 feature-dev 後，流程會在多個關鍵節點暫停等待人工確認，避免 Claude 在架構選擇上過度自主。

接下來是 Superpowers，這是一個在 GitHub 上累積超過四萬兩千顆星的工作流框架，並於二零二六年一月進入 Anthropic 官方 Marketplace。它的設計思路是將資深工程師做設計決策的思維過程，編碼為可重複的工作流，分三個階段進行。第一個階段是 brainstorming，在實際動筆寫程式之前強制進行設計討論，Claude 讀取真實的 codebase，然後針對需求提問，在設計被確認之前無法進入實作階段。第二個階段是 write-plan，將確認後的設計分解為每個任務兩到五分鐘的實作單元，計劃以 Markdown 檔案形式儲存在磁碟，解決了長 session 中 Claude 失去上下文的問題。第三個階段是 execute-plan，按計劃執行，每個階段完成後有驗證關卡。

在架構模式分析方面，有一個名為 levnikolaevich/claude-code-skills 的收藏庫，收錄了一百零九個生產就緒的 skill，其中架構分析系列包含多個審計工具：pattern-analyzer 對架構模式打分評等；layer-boundary-auditor 偵測分層架構與 IO 隔離的違規；api-contract-auditor 識別層洩漏與缺少的 DTO；dependency-graph-auditor 檢查耦合度指標、偵測依賴循環；project-structure-auditor 驗證物理目錄結構是否符合框架慣例。

在 ADR 管理方面，blueprint-derive-adr 可以從現有程式碼庫自動逆向工程，產生 MADR 格式的架構決策記錄。流程包括使用 Explore 代理掃描框架和資料層的模式、計算與既有 ADR 的衝突分數、自動生成涵蓋語言選擇和框架選擇等標準 ADR 集，以及雙向更新機制。另有 ADR Creator 在標準模板之外額外加入 AI 導向欄位，指定未來 AI 代理應如何實作這些決策，確保架構決策可被後續 AI 工具繼承。

claude-code-spec-workflow 實作了需求到設計到任務到實作的四階段流程，維護持久性的舵手文件，包括 product.md、tech.md、structure.md，跨 session 保存技術棧選擇、檔案組織原則、命名慣例，提供架構決策的持續性情境。

在多代理架構分工方面，wshobson/agents 系統採用分層角色設計：Opus 模型負責處理 Backend Architect、Database Architect、Cloud Architect 等關鍵設計決策；Sonnet 模型負責 Kubernetes Architect 的部署架構；Haiku 模型負責具體實作。這種分層設計讓昂貴的 Opus 模型專注在需要深度判斷的架構問題，降低整體成本。

在架構文件與逆向工程方面，O'Reilly Radar 記錄了一個案例：以 Mermaid 圖表搭配結構化 metadata 為現有系統建立端對端流程文件。一旦建立文件，Claude 在後續跨 repo 的架構分析中可以直接對照這些流程，不需要從零開始分析程式碼。這套做法的限制是文件容易因程式碼更新而過時。

採用狀況

截至二零二六年初，claude-plugins.dev 社群登記的 skills 已超過一千五百三十七個，其中一千兩百九十八個是獨立 skill，兩百三十九個嵌入在 plugin 中。GitHub 上有超過六千九百五十九個相關 repository。整個生態系從二零二五年十月公開 beta 到二零二六年初，不到半年累積超過九千個 plugin。

從社群實際採用的場景來看，有幾個主要模式：逆向工程既有架構、新功能的架構設計、ADR 標準化，以及依賴健康監測。其中依賴健康監測的做法是將架構審查 skills 整合進 PR 流程，在合併前自動偵測耦合惡化、層邊界違規等架構債。

方法分析

Skills 最大的價值在於把隱性的架構決策變成可複用的工具。把慣例寫進 skill 後，每次 Claude 處理相關工作時都能自動套用。這和把同樣的指令放在 CLAUDE.md 的差異在於：CLAUDE.md 裡的內容每次 session 都載入，適合全局不變的規範；skill 按需載入，適合有特定觸發條件的工作流。

角色分離的多代理設計也是一個有效的方法。feature-dev 的三代理架構，讓 Explore 代理以唯讀方式理解程式碼，code-architect 專注設計，code-reviewer 獨立評估，防止了 Claude 在單一對話中混淆角色。

在 Claude Code 中以唯讀的 Plan Mode 進行初期架構分析也是關鍵實踐，先讀取、先映射、先提出方案，再執行修改，降低了 Claude 在理解不充分的情況下直接動工的風險。

這個工具範式也有清楚的限制。首先是靜態分析邊界：執行時期行為、依賴注入模式、動態派發在靜態分析中都是不可見的，Skill 不會像 ArchUnit 那樣產生確定性結果。其次是幻覺問題：從程式碼逆向工程出來的架構文件並非百分之百準確，需要接受一定程度的不精確，用定期重新生成和 CI 驗證來維護準確性。第三是 context budget 上限：安裝過多 skill 時，超出預算的 skill 會被自動排除在 Claude 的可見範圍之外。第四是時間低估：靜態分析得出的架構工作量估計通常需要乘以兩到三倍才接近現實。

實踐建議

從最高投報率的項目開始建立 skill：先把現有的 coding standards 和架構原則寫成設定為不可使用者呼叫的 skill，讓 Claude 在相關工作時自動套用；再從社群的 ADR skill 入手，根據團隊格式調整；最後針對複雜的跨 repo 分析，建立使用 Forked subagent 模式加 Explore 代理的 skill，讓分析在隔離情境中進行。

Skill 設計有幾個關鍵原則：精準的 description 決定 Claude 是否自動觸發；有副作用的操作要設定禁止自動觸發；SKILL.md 控制在五百行以內，詳細資料放在子目錄；allowed-tools 只列實際需要的工具，架構分析類通常只需要 Read、Grep、Glob。

CLAUDE.md 定義全局背景，skill 定義特定工作的操作流程，兩者互補。在 CLAUDE.md 中適合記載整體技術棧、模組邊界、命名慣例，以及重要架構決策摘要。

ADR 文件與程式碼脫鉤是長期維護的主要痛點。可行的對策是季度性重新執行 ADR 逆向工程工具並比對差異，在重大 PR 的 CI 流程中加入架構層面的靜態分析，並用 ADR 的 AI 導向欄位明確指定未來 Claude 在修改元件時應遵守的約束。

最後，如果有一套在多個 repository 使用的架構原則，將 skill 打包為 plugin 透過 Git 分發給團隊，比要求每個人手動複製更容易維護版本一致性。

Claude Code 的 Skills 與 Plugins 機制在架構設計上提供了一個務實的工具層，它不試圖取代架構師的判斷，而是降低把架構知識帶入每次對話的摩擦力。真正的價值在於把隱性知識編碼化，讓架構原則和設計慣例在每次 Claude 工作時持續生效，而不是停留在某位資深工程師的腦子裡。
