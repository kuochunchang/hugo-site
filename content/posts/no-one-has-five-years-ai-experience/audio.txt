二零二二年十一月三十日，ChatGPT 上線。GPT API 和 Claude API 在二零二三年三月才正式開放。到二零二六年三月，大語言模型應用開發這個領域的實際歷史不超過三年。

當一間公司在招聘條件裡寫下「需要五年 AI 開發經驗」，這個要求在邏輯上無法被滿足。不是因為人才稀少，而是因為那段時間根本不存在。這個事實有幾個不那麼顯而易見的推論，值得仔細想清楚。

先釐清一個常見的混淆：AI 研究是有歷史的。達特茅斯會議、專家系統、深度學習革命、Transformer 架構——這些都有幾十年的積累。但「AI 應用工程師」和「AI 研究員」是兩種截然不同的角色。

二零二二年之前，AI 開發對工程師而言通常意味著傳統機器學習：訓練分類器、調超參數、設計特徵工程。這套知識體系有十幾年的歷史，和現在的大語言模型應用開發幾乎是兩個不同的領域。

大語言模型應用開發的核心技能——提示詞工程、RAG 架構、向量資料庫、函數呼叫、Agent 設計、輸出評估——這些概念在二零二二年底之前幾乎都不存在於主流軟體工程的日常討論中。RAG 的論文發表於二零二零年，但直到 GPT API 開放後才真正成為工程師每天討論的話題。

即使是最早進入這個領域的人，現在也只有大約三年的開發經歷。根據 Axial Search 對超過一萬個 AI/ML 職位的分析，百分之七十八的職缺要求五年以上工作經驗。問題不在這個數字本身，而在於這五年的定義是什麼。大多數情況下，招聘方自己也說不清楚。

更麻煩的是，這三年的積累本身也在快速貶值。Skillable 的分析指出，AI 技能的半衰期可能短至十五個月。Harvard Business Review 的研究顯示，技術領域整體的知識折舊速度已從過去的十到十五年壓縮至兩點五年，AI 領域則更快。

LangChain 是個典型案例。二零二三年初，幾乎所有大語言模型應用教程都以 LangChain 為基礎。但在短短幾個月內，LLMChain 和 AgentExecutor 先後被棄用。有工程師回報，他們寫的代碼在一週內就因為框架更新而完全無法運行。Octomind 在生產環境使用 LangChain 十二個月後決定移除它，理由是高層抽象反而阻礙了功能實作。他們的結論是：在快速演化的領域裡，為底層建構塊設計薄抽象，比用全面框架包裝一切要安全得多。

二零二三年初積累的「LangChain 使用經驗」，到了二零二四年很可能成為負資產——需要先忘掉舊的範式，才能接受新的設計思路。

這個循環並未停止。二零二四年，LangChain 推出 LangGraph 重構架構；二零二五年，OpenAI 推出了新的 API 規格，Context Engineering 取代 Prompt Engineering 成為核心概念。AI 領域論文的提交速度在二零一九年已超過每小時三篇，在生成式 AI 興起後還在加速。工程師花三個月學習的「最佳實踐」，六個月後可能已被更好的方案取代。

三年並非什麼都沒有留下。從集體層面看，業界正在從大量失敗的生產部署中提煉出可複用的模式。Martin Fowler 的團隊總結出九個核心模式：直接提示、評估、向量嵌入、RAG、混合檢索、查詢重寫、重排序、護欄、微調。二零二三年初，大多數人甚至不清楚「evaluation strategy」在大語言模型系統中意味著什麼，因為當時還沒有足夠的生產案例可以學習。

但這些積累存在於整個社群的集體記憶中，而不是集中在某個個人的簡歷上。這也解釋了為什麼在 AI 開發領域，社群討論區和 GitHub Issues 的重要性遠超過工作年資。知識還在被快速生產，還沒有充分整理進書本或課程，更沒有轉化為可量化的認證標準。

從個人層面看，有幾類知識確實會隨時間積累。工程判斷力——知道什麼時候不用 AI。生產環境中，一個複雜的 RAG 管線不一定比精心設計的全文搜尋更好。失敗模式的識別——見過系統在生產環境崩潰的工程師，知道大語言模型的不確定性如何在邊界情況下被放大。跨模型遷移的成本感知——把 GPT-4 換成 Claude 不只是換個 API 端點，提示詞行為差異和上下文限制都可能導致系統行為改變。

Andrej Karpathy 把大語言模型描述為「不規則的智能」，他的隱喻是「召喚幽靈」而不是「培養動物」。傳統軟體的確定性假設——同樣的輸入產生同樣的輸出——在大語言模型系統中不成立，需要建立完全不同的設計直覺。

但這些知識的共同特徵是：它們在兩到三年內就可以積累到足夠深度，額外的年份邊際效益遞減。更重要的是，它們來自生產環境的實際失敗，不是「做了多少年 AI」自動獲得的。

關於招聘市場，IntuitionLabs 的分析顯示，AI 工程師需求比整體科技職位增長高出百分之十六，但同時百分之三十二的求職者虛報自己的 AI 能力。形成邏輯並不複雜：需求端用不切實際的年資要求篩選候選人，供給端的應聘者則把傳統背景包裝成 AI 開發經驗。誠實描述自己有兩年大語言模型開發經驗的工程師，在自動篩選系統中被過濾；自稱五年的候選人進入面試。

Stack Overflow 的數據顯示，二十二到二十五歲的軟體工程師就業人數從二零二二年底高峰下降了近百分之二十。公司在裁減初級人才的同時，又聲稱找不到有足夠 AI 開發經驗的工程師。分析師 Pawel Brodzinski 直接點出了問題核心：我們仍然沒有替代判斷力的方法。真正短缺的不是代碼生成能力，而是判斷力。

Fastly 的研究顯示，資深工程師在 AI 時代的生產力是初級工程師的兩點五倍。但這個優勢的來源值得注意：不是 AI 開發多年積累，而是他們知道要問什麼問題、知道如何在不確定性中做出決策。轉移的不是領域知識，而是工程判斷力本身。

這意味著一個二零二一年入行的工程師和一個有十五年 Web 開發經驗的工程師，在構建大語言模型系統的起點上幾乎相同。大多數技術領域都有很深的護城河，但大語言模型應用開發的護城河還沒有被完全挖出來，最佳實踐還在被集體發現，沒有人可以憑藉多年的專業積累來睥睨新入場者。

Gartner 在二零二四年的報告估計，百分之八十的工程師需要在二零二七年前完成技能升級以適應 GenAI 工作流程。這個領域的「資深從業者」本身也在持續學習，在不同的子領域上都是某個版本的初學者。

「沒有人有五年的 AI 開發經驗」這個命題，指向的不只是時間線的算術問題，而是一個更深的挑戰：當一個領域變化的速度超過知識系統化的速度，「經驗」本身的含義需要被重新定義。

更有用的評估維度可能是：這個人從失敗中學習的速度？他們是否真的把系統部署到生產環境？他們對最新工具的判斷是基於實驗還是道聽途說？他們能解釋為什麼某個設計選擇在三個月後被自己推翻嗎？

對個人職涯而言，這個現實既解放又嚴峻。解放，是因為這個領域不是靠累積年資就能建立護城河的地方，判斷力和學習速度比履歷的年資更重要。嚴峻，是因為沒有地方可以躺在經驗上，停止學習就是開始落後。

整個行業正在以集體的方式學習如何在一個不確定的介質上構建可靠的系統。在這個過程中，謙遜和實驗精神，比任何職稱或年資都更接近「有經驗」的定義。
