在構建 AI 代理系統的時候，面對複雜任務有兩條截然不同的路可以走：一條是把單一代理的能力做強，透過 MCP 掛載各種工具；另一條是讓多個代理協作，組成所謂的 AI 蜂群。表面上這兩種方式都能處理複雜任務，但它們在架構上的差異，決定了各自在哪些場景會成功，在哪些場景會踩坑。

兩條擴展路徑

第一條路叫垂直擴展，也就是強化單一代理的能力。Model Context Protocol，簡稱 MCP，是這條路的核心協定。Anthropic 在 2024 年 11 月推出 MCP，提供了一套標準化的介面，讓代理可以透過統一的方式連接外部工具、資料庫和各種服務。到 2025 年底，MCP 已經成為事實上的標準，SDK 每個月的下載量超過九千七百萬次，OpenAI、Google、Microsoft 都跟進支援了。

透過 MCP，一個代理可以同時具備搜尋網路、查詢資料庫、執行程式碼、操作瀏覽器的能力，而且這些全部都在同一個 LLM 實例裡發出請求、在同一個 context 裡累積推理。

第二條路叫水平擴展，就是 AI 蜂群的概念。讓多個具備不同專業能力的代理，透過某種協調機制共同解決問題。蜂群的靈感來自生物界的集體行為，螞蟻、蜜蜂、鳥群，每個個體遵循局部規則，整體展現的能力卻超過任何個體。

架構上的結構差異

先看單一代理加 MCP 工具的架構。核心是一個 LLM 實例，維護一條連續的推理鏈。工具呼叫是這條推理鏈的延伸，代理思考、決定用哪個工具、執行、把結果折回同一個 context 窗口，然後繼續推理。整個過程是線性累積的，每次工具呼叫的結果都附加進去，完整的推理歷程對代理始終可見。

蜂群架構則把複雜任務拆解成可由不同代理承擔的子任務，每個代理有自己獨立的 context 和職責範圍。協調方式有幾種：有調度器的蜂群，由一個 orchestrator 代理負責任務分解和結果整合，下面的 worker 代理負責執行；去中心化蜂群，代理之間透過 handoff 機制傳遞控制權，沒有中央指揮；還有並行管線，把完全無依賴的任務同時發給多個代理執行。

在協定層面，現在的蜂群架構通常用 MCP 處理工具連接，用 A2A 協定，也就是 Agent-to-Agent 協定，處理代理之間的通訊。這兩個協定是互補關係，MCP 解決代理怎麼使用工具，A2A 解決代理怎麼跟其他代理溝通。

本質差異：推理能不能分散

理解兩種架構的根本差異，關鍵在於推理能不能分散。

單一代理是單執行緒推理，一條思考序列從頭到尾，工具只是行動的延伸，知識整合、決策、工具選擇全在同一個注意力機制下運作。蜂群是多執行緒推理，多條思維鏈並行或串接，最終匯聚成答案，但這本身就帶來了協調的複雜度。

有個簡單的比喻：MCP 代理像一個配備了各種專業工具的全才工程師，蜂群像一個工程團隊。全才工程師在獨立工作時通常比團隊快，但當任務規模超過個人能處理的範圍，或需要同時推進多個方向時，團隊是唯一的選擇。

Context 管理與 Context Rot

Context 管理是選型時非常實際的考量。LLM 的注意力機制計算成本隨著序列長度二次方增長。Chroma Research 在 2025 年的研究測量了十八個 LLM，發現隨著輸入長度增加，模型效能越來越不穩定，這個現象叫做 context rot，也就是 context 腐化。

對單一代理來說，多步驟工作流的每次工具呼叫都把結果塞進 context，很快就累積大量中間狀態，代理開始忽略早期的重要資訊，工具選擇錯誤增加，需求也可能在推理過程中被遺忘。

蜂群裡每個代理的 context 是聚焦的，研究代理只看研究相關的內容，寫作代理只看已整理好的資料，每個代理的認知負擔都小得多。

另外，當單一代理掛載超過十六個工具時，選擇工具的準確率開始下降。如果你的工具清單一直在膨脹，這往往是任務本身需要拆解的信號。

平行性與效能數據

單一代理無法真正並行工作，每次工具呼叫都要等上一次結果回來才能繼續。蜂群可以讓多個代理真正並行執行，把一百頁文件分成十份，派十個代理同時分析，這在單一代理架構中做不到。

Google Research 的量化實驗給出了很清楚的邊界：在可並行的任務上，多代理協作帶來百分之八十點九的效能提升；但在需要嚴格邏輯順序的任務上，各種多代理變體反而導致百分之三十九到七十的效能下降。並行化本身不創造價值，只有任務結構允許並行時才有意義。

錯誤傳播與成本

兩種架構的錯誤傳播模式也截然不同。Google Research 量化了這個差異：獨立的多代理蜂群，錯誤放大係數是十七點二倍；有中央調度的多代理，錯誤放大係數是四點四倍；而單一代理是基準值。

去中心化蜂群最嚴重，因為每個代理可能各自產生不一致的輸出，而下游代理盲目接受。有調度器的蜂群好得多，orchestrator 扮演驗證關卡，阻止錯誤級聯傳播。

成本方面，研究資料顯示單一代理每天基準成本大約 0.41 美元，而基本的多代理設置約 10.54 美元，差距超過二十五倍。低效實作的多代理系統，token 使用量可比單一代理多出七十七倍。Mayo Clinic 的案例顯示多代理在複雜診斷任務上準確率從七十四%提升到八十九%，但成本代價也相應很高。

選型框架

怎麼選？有幾個判斷條件。

選單一代理加 MCP 的情況：任務有清楚的依序邏輯，不同工具的結果需要互相參照，整體工作量在 context window 內可以可靠完成，對成本敏感，速度要求高，或者需要簡單的 debug 路徑。典型場景是客服問答機器人、文件摘要、程式碼補全。

選 AI 蜂群的情況：任務可以分解為真正相互獨立的子任務，任務規模超過單一 context 能可靠處理的範圍，需要多個專業領域同時深入，或者容錯要求高、單點失敗不可接受。典型場景是大型程式碼庫分析、長篇研究報告生成。

Google Research 開發了一個基於任務屬性的預測模型，能以八十七%的準確率判斷適合哪種架構。關鍵維度是工具密度和任務可分解性：高可分解性加高工具密度，適合蜂群；低可分解性，無論工具多少，優先考慮單一代理。

混合架構

大多數真實的生產系統是混合架構，不是純粹的任何一種。

最常見的模式：一個有工具的 orchestrator 代理負責路由和決策，下面掛幾個專門化的 worker 代理，每個 worker 有自己的工具集。orchestrator 本身是完整的單一代理，但它把部分工作外包給子代理而非直接執行。

業界還常見四角色分工：分流代理負責理解請求和路由，檢索代理負責查詢資料，執行代理負責寫操作和 API 呼叫，審核代理負責驗證結果。這四個角色各自是配備工具的單一代理，合在一起構成一個協作蜂群。

舉個具體例子。每日競品監控報告這個任務可以這樣設計：orchestrator 代理分析今日競品清單，決定任務分解策略；並行資料蒐集代理群，每個代理負責一個競品同時蒐集資訊；然後分析代理整合所有結果做跨競品比較，這一步需要在同一個 context 裡關聯所有資訊，所以用單一代理；最後報告生成代理產出結構化報告。資料蒐集適合蜂群，跨競品分析適合單一代理，在錯誤的地方強行並行化只會讓架構更複雜。

實作注意事項

在蜂群架構裡，token 成本最容易失控。幾個控制手段：代理在交棒前壓縮自己的執行歷史，而不是把完整 context 傳遞下去；每個 orchestrator 管理五到六個 worker 效率最佳，超過後協調效率遞減；對重複使用的工具結果設置快取，避免相同查詢消耗多份 token。

任務邊界的劃定是蜂群成功的關鍵設計決策。原則是最小化共享可變狀態，如果兩個代理需要修改同一個文件，這個任務切分本身就是錯誤的；如果兩個代理需要頻繁溝通，考慮把它們合併為一個代理。

錯誤處理必須明確設計，不能事後補救。在每個 handoff 節點加入驗證邏輯，是把錯誤放大係數從十七點二倍壓低到四點四倍的主要手段。

結論

單一代理加 MCP 和 AI 蜂群的本質差異，在於推理能不能分散。

當問題需要連貫的跨步驟推理，當上下文關聯是答案的關鍵，當任務步驟有嚴格的因果依賴，集中在單一 context 裡推理是正確的選擇。MCP 讓這個代理能連接任意數量的工具，垂直擴展能力的上限。

當問題可以分解為真正獨立的子任務，當規模超過單一代理的處理能力，當需要同時推進多個方向，蜂群讓多個代理並行運作，水平擴展吞吐量的上限。

在實踐中，生產系統通常從單一代理加工具開始，在效能瓶頸出現時才引入多代理。選擇的核心始終回到任務本身的結構：它是線性的還是可分解的，這個問題的答案決定了架構應該長什麼樣子。
