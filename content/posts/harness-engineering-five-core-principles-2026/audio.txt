以下是「Harness Engineering 五大核心原則：AI 時代工程師的全新工作方式」的語音版本。

背景與概述

二零二五年末，OpenAI 發布了一篇引發廣泛討論的工程文章，描述了一個前所未有的實驗：一支僅三到七人的小型團隊，在五個月內利用 Codex 這個 AI Agent，在完全不手寫任何原始碼的情況下，完成了一個超過百萬行生產代碼的應用系統。平均每位工程師每天合併三點五個 Pull Request，效率遠超傳統開發模式。

這個實驗催生了「Harness Engineering」這個新概念。Harness 原本是馬具或駕馭工具的意思，在這裡指的是圍繞 AI Agent 所建立的整套系統框架，包括限制、回饋迴路、可觀察性基礎設施與品質強制機制。換言之，工程師不再直接撰寫代碼，而是設計讓 AI Agent 能夠可靠運作的環境。

Martin Fowler 進一步定義：Harness Engineering 是代碼之外的一切，是架構約束、回饋迴路、可觀察性工具，以及讓 Agent 持續可靠運作的強制執行機制的總和。

OpenAI 從這次實驗中提煉出五大核心原則，本文將逐一深入剖析這五大原則，並提供具體的實踐方式與真實案例。

原則一：設計環境，而非撰寫代碼

第一個原則是「設計環境，而非撰寫代碼」，英文是 Design the Environment, Not the Code。

傳統工程師的價值在於寫出好代碼，但在 Harness Engineering 中，工程師的核心職責轉變為為 AI Agent 設計可運作的環境。當 Agent 卡住時，工程師的問題不是「如何讓 Agent 更努力嘗試」，而是「這個環境缺少了什麼能力？」

OpenAI 工程團隊的核心觀察是：Agent 的效能瓶頸幾乎從來不是模型的智識能力不足，而是環境的可見性與可操作性不夠。

在具體實踐上，有三個重要方向。

第一是工具化，而非指令化。當 Agent 需要某項能力時，工程師的任務不是自己寫一個工具，而是讓 Agent 自己生成那個工具。OpenAI 團隊讓 Codex 撰寫 Codex 所需的命令列工具、MCP 伺服器，以及調試腳本。工程師的工作變成設定邊界與驗收標準，而非具體實作。

第二是環境隔離與按需啟動。每個 Git 工作樹都有一個對應的獨立開發環境，包含完整的資料庫實例、服務容器與監控堆疊。Stripe 的做法更進一步，為旗下的 AI Agent 群組準備了預熱的隔離容器，讓 Agent 能夠在毫秒內啟動並行任務而不互相干擾。

第三是能力缺口診斷流程。當一個 Agent 連續失敗三次以上時，工程師建立一個診斷清單：Agent 是否能看到系統狀態？Agent 是否有適當的工具可以操作環境？Agent 是否清楚地知道任務的驗收標準是什麼？每一個缺口，都是環境需要改善的地方。

真實案例：Stripe 為其 AI Agent 開放了超過四百個內部工具，透過 MCP 伺服器提供統一介面。工程師在 Slack 頻道中貼出任務描述，AI Agent 自動領取、執行，最後提交 Pull Request。這套系統每週產出超過一千個合併 PR。

原則二：機械式強制架構

第二個原則是「機械式強制架構」，英文是 Enforce Architecture Mechanically。

傳統架構管理依賴代碼審查時的人工判斷。在 AI Agent 每天產出數十個 PR 的高吞吐量環境下，這種方式根本無法擴展。Harness Engineering 的解法是：不要告訴 Agent 要有好品味，要讓壞品味變得不可能發生。

在具體實踐上，有四個重要方向。

第一是嚴格的分層依賴規則。OpenAI 團隊定義了固定的六層架構，代碼只能向前依賴，不允許逆向引用，順序是 Types、Config、Repo、Service、Runtime 到 UI。任何違反這個依賴方向的代碼都會被自動 Lint 工具攔截，無法通過 CI/CD 流程。

第二是能自我說明的 Linter 錯誤訊息。傳統 Linter 只告訴你哪裡錯了，OpenAI 的自訂 Linter 在報告錯誤時同時提供如何修正的完整指引。錯誤訊息本身就是上下文，讓 Agent 在下一次嘗試時能直接學習並修正。

第三是結構測試。除了 Linter，OpenAI 也撰寫了靜態分析測試，在 CI 流程中自動驗證整體依賴圖的合法性。這些測試不針對業務邏輯，只針對架構結構本身。如果某次合併破壞了分層原則，即使功能測試全數通過，結構測試也會阻止合併。

第四是「品味不變量」的編碼。除了結構性規則，OpenAI 還將一些主觀的代碼品質標準，例如命名慣例、副作用隔離原則，編碼為可機械驗證的規則，確保代碼庫整體風格的一致性。

真實案例：終端模擬器 Ghostty 的開源社群維護了一份詳細的 AGENTS.md 文件，其中逐條記錄了歷史上每一次 AI Agent 犯過的錯誤以及對應的修正規則。這份文件本身就是機械式強制架構的一部分，讓後續的 Agent 不再重蹈覆轍。

原則三：讓版本庫成為唯一真相來源

第三個原則是「讓版本庫成為唯一真相來源」，英文是 Make the Repository the Single Source of Truth。

Agent 的知識來源只有它能讀取的文字。存在於 Slack 對話、Google Docs、口頭討論中的知識，對 Agent 而言等同於不存在。Harness Engineering 要求工程師將所有決策、設計原則、架構規範都以版本控制的形式儲存在代碼庫本身。

在具體實踐上，有四個重要方向。

第一是 AGENTS.md 作為地圖，而非百科全書。OpenAI 的 AGENTS.md 只有約一百行，其功能是導航地圖，告訴 Agent 各類資訊在哪裡可以找到，而非試圖把所有資訊塞入一個巨大的指令文件。

第二是 docs 目錄作為系統記錄。所有設計決策、API 規範、功能需求文件都存放在代碼庫的 docs 目錄中，且受到版本控制。Agent 在執行任務前，會讀取相關的 docs 文件作為其上下文基礎。設計規格書是 Agent 行動的唯一授權來源。

第三是任務計劃的版本化。每個大型功能的開發計劃以 Markdown 文件形式儲存，包含任務分解、驗收標準、已知風險和歷史決策記錄。Agent 在執行子任務時，會更新計劃文件以反映進度，形成持續的「文件即代碼」工作流。

第四是背景 Agent 的文件維護。OpenAI 部署了專門的背景 Codex 任務，定期掃描 docs 目錄，識別過時或與實際代碼不符的文件，並自動提出更新的 PR 供人工審核。

真實案例：Cloudflare 的工程師分享了一個關鍵洞見：在讓 Agent 開始寫代碼之前，花更多時間制定詳盡的計劃文件，能夠大幅降低後續的修正成本，同時也減少了 Token 消耗。在代碼產生之前，真相就已經存在於版本庫中了。

原則四：將可觀察性連接到 Agent

第四個原則是「將可觀察性連接到 Agent」，英文是 Connect Observability to the Agent。

人類工程師在調試時會開瀏覽器開發者工具、查看日誌終端、觀察指標儀表板。AI Agent 也需要這些能力，但它需要的是機器可讀的介面。可觀察性在 Harness Engineering 中的角色是：讓抽象的驗收標準變成可量測、可驗證的具體目標。「啟動時間控制在八百毫秒以內」必須有一個 Agent 能夠查詢並驗證的量測機制，否則這個目標對 Agent 而言毫無意義。

在具體實踐上，有四個重要方向。

第一是每個工作樹一套獨立的可觀察性堆疊。OpenAI 為每個並行運作的 Agent 工作樹配置了獨立的日誌系統、指標系統和分散式追蹤。Agent 可以直接執行 LogQL 和 PromQL 查詢來獲取系統狀態，而不需要人工協助解讀日誌。

第二是 Chrome DevTools Protocol 整合。OpenAI 將這個協議整合進 Agent 的工具集，讓 Agent 能夠截取 UI 的 DOM 快照、監聽網路請求、測量頁面效能指標、模擬使用者操作並驗證視覺結果。這讓「UI 看起來是否正確」這種原本需要人眼判斷的問題，變成了 Agent 可以機械化驗證的問題。

第三是量化的驗收標準。所有任務的驗收標準必須以可量測的形式定義。例如，不說「頁面載入要快」，而是說「首次內容繪製小於一點二秒，最大內容繪製小於二點五秒」。Agent 在完成任務後，會主動執行這些查詢來自我驗證。

第四是可觀察性驅動的 Bug 重現。Bug 報告必須附帶足夠的可觀察性數據，讓 Agent 能夠在隔離環境中精確重現問題。

真實案例：OpenAI 工程師給 Codex Agent 設定了一個任務：將應用程式啟動時間控制在八百毫秒以內。工程師事先建立了自動化的啟動時間量測腳本、PromQL 查詢模板，以及效能回退警報規則。Agent 在每次代碼修改後自動執行量測，根據數據判斷優化是否有效，形成完整的可觀察性驅動開發循環，完全不需要人工介入。

原則五：對抗熵增

第五個原則是「對抗熵增」，英文是 Fight Entropy。

在高速生成代碼的環境下，代碼庫的品質衰退速度與代碼生成速度成正比。如果不主動對抗，AI 生成的代碼會隨著時間積累大量「AI 糟粕」，也就是冗餘代碼、不一致的命名、過時的文件、微妙的架構漂移。

OpenAI 在實驗初期，工程師每週五要花費約百分之二十的工作時間手動清理 AI 產生的低品質代碼。這個發現催生了第五個原則：品質維護本身也必須被自動化，讓清理能力能夠與生成能力同步擴展。

在具體實踐上，有四個重要方向。

第一是黃金原則的代碼化。OpenAI 將代碼品質標準直接「燒錄」進版本庫，形成一套可機械執行的規則集合，包括命名慣例、副作用隔離規則、模組大小限制、測試覆蓋率最低門檻等。

第二是背景垃圾回收任務。OpenAI 建立了一套定期自動執行的 Agent 任務，類似程式語言的垃圾回收機制，包含文件一致性掃描、架構約束違規掃描、品質分數更新，以及自動生成並合併低風險的代碼改善。

第三是 CI/CD 作為熵的防火牆。每個 PR 都必須通過一套熵檢測關卡才能合併，包括 Linter 規則、架構測試、測試覆蓋率門檻，以及文件更新驗證。

第四是高吞吐量的 PR 流程設計。OpenAI 採用了最小化阻塞的合併流程：短生命週期的 PR、自動化的基礎品質門檻，以及高層次的人工審查。核心理念是：在 AI 開發環境下，PR 是廉價的，修正也是廉價的，阻塞才是昂貴的。

這個轉變的關鍵洞見是：清理工作的規模必須與代碼生成的規模相匹配。在 AI 每天生成數千行代碼的環境下，手動清理從根本上就無法跟上。

結論與展望

Harness Engineering 的五大核心原則代表著一個根本性的典範轉移：工程師從代碼的作者，轉型為讓 AI Agent 能夠可靠產出代碼的系統設計師。

這個轉型的核心洞見是：AI Agent 的效能上限不是模型本身的智識能力，而是環境的設計品質。一個設計精良的 Harness，能讓普通的 AI 模型完成驚人的成就；一個設計糟糕的環境，則會讓最強大的模型也頻繁失敗。

未來，Harness Engineering 的模式將從個別團隊的實踐，演變為組織層面的標準化基礎設施，如同 Git 如今已成為連接開發環境與 CI/CD 系統的標準基礎設施一樣。

對於正在思考如何擁抱 AI 輔助開發的工程組織而言，這五大原則提供了一個清晰的行動框架：不只是導入 AI 工具，更是重新設計整個工程系統，讓 AI 成為可靠的、可信任的、自律的開發夥伴。
