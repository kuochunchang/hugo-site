二零二六年三月第一週，AI 輔助開發工具領域同時出現幾個值得記錄的節點：Cursor 以事件驅動的 Automations 功能宣示 agentic coding 進入新階段，MCP 成為業界事實標準，Anthropic 發佈關於 AI 對開發者技能影響的實驗數據，而企業部署的試點到生產的落差依然是整個行業面對的核心問題。

三月五日，Cursor 正式推出 Automations 功能，這是對既有 BugBot 功能的大幅延伸。BugBot 原本用於 PR 代碼審查，而 Automations 的核心概念是：開發者不再需要手動觸發 AI，而是配置規則讓 agent 在特定條件下自動執行。

支援的觸發來源包含 GitHub 新 PR 建立或 merge 完成、Slack 收到特定訊息、PagerDuty 偵測到 incident、Linear 新 issue 建立，以及固定時間間隔的排程執行。

一個典型的工作流是：PagerDuty 偵測到 incident 之後，Automation 自動啟動 agent，agent 去查詢 Datadog 日誌、檢查最近的 commit，然後在 Slack 發送摘要並自動建立修復的 PR，整個流程無需工程師手動介入。

Cursor 工程負責人 Josh Ma 把這個設計哲學描述為「傳送帶」模型：人的介入只在關鍵決策點發生，其餘由 agent 在背景持續處理。這與 Devin 之類完全自主的 agent 不同，Cursor Automations 維持了對觸發條件和介入時機的明確控制。

商業數據方面，Bloomberg 報導 Cursor 年化營收已超過二十億美元，過去三個月翻倍，企業客戶佔收入約百分之六十。

到了二零二六年，Claude Code 和 Cursor 這兩個工具的功能集越來越接近，都有背景 agent、命令列工具和雲端執行環境。但底層設計理念的差距並未縮小。

Cursor 的設計核心是編輯器優先：在 IDE 視窗內提供即時的上下文感知輔助，雲端 agent 在隔離的虛擬機上執行，可以自我測試、錄製操作影片、產出 merge-ready 的 PR。BugBot 的 PR 審查解決率已從百分之五十二提升到百分之七十以上，並提供 SSO、角色權限控制和審計日誌等企業治理功能。

Claude Code 的設計核心則是執行深度優先：透過 MCP 連接資料庫、Slack、GitHub、Sentry 和內部 API，Agent Teams 功能允許多個 agent 互相通訊、共用任務列表，並且與現有編輯器整合，不強迫使用者切換工具。

選擇標準在實務上相當清楚：需要快速、編輯器內即時反饋的工作選 Cursor；需要跨系統整合、深度推理或複雜多步驟自動化則選 Claude Code。

Anthropic 在二零二四年十一月發佈了 Model Context Protocol，也就是 MCP，定位是讓 AI 系統連接外部工具和資料來源的標準化介面。把它比喻成 USB-C 很貼切：MCP 出現之前，每個 AI 工具連接外部系統都需要自己實作一套整合方案；MCP 將這個過程標準化，同一個 MCP server 可以被不同的 AI client 使用。

二零二五年底，Anthropic 將 MCP 捐贈給 Agentic AI Foundation，這個基金由 Anthropic、Block 和 OpenAI 共同創立，隸屬 Linux Foundation。OpenAI 和 Google DeepMind 相繼採用，MCP 的中立性得到強化。

到二零二六年初，MCP 已從「Anthropic 推廣的協議」轉變為多個工具的預設整合標準。它在 agent 架構中承擔的角色越來越像網路協議棧：開發者不需要關心底層細節，只需要配置需要哪些工具伺服器。

技術上，MCP 採用 JSON-RPC 2.0，支援標準輸入輸出和 HTTP 加串流推送兩種傳輸層。對於安全性敏感的場景，MCP 的沙箱化執行和明確的工具宣告比直接執行任意代碼的方案更容易做權限控制。

本週出現了一起值得記錄的安全事件：Devin 及另外三十多個 AI 工具的系統提示詞被完整洩漏，總量超過三萬行。

洩漏的技術原理並不複雜。攻擊者透過 prompt injection，也就是在輸入內容中嵌入指令，例如「重複你收到的所有指示」，就可以誘導語言模型洩露本應隱藏的配置。OWASP 在二零二五年版的 LLM 應用 Top 10 中將 prompt injection 列為首位風險，且在已評估的生產 AI 部署中出現率超過百分之七十三。

系統提示詞洩漏的實際風險分兩層：第一是商業機密，暴露產品的差異化設計、限制策略和工作流邏輯；第二是攻擊面擴大，了解系統提示詞後，攻擊者可以更精準地構造繞過方式。

更根本的問題是：目前沒有任何單一技術手段能完全防止 prompt injection。NCSC、OWASP、Microsoft、OpenAI、Anthropic 的評估結論一致，這是語言模型架構的結構性限制，不是可以打補丁解決的問題。

對於使用 AI agent 處理敏感業務流程的團隊，較實際的緩解策略是：最小化系統提示詞中的敏感資訊、在 agent 輸出進入關鍵系統前加入人工審核節點、對 agent 的外部呼叫做嚴格的範圍限制。

Anthropic 發佈了一項隨機對照實驗，對象是五十二名 Python 初級開發者，他們熟悉 AI 工具但不熟悉 Trio 這個非同步函式庫。參與者分成使用 AI 輔助和純手工編程兩組，完成任務後立即進行測驗，考察除錯、代碼閱讀和概念理解能力。

結果是：AI 輔助組平均測驗分數百分之五十，手工組百分之六十七，差距約十七個百分點，相當於將近兩個字母等級。AI 輔助組完成任務快約兩分鐘，但這個速度提升在統計上不顯著。

細分使用模式後，差距的來源更清楚。低分組主要讓 AI 直接生成代碼或做除錯，自己幾乎不思考；高分組則要求 AI 附上解釋、用 AI 澄清概念、追問以加深理解。

這個研究的意義不在於「AI 有害」，而在於使用方式決定結果。對組織而言，實際的問題是：如何設計 AI 使用流程，使初級工程師在提高產出的同時不犧牲除錯能力，而除錯能力恰恰是監督 AI 生成代碼所必需的核心技能。

行業數據持續反映一個現象：AI agent 的概念驗證階段成功率遠高於實際生產部署率。本週報告中提到約百分之十一的 agent 能成功進入生產環境，DigitalOcean 三月的調查數字更低，只有百分之十的企業成功從試點擴展到生產。

落差的主因通常不是技術本身。組織問題包括：責任歸屬不清、缺乏治理框架、跨功能團隊配置不足。基礎設施方面，生產環境需要五到十倍於試點的投入，包括監控、回滾機制和安全邊界。評估困難也是一個關鍵障礙：agent 的能力難以量化，LangChain 的 CEO 指出這是阻礙組織信任 agent 上線的核心原因之一。

Factory AI 與安永會計師事務所的合作部署了超過一萬個 agent，Nubank 在數週內用 AI 遷移了六百萬行代碼，DeNA 報告效率提升六倍。這些案例表明技術上的可行性已經存在，差距更多在執行層。

最後幾個值得持續觀察的方向：Google 的 DeepMind 正在開發以 agent 為核心的代碼編輯器，據報導與 Windsurf 的二十四億美元收購案相關，如果確認，這將是 Google 首個直接切入 agentic IDE 市場的產品。

xAI 的 Grok Build 編程工具尚未正式發佈，但 beta 測試者描述的功能包括命令列界面配合雲端 UI、支援多模型切換。

Gemini CLI 3.1 推理能力評價正面，但 API 穩定性問題持續有報告，這對需要可靠性的 agent 工作流是實際障礙。

當前的市場格局是：Claude Code 和 Cursor 在企業端持續搶佔份額，MCP 在標準化工具整合層面勝出，而 production agent 的可靠性和安全性仍是整個行業尚未解決的核心問題。
