安德列・卡帕西在二零二六年三月六日發布了 autoresearch，這個項目讓 AI 程式開發代理人在你睡覺時自主修改訓練程式碼、跑實驗、評估結果、留下改善的版本，然後繼續循環。發布後一週內累積超過三萬個 GitHub 星號，社群隨即分叉出至少五個變體。

傳統機器學習實驗的流程大致是：提出假設、修改程式碼、跑訓練、看指標、決定要不要保留、再提出下一個假設。這個循環中，GPU 訓練時間往往是幾小時起跳，一個工作天能跑的實驗數量有限。卡帕西的切入點是：如果把訓練時間壓縮到固定五分鐘，並且把修改程式碼、跑實驗、決策這三步交給 AI 代理人自動執行，那麼一台 GPU 一夜可以完成約一百次實驗迭代。研究人員只需要在前期定義目標，然後早上來看結果。這個想法本身並不新奇，AutoML、神經架構搜索等領域已經有多年的自動化嘗試。但 autoresearch 的特點在於它的設計極度精簡：整個系統核心只有約六百三十行 Python，依賴一個通用的程式開發代理人，例如 Claude，而不是專用的搜索演算法。

整個系統圍繞三個文件組織。第一個是 prepare.py，這個文件負責數據預處理，包括下載數據集、訓練分詞器，詞彙表預設八千一百九十二個詞。它也定義評估函數、固定驗證集數據分片，以及設置訓練時間預算三百秒，不含啟動時間。代理人被明確禁止修改 prepare.py，固定的評估函數確保每次實驗的指標具有可比性，也防止代理人透過修改評估方式來作弊提高分數。

第二個是 train.py，這是整個系統唯一可以被代理人修改的文件。它包含完整的 GPT 模型定義、優化器配置和訓練循環。代理人可以調整的內容包括模型層數、詞彙表大小、序列長度、批次大小，以及注意力機制的模式。優化器採用混合設計，AdamW 處理嵌入層和純量，Muon 負責權重矩陣。

第三個是 program.md，這是人類與代理人的溝通界面。它告訴代理人可以修改哪些部分、如何執行實驗、如何判斷成功，還有一條「永不停止」的指令，避免代理人在無人值守時自行中止。卡帕西把這個文件稱為研究組織的程式碼。你不是在寫 Python，你是在用 Markdown 告訴代理人如何做研究。

每次迭代的流程是這樣的：代理人先讀取 program.md 和當前的 train.py 狀態，接著提出一個假設，比如增加暖機步數可能改善收斂，然後修改 train.py，執行訓練三百秒，解析日誌中的驗證損失指標。如果有改善，就用 git commit 保留變更；如果沒有改善，就用 git reset 回滾，回到上一個最佳狀態。然後重複。這個設計是一種有限制的爬山算法。每個實驗從當前最佳狀態出發，單次修改，評估後二選一。成功的改動會疊加，失敗的修改不留痕跡。Git 在這裡不只是版本控制工具，而是系統狀態管理的核心機制。

系統選用「每字節的驗證位元數」作為唯一優化目標，英文縮寫是 val_bpb，有幾個原因：它不受詞彙表大小影響，使得不同架構的實驗可以公平比較；因為每次實驗的時間預算相同，這個數值直接反映模型在等量計算下的效率；語言建模的這項指標改善，通常也能遷移到更大的模型上。

卡帕西分享了兩天自主運行的結果。一個十二層的模型，代理人生成了約七百次程式碼修改，其中大部分在五分鐘訓練後被丟棄，約二十次改動被保留。在 nanochat 排行榜上，驗證損失在八十九次實驗後從零點九九七九降至零點九七七三，在一百二十六次實驗後降至零點九六九七。訓練至 GPT-2 水準的時間從兩點零二小時縮短到一點八零小時，效率提升約百分之十一。

另外，Shopify 執行長 Tobi Lutke 也報告了他使用這個框架的結果：跑出了百分之十九的驗證分數改善，代理人優化後的較小模型甚至超過了以標準方式配置的較大模型。在約七百次嘗試中只有二十次成功，成功率大約百分之三。這並不意味著代理人的決策品質差，而是在探索一個複雜的超參數空間時，大多數方向本來就沒有改善。傳統人工調參同樣面對這個現實，只是每次嘗試的成本高得多。

機器學習實驗有特定的失敗模式：GPU 記憶體不足、張量形狀不匹配、損失值變成無效數字。autoresearch 的設計中，這些錯誤訊息會回饋給代理人，讓它理解失敗原因並修正程式碼，而不是直接崩潰中止。這是讓系統能夠在無人值守情況下穩定運行的關鍵。

發布後一週，社群出現了多個針對不同場景的分叉。autoresearch-mlx 適配了 Apple Silicon 的 MLX 框架；autoresearch-win-rtx 支援 Windows 加 RTX GPU；autokernel 將相同邏輯應用於 GPU 核心優化；autoresearch-at-home 增加了分散式多節點協調；pi-autoresearch 則擴展到任意指標，不限於語言建模。官方版本目前只測試了單張 NVIDIA H100，這些社群版本解決了硬體多樣性的問題。

autoresearch 示範的是一個有嚴格約束的成功場景。它能做的是在已有訓練框架上做超參數搜索、架構微調、優化器配置探索。它不能做的是推導新的數學理論、設計全新的研究方向、跨越既有框架的限制。這個邊界在設計上是刻意的：固定評估函數防止代理人作弊；限制只能改單一文件確保每次差異可審查；固定時間預算讓不同實驗可以比較。

autoresearch 更像是一個高效的自動化調參工具，而不是自主做科學研究的完整示範。真正的研究還包括：提出有創意的假設、解讀違反直覺的結果、決定整體研究方向。這些目前仍在 program.md 的人類撰寫部分。

autoresearch 在工程設計上有幾個值得借鑒的選擇。單一指標加固定預算：把複雜的優化問題簡化為一個可比較的數字，讓代理人有明確的反饋信號。可逆操作：利用 git 現有的語義做狀態管理，不需要額外設計回滾機制。分離可變與不可變：prepare.py 鎖定評估，train.py 開放修改，讓系統既有彈性又有防護。Markdown 作為意圖表達：program.md 讓人類用自然語言表達研究策略，而不需要編程，降低了使用門檻。這種設計思路可以遷移到其他自動化場景：任何有明確指標、可逆的修改操作，以及固定評估成本的循環，都可以考慮類似的架構。

最後是安全考量。代理人在執行過程中會讀取訓練程式的輸出並帶入上下文，這產生了提示注入的潛在風險：惡意構造的訓練輸出可能影響代理人的後續行為。此外，讓一個代理人長時間自主修改並執行程式碼，在沒有明確沙盒的情況下存在一定的系統風險。這些問題在 autoresearch 的設計中沒有明確解決，在生產環境中部署類似系統時，需要額外考慮隔離措施。
