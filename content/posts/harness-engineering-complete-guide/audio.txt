以下是「Harness Engineering 完整指南：AI 時代軟體工程的典範轉移」的語音版本。

二零二五年底，OpenAI 發布了一篇引起廣泛討論的工程部落格文章，描述了一個前所未有的實驗。一支只有三到七個人的小型團隊，在短短五個月內，使用 Codex AI Agent 這個工具，在完全不手寫任何原始碼的情況下，完成了一個超過一百萬行的生產代碼應用系統。這個效率有多驚人呢？平均每位工程師每天合併三點五個 Pull Request，遠遠超過傳統的開發模式。

這個實驗催生了一個新的概念，叫做「Harness Engineering」。「Harness」原本是指馬具上的線束，用來控制和引導馬匹的一套結構性裝置。OpenAI 借用這個比喻，將其定義為：讓 AI 代理保持高效、可靠運作所需的一切約束、工具、文件與回饋迴路。換句話說，工程師不再直接撰寫代碼，而是設計讓 AI Agent 能夠可靠運作的環境。

Martin Fowler 在探索生成式 AI 的系列文章中進一步定義，Harness Engineering 就是代碼之外的一切。它是架構約束、回饋迴路、可觀察性工具，以及讓 Agent 持續可靠運作的強制執行機制的總和。他的技術部落格分析師 Birgitta Böckeler 將 Harness 定義為三個核心組成部分：Context Engineering 也就是情境工程、Architectural Constraints 架構約束，還有 Garbage Collection 垃圾回收。

現在讓我們來看 Harness Engineering 與相關概念之間的層次關係。最底層是 Prompt Engineering，它優化的是給 LLM 的指令文字。中間層是 Context Engineering，它管理所有輸入給 LLM 的 token，包括 RAG、工具、記憶體和各種 schema。最上層是 Harness Engineering，它則是在 LLM 之外，管理整個系統的行為約束、回饋機制與持續改進循環。簡單來說，Context 幫助 Agent 思考，而 Harness 防止系統偏離正軌。

那 Context Engineering 和 Harness Engineering 有什麼具體的差異呢？首先在關注點上，Context Engineering 著重於優化輸入給 AI 的上下文，而 Harness Engineering 則是設計整個開發環境的生態。在工具方面，Context Engineering 使用 Prompt 設計和 RAG 技術，Harness Engineering 則是用架構約束、Linter 和 CI/CD 這些工具。在應用對象上，Context Engineering 針對的是單次的 AI 互動，而 Harness Engineering 處理的是長期、自主的 Agent 工作流。在熵的處理上，Context Engineering 較少涉及，但 Harness Engineering 則是主動管理代碼庫的衰退。

Birgitta Böckeler 明確指出，Context Engineering 其實是 Harness Engineering 的一個子集，但 Harness Engineering 還涵蓋了架構約束的機械性執行與熵的主動管理。

OpenAI 的百萬行代碼實驗給了我們一個非常有趣的案例研究。他們的工程師團隊設計了一個強制函數，也就是說，他們完全不允許任何手動輸入代碼。第一筆 commit 是在 2025 年 8 月底開始的，然後短短五個月後，整個代碼庫就已經超過了一百萬行。而且這不是一個實驗性的項目，而是真的成功上線成為了一個實際運作的 beta 產品。

這個實驗揭示了一個深刻的認識，就是當 AI Agent 成為主要開發者的時候，整個代碼庫的架構需要首先為 AI 的可讀性而優化，而不是傳統上為人類工程師來優化。整個倉庫實際上變成了 Codex 推理業務領域的主要資訊來源。

讓我用一些具體的數據來說明成果有多驚人。代碼庫規模超過一百萬行，涵蓋了應用邏輯、基礎設施、工具和文件。在這個過程中，他們開啟並合併了約 1,500 個 pull request。團隊規模也從最初的 3 個人成長到了 7 個人。而這就是最驚人的地方了，平均每位工程師每天能產出 3.5 個 PR。隨著團隊人數增加，整個團隊的產出甚至呈現出線性，甚至是超線性的增長。

外部的研究也驗證了這些優化方法的有效性。Can.ac 的研究顯示，透過 Harness 的優化，同一個模型的成功率從 6.7% 提升到了 68.3%。LangChain 的基準測試中也看到類似的改進，透過 Harness 的調整，排名從第 30 名一舉躍升到了第 5 名。

今天我們來聊聊 OpenAI 從實驗中總結出來的五大核心原則，這些原則構成了 Harness Engineering 的方法論基礎。

首先是第一個原則，叫做「設計環境，而非撰寫代碼」。這裡的核心想法是什麼呢？在傳統的工程師工作中，我們的價值體現在寫出優質的代碼。但在 Harness Engineering 的時代，工程師的角色發生了根本的轉變。現在我們的核心職責變成了為 AI Agent 設計一個能夠正常運作的環境。當 Agent 遇到問題卡住的時候，我們要問的不是「怎麼讓 Agent 更努力地去嘗試」，而是「這個環境到底缺少了什麼能力？」

OpenAI 的工程團隊做過一個重要的觀察，他們發現 Agent 的效能瓶頸幾乎從來不是模型本身的智識能力不足，真正的問題其實是環境的可見性和可操作性不夠。用他們的話來說，當代理掙扎時，我們應該把它看作一個信號，找出缺失的東西——可能是工具、護欄或者文件——然後把它補充回來。

那具體怎麼做呢？首先是工具化而不是指令化。當 Agent 需要某項能力的時候，工程師的任務不是自己寫一個工具，而是讓 Agent 自己去生成那個工具。OpenAI 團隊就是這樣做的，他們讓 Codex 來撰寫 Codex 所需要的命令行工具、MCP 伺服器，還有調試腳本。工程師的工作變成了設定邊界和驗收標準，而不是具體去實作每一個功能。

其次是環境隔離和按需啟動。每個 Git Worktree 都對應一個獨立的開發環境，包含完整的資料庫實例、服務容器和監控堆疊。Stripe 的做法更進一步，他們為旗下的 AI Agent 群組準備了預先啟動好的隔離容器，讓 Agent 能夠在毫秒內啟動並行任務，而且它們之間互不干擾。

第三個實踐方式是能力缺口診斷流程。當一個 Agent 連續失敗三次以上的時候，工程師就會建立一個診斷清單。要問的問題包括：Agent 是否能看到系統的狀態，比如日誌、指標還有用戶界面？Agent 是否有適當的工具可以操作環境？Agent 是否清楚地知道任務的驗收標準是什麼？每一個找到的缺口，都代表環境需要改善的地方。

第四個是刻意選擇「無聊技術」。OpenAI 團隊有意識地選擇那些穩定且有大量訓練數據的技術，這樣 Codex 能更好地理解和運用它們。現在技術棧的選擇越來越受到「AI 友好性」的影響。

讓我們看一個真實的案例。Stripe 為他們的 AI Agent，也就是他們稱為 Minions 的系統，開放了超過四百個內部工具，這些工具通過 MCP 伺服器提供統一的介面。工程師在 Slack 頻道中貼出任務描述，Minions 就會自動領取、執行，最後提交代碼審查請求。這套系統每週產出超過一千個合併的代碼請求。

這裡最關鍵的一點是，Stripe 並沒有為 Agent 開發什麼特殊的工具集。相反，他們讓 Agent 使用的工具和人類工程師完全一樣，只是透過標準化的命令行介面和 MCP 伺服器把它們暴露出來。正因為環境的一致性，大幅降低了 Agent 的學習成本和出錯的機率。

我們來談第二個原則，叫做「機械式強制架構」。

傳統的架構管理是怎麼做的呢？就是靠人工審查代碼時做判斷。但在 AI Agent 時代，一天可能要產出幾十個拉取請求，光靠人工根本應付不了。Harness Engineering 的解法其實很簡單，就是一句話：不要告訴 Agent 要有好品味，反而要讓壞品味變得不可能發生。

這背後的理念是什麼呢？與其期待 Agent 正確地理解規則，不如把規則變成一道物理障礙。如果一個規則無法被機械式強制執行，Agent 遲早會偏離它。

OpenAI 團隊就是這樣做的。他們為整個代碼庫定義了一個固定的六層架構。這六層分別是：Types、Config、Repo、Service、Runtime，最後是 UI。代碼只能向前依賴，不允許逆向引用。

Types 層放的是所有共享的型別定義，它不依賴任何其他層。Config 層是設定管理，只能用 Types。Repo 層是資料存取，它可以依賴 Types 和 Config。Service 層是業務邏輯，可以依賴 Repo 及以下層。Runtime 層負責執行期管理，可以依賴 Service 及以下層。UI 層在最上面，可以用所有層，但通常只依賴 Service。

任何違反這個依賴方向的代碼都會被自動檢查工具攔截，無法通過自動化流程。如果要跨層共用一些東西，比如認證、連接器、觀測數據或功能開關，唯一的做法就是透過一個叫做 Providers 的統一介面注入，其他方式一律禁止。

OpenAI 用的是一種混合式強制機制。一方面，他們用 LLM Agent 來理解語意、給出高層次的設計建議。但另一方面，他們用決定性工具，也就是自動檢查和結構測試，來機械式地強制執行規則。這個組合的妙處在於：LLM Agent 可能被說服或誤導，但決定性工具不會。

說到檢查工具，OpenAI 的做法也很有意思。傳統的檢查工具只會告訴你哪裡出錯了，但 OpenAI 自訂的檢查工具會同時提供完整的修正指引。檢查工具的錯誤訊息本身就是上下文，讓 Agent 下一次嘗試時能直接學習並修正。工具本身就在教導 Agent。

除了檢查工具，他們還寫了靜態分析測試來驗證整個代碼庫的依賴圖是不是合法的。這些測試不關心業務邏輯，只關心架構結構本身。如果某次合併破壞了分層原則，即使所有功能測試都通過，這些結構測試也會擋下來。

但強制執行不只是關於架構結構。OpenAI 還把一系列主觀的代碼品質標準編碼成可以機械驗證的規則。比如，結構化日誌不允許用 console.log，所有日誌必須用結構化格式。命名慣例有約定，Schema 和 Type 的名字必須符合。純函數不能包含輸入輸出操作。單一文件超過五百行就被當成需要重構的信號。在系統邊界必須用專門工具進行資料驗證。核心業務邏輯必須達到百分之九十的測試覆蓋率。

這裡面有個有趣的做法叫「邊解析邊驗證」。就是在系統邊界用專門工具直接解析數據，而不是手動逐個欄位去驗證。好處是，一旦解析成功，數據就完全是型別安全的，不會有遺漏。

最後我想分享一個真實案例。有個叫 Ghostty 的開源終端模擬器，它的社群維護了一份詳細的文件，逐條記錄了歷史上每一次 Agent 犯過的錯誤，以及對應的修正規則。這份文件本身就是機械式強制架構的一部分。它把過去的失敗轉化成未來的保護機制，讓後續的 Agent 不再重蹈覆轍。

接下來是第三個原則，讓版本庫成為唯一真相來源。

Agent 的知識來源只有它能讀取的文字。存在於 Slack 對話、Google Docs、口頭討論中的知識，對 Agent 而言等同於不存在。Harness Engineering 要求工程師將所有決策、設計原則、架構規範都以版本控制的形式儲存在代碼庫本身。

這不僅是為了讓 Agent 能夠存取，更重要的是版本控制讓知識的變更有跡可循，讓過時的規則能夠被識別並更新。

OpenAI 的 AGENTS.md 只有約一百行，其功能是導航地圖，告訴 Agent 各類資訊在哪裡可以找到，而不是試圖把所有資訊塞入一個巨大的指令文件。這種漸進式揭露的模式有其根本原因，因為 Context 是稀缺資源。一份龐大的指令文件會擠壓任務描述、程式碼和相關文件的空間。

OpenAI 團隊創建了一種叫做 ExecPlans 的做法，存放在 PLANS.md 的設計文件，詳細到一個初學者讀完就能完整實作這個功能的程度。這不是傳統意義的技術文件，而是一種可執行的意圖聲明，工程師用自然語言清晰描述目標，Agent 則負責將其轉化為代碼。

所有設計決策、API 規範、功能需求文件都存放在代碼庫的 docs 目錄中，且受到版本控制。文件變更需要透過 PR 流程審核，確保每次變更都有記錄和審核。Agent 在執行任務前，會讀取相關的 docs 文件作為其上下文基礎。設計規格書是 Agent 行動的唯一授權來源，而不是 Prompt 中臨時撰寫的指令。

每個大型功能的開發計劃以 Markdown 文件形式儲存在 docs/tasks 中，包含任務分解、驗收標準、已知風險、歷史決策記錄。Agent 在執行子任務時，會更新計劃文件以反映進度，形成持續的文件即代碼工作流。

ARCHITECTURE.md 被設計為一張代碼地圖，而不是詳盡的操作手冊。它提供邊界定義、明確聲明這裡不存在什麼、較少變動的架構概覽。這種負空間的定義，明確告知 Agent 哪些方向不應該走，比冗長的規定性文件更能有效約束實作。

OpenAI 更進一步，部署了專門的背景任務，定期掃描 docs 目錄，識別過時或與實際代碼不符的文件，並自動提出更新的 PR 供人工審核。文件的新鮮度被視為系統健康度的一部分指標。

Cloudflare 的工程師分享了一個關鍵洞見，在讓 Agent 開始寫代碼之前，花更多時間制定詳盡的計劃文件，能夠大幅降低後續的修正成本。詳細的計劃讓 Agent 保持在正確的架構路徑上，同時也減少了 Token 消耗，因為 Agent 不需要在執行過程中不斷猜測意圖。

這個策略的本質正是讓版本庫成為唯一真相來源的體現，在代碼產生之前，真相就已經存在於版本庫中了。

我們來談談第四個原則，也就是把可觀察性連接到 Agent 身上。

首先要理解的是，人類工程師在調試的時候會做什麼呢？開瀏覽器的開發者工具、看日誌終端、看指標儀表板。AI Agent 也需要這些能力，只不過它需要的不是漂亮的視覺化介面，而是機器能讀懂的介面。

在 Harness Engineering 這個領域裡，可觀察性的作用就是讓那些模糊的驗收標準變成具體的、能量測、能驗證的目標。比如說「啟動時間要快」這個要求對 Agent 來說毫無意義，但如果你告訴它「啟動時間要控制在 800 毫秒以內」，而且 Agent 能查詢到這個數據，那就變成了一個它能理解並執行的目標。

實際怎麼做呢？首先，OpenAI 為每個 Agent 工作樹配置了獨立的可觀察性堆疊。也就是說，每個並行運作的 Agent 都有自己獨立的日誌系統、指標系統、還有分散式追蹤。Agent 可以直接執行查詢來了解系統現在的狀態，完全不需要人工來幫它解讀日誌。

其次，OpenAI 把 Chrome DevTools Protocol 整合進了 Agent 的工具集裡。這讓 Agent 能夠截取網頁的 DOM 快照、監聽網路請求和回應、測量頁面效能、模擬使用者操作並驗證結果。以前「UI 看起來是否正確」這種事情需要人眼來判斷，現在 Agent 可以機械化地驗證了。

再來是量化的驗收標準。所有任務的驗收標準都必須用數字來定義。你不能說「頁面要快速載入」，要說「首次內容繪製要小於 1.2 秒，最大內容繪製要小於 2.5 秒」。你不能說「不能有錯誤」，要說「過去 24 小時內錯誤級別的日誌必須是零」。你不能說「API 要穩定」，要說「99 百分位數延遲要小於 200 毫秒，成功率要大於 99.9 百分比」。Agent 完成任務後會自動執行這些查詢來驗證自己的工作，根本不需要等人工的品質檢查。

還有一個重要的是可觀察性驅動的 Bug 重現。傳統上重現 Bug 靠的是工程師的記憶和直覺。但在 Harness Engineering 的模式下，Bug 報告必須附帶足夠的可觀察性數據，讓 Agent 能在隔離環境中精確重現問題，不用靠猜測。

來看個真實案例。OpenAI 的工程師給 Codex Agent 設了一個任務，就是把應用程式啟動時間控制在 800 毫秒以內。這個目標之所以能被 Agent 執行，是因為工程師事先建立了自動化的啟動時間量測腳本、PromQL 查詢模板、效能回退警報規則。Agent 在每次代碼修改後自動執行量測，根據數據判斷優化是否有效，形成完整的可觀察性驅動開發循環。整個過程完全不需要人工介入。

在高速生成代碼的環境下，代碼庫的品質會隨著生成速度加快而衰退。如果我們不主動對抗這個趨勢，AI 生成的代碼就會慢慢累積大量「AI 糟粕」，什麼是 AI 糟粕呢？就是那些冗餘代碼、命名不一致、過時的文件，還有一些微妙的架構漂移。

OpenAI 在實驗初期發現了一個有趣的現象，工程師每週五都要花大概百分之二十的工作時間，也就是半天左右，手動清理 AI 產生的低品質代碼。這個發現催生了第五個核心原則，那就是品質維護本身也必須被自動化。只有這樣，清理能力才能夠跟得上生成能力。

OpenAI 對技術債的態度很明確，他們把它看作高利貸，幾乎總是值得立即償還的。為什麼呢？因為在 AI Agent 大規模生成程式碼的環境下，如果一個壞的模式出現在程式碼庫中，Agent 就會複製它，讓問題以指數速度擴散。

那他們怎麼具體實踐呢？首先，他們把代碼品質標準直接寫進版本庫，形成一套可以機械執行的黃金原則集合。這包括命名慣例、副作用隔離規則、模組大小限制、還有測試覆蓋率的最低門檻。

其次，OpenAI 建立了一套定期自動執行的 Agent 任務，有點像程式語言裡的垃圾回收機制。這些垃圾回收 Agent 會定時啟動，進行文件一致性檢查、架構違規掃描、識別死代碼、檢測命名慣例的偏差。那些低風險的修復會自動合併，而高風險的修復就交由人工審查。

在持續整合和持續部署這個環節，他們把它當作熵的防火牆。每個 PR 都必須通過一套熵檢測關卡才能合併，包括 Linter 規則、架構測試、測試覆蓋率門檻，還有文件更新驗證。

最後，他們重新設計了 PR 審查流程，讓它適應高吞吐量的需求。傳統軟體開發的 PR 審查流程是為人類設計的，假設 PR 很昂貴，需要詳細審查。但在 Harness Engineering 中，PR 是廉價的，修正也是廉價的，真正昂貴的是阻塞。所以 OpenAI 採用了最小化阻塞的合併流程，特點是 PR 生命週期很短、基礎品質門檻是自動化的、人工審查聚焦在高層次的問題上。

自動化之前是什麼樣子呢？每週五花百分之二十的人工時間清理、手動發現問題、只清理明顯的問題、每週進行一次。自動化之後呢？清理工作量近乎為零、按計劃定期執行、系統性地掃描整個代碼庫、持續在背景執行。這個差異是非常顯著的。

在我們來看 Birgitta Böckeler 提出的三大核心組成框架之前，我得先說五大原則只是告訴我們怎麼做，但這個框架提供了一個更高層次的結構性視角。

首先說 Context Engineering，也就是情境工程。你可以想像它是一個持續精煉的知識庫，裡面包含了靜態情境，比如設計規格、架構地圖、還有那些架構決策記錄，我們叫做 ADR。同時它還包含動態情境，像是可觀測性的數據、以及 CDP 瀏覽器的實時狀態。整個倉庫是針對 AI 可讀性優化的，這樣 Agent 就能從中直接推理出業務邏輯。這裡有個很關鍵的想法，當 Agent 卡住的時候，我們不應該「更努力地寫 prompt」，而是應該反問自己，到底缺少什麼能力？怎樣才能讓它對 Agent 既清晰又可強制執行？

再來是 Architectural Constraints，也就是架構約束。這層用的是一個雙重執行機制。LLM based agents 負責做高層次的設計判斷，決定事情怎麼做，但決定性的 linter 和結構性測試負責做機械性的規則驗證。這兩者一起合作，確保在 Agent 長時間自主開發之後，代碼品質不會下降。

第三個是 Garbage Collection，也就是垃圾回收。這是週期性的 Agent 程序，它會主動識別文件不一致、架構違規，還有對那些「黃金原則」的偏離。垃圾回收的目標是把人類的「品味」捕捉一次，然後在每一行程式碼上持續地強制執行。

這三個核心的作用方式是循環強化的。首先 Context Engineering 提供知識和動態情境，讓 Agent 理解應該做什麼。然後 Architectural Constraints 機械式地強制怎麼做，防止架構偏離，當出現錯誤的時候錯誤訊息會教導 Agent。接著 Garbage Collection 定期清理已經做錯的東西，並且把這些發現回饋到 Context Engineering，更新文件，強化 Architectural Constraints 增加新的規則。這個循環讓系統具備自我修復的能力，完全不需要人工介入就能持續改進。

對比一下傳統的技術債管理方式。傳統的做法是週期性的清理衝刺，手動識別問題，技術債積累好幾週，而且很依賴人力記憶品味標準，清理工作還會佔用開發時間。但 Garbage Collection 就不一樣了，它是每日自動掃描，Agent 自動偵測，問題在數小時內就被處理，品味的規則是機械式強制的，不會變，幾乎不會佔用工程師的時間。

現在我們從另一個角度，從「模型對上基礎設施」的視角重新思考，一個 Agent Harness 包含五個關鍵層。第一是情境管理，也就是控制進入模型注意力窗口的內容。第二是工具選擇，也就是設計 Agent 可以調用的能力集合。第三是錯誤恢復，處理失敗和重試邏輯。第四是狀態管理，跨會話的持久化進度。第五是外部記憶，在上下文窗口之外儲存資訊。

這邊有個很核心的洞察，當模型達到一定能力門檻之後，真正決定 Agent 可靠性的第一因素就是 Harness 的設計。我用個比喻來說，模型就像是汽車的引擎，Harness 就像是整臺車。一臺好引擎放在設計糟糕的車輛中，表現遠不如一臺普通引擎在精良設計的車輛中。

我們來看工程師角色在這波變化中的轉變。

Harness Engineering 重新定義了工程師的三大核心工作。首先是設計環境，也就是建立 Agent 能夠可靠運作的基礎設施。其次是指定意圖，把那些模糊的需求轉化為可以量測的驗收標準。最後是建立回饋循環，確保整個系統能夠持續自我校正。

這個趨勢正在從根本上改變軟體工程師的職業道路，有兩個截然不同的方向在形成。

一個方向是環境構建者。這類工程師專注於設計讓 Agent 成功的結構性條件，包括架構、工具鏈、文件系統。這需要深厚的系統設計能力，還要培養一種直覺，理解什麼樣的環境讓 AI 更加有效。

另一個方向是工作管理者。他們擔當 Agent 輸出的協調者和品質守門員。PSPDFKit 的創辦人 Peter Steinberger 就採用了這種方式，作為一個架構守門員，根據設計原則來接受或拒絕 Agent 的工作，而不是逐行審查程式碼。

這兩個方向同時發生著互動。Agent 的失敗不斷為環境改進提供輸入，而更好的環境則讓工作管理變得更加有效。前 HashiCorp 聯合創辦人 Mitchell Hashimoto 有句話總結得很到位：每當你發現 Agent 犯了一個錯誤，與其馬上去修，不如花時間設計一個解決方案，讓 Agent 永遠不會再犯同樣的錯誤。

在這個新世界中，最有價值的工程技能包括幾個方面。首先是系統設計直覺，要能夠設計出讓 Agent 運作順暢的環境。其次是意圖表達能力，怎樣把需求清楚地傳達給 Agent。再來是反思式偵錯，能夠發現問題的根本原因而不只是表面症狀。最後是品質判斷力，知道什麼樣的工作是可以接受的。

傳統開發模式和 Harness Engineering 之間的差異是全面性的。在傳統工程裡，工程師的核心工作就是撰寫程式碼。架構管理靠人工程式碼審查，知識管理用 Slack 和各種文件還有口頭溝通，品質保障靠人工 QA，程式碼清理靠計劃性的重構，所以 PR 的速度大約是每個人每週五到十個。

而在 Harness Engineering 的世界裡，核心工作變成了設計 Agent 環境。架構管理交給 Linter 加上結構化的自動化測試。知識管理利用版本庫讓機器能夠讀取。品質保障靠可觀察性驅動的自動驗證。程式碼清理變成持續的背景垃圾回收。最顯著的是 PR 的速度，現在達到了每個人每天三點五個。

現在我們來看看有哪些公司和開源項目已經在這麼做了。

Stripe 建立了超過四百個內部工具，通過 MCP 伺服器暴露給 Agent，這些工具運行在預熱的開發環境中。他們每週會合併超過一千個由 Agent 提交的 PR。

OpenClaw 是一個獨立開發者的項目。一個單一開發者通過同時運行五到十個並行的 Agent，每個月就能提交超過六千六百次的 commit。

Vercel 發現了一些反直覺的東西。他們把專門工具從十五個減少到只有兩個，結果準確率反而從百分之八十提升到百分之百。Token 的消耗減少了百分之三十七，執行速度提升了三點五倍。這印證了 Harness Engineering 中一個重要的原則，就是簡單的工具介面。在足夠強大的模型面前，過於專門化的工具反而成為瓶頸。

Cloudflare 的工程師分享了一個叫計劃優先的策略。他們發現在讓 Agent 開始寫程式碼之前，花更多時間制定詳盡的計劃文件，能夠大幅降低後續修正的成本。

最後，開源項目 Ghostty 的社群維護了詳細的 AGENTS.md 歷史記錄，把每次 Agent 犯過的錯誤轉化為未來的保護機制，這種做法完全體現了 Harness Engineering 的核心思想。

五大原則形成了一個完整的框架。原則一「設計環境」讓 Agent 能運作，原則二「機械式架構」讓壞設計不可能發生，原則三「版本庫真相」讓知識可被 Agent 存取，原則四「可觀察性連接」讓目標可被量化驗證，原則五「對抗熵增」讓品質自動維護。最終的結果就是這樣：工程師設計環境，Agent 執行實作。

對於想要開始實踐的中小型團隊，可以按照這樣的路徑來做。第一週，先建立 Context Engineering 的基礎。建立一個 AGENTS.md 檔案，一百行以內就夠了，作為整個目錄用。然後建立 docs/architecture 目錄，記錄核心架構決策。最後為常見任務建立執行模板。

第二週開始引入架構約束。定義系統的層級結構，可以是簡化版，三到四層就可以了。設定 ESLint 或 custom linter 的規則。在 CI/CD 裡加入架構合規檢查。最重要的是確保 linter 的錯誤訊息包含修復指引。

第三到四週，建立垃圾回收機制。設定每週自動化掃描，掃描文件、死程式碼和命名慣例。建立自動 PR 機制，讓低風險修復自動合併。建立技術債看板，追蹤 GC 發現的高複雜度問題。

對於更大的組織，可以考慮多個 Agent 並行運作，不同的 GC Agent 負責不同的領域，比如安全、性能和一致性。在 monorepo 或多 repo 的環境中，要統一 Harness 規則。採取漸進式的導入方式，從最容易違規的模組開始，逐步擴展約束範圍。

那什麼場景適合採用 Harness Engineering 呢？首先是新建系統，從空白的版本庫開始，最容易建立完整的 Harness 基礎設施。其次是高重複性的開發工作，像是 CRUD API、測試套件和文件生成，這些都可以高度模板化。再者是需要高速迭代的產品，當市場壓力要求快速出貨的時候，Harness Engineering 能在維持品質的前提下大幅加速開發。

當然也有挑戰和局限性。前期投入很高，OpenAI 花了整整五個月才構建好基礎設施。遺留代碼庫的問題目前仍缺乏成熟實踐。架構約束能防止結構偏離，但無法保證業務邏輯的正確性。隱性知識密集的領域難以轉化為機械可驗證的標準。Harness 本身也需要持續維護。過度依賴自動化可能造成系統性偏差。組織文化的轉型需要刻意投入。AI 生成代碼的長期技術債積累仍有待觀察。

Harness Engineering 的五大核心原則與三大核心組成，代表著一個根本性的典範轉移。工程師從代碼的作者，轉型為讓 AI Agent 能夠可靠產出代碼的系統設計師。

這個轉型的核心洞見就是：AI Agent 的效能上限不是模型本身的智識能力，而是環境的設計品質。一個設計精良的 Harness，能讓普通的 AI 模型完成驚人的成就。反過來，一個設計糟糕的環境，則會讓最強大的模型也頻繁失敗。

展望未來，Martin Fowler 等人預測，Harness Engineering 的模式將從個別團隊的實踐，演變為組織層面的標準化基礎設施。就像 Git 如今已成為連接開發環境與 CI/CD 系統的標準基礎設施一樣。可能會出現標準化的 Harness 模板，類似現代微服務模板，成為新專案的起點。

正如 OpenAI 所揭示的：在 Agent 優先的世界中，人類工程師的稀缺資源不再是寫程式的時間，而是思考、判斷和設計系統的能力。

對於正在思考如何擁抱 AI 輔助開發的工程組織而言，這五大原則與三大核心組成提供了一個清晰的行動框架。不只是導入 AI 工具，更是重新設計整個工程系統，讓 AI 成為可靠的、可信任的、自律的開發夥伴。
