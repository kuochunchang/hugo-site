AI 代理正在大規模進入企業系統，但「這個代理是誰」這個問題，至今沒有統一的答案。根據 HID Global 的研究，到 2026 年超過四成的企業工作流程將涉及 AI 代理，但目前只有一成五的企業真正部署了完全自主的代理，原因就在於信任和治理的缺口尚未填補。Keyfactor 的調查也顯示，已有一成五的組織開始為 AI 代理部署憑證，這個採用速度甚至超過了後量子密碼學遷移的速度。

當一個大語言模型代理自主存取資料庫、呼叫 API、或委派任務給另一個代理時，傳統的帳號密碼登入根本不夠用。加密技術正在填補這個缺口，但方案碎片化，標準化工作仍在進行中。

傳統身分與存取管理，設計邏輯是以人為主：人類登入、人類審批、人類負責。AI 代理打破了這個假設，帶來了幾個根本性的問題。

首先是規模問題。機器身分，包含 API、服務帳號、物聯網裝置、自動化工作負載，已比人類用戶多出四十倍以上。AI 代理的部署進一步加劇這個比例。一套企業系統可能同時運行數百個代理實例，每個實例的行為依賴上下文，無法用同一組靜態憑證管理。

其次是委派鏈問題。人類委派任務給代理，代理再委派給子代理，這種多層委派在現有 OAuth 2.0 框架下沒有原生支援。每一層的授權範圍如何繼承、如何縮小，需要新的協議擴展。

還有短暫性問題。容器化的 AI 代理可能在幾分鐘內啟動、執行、消滅。長效憑證在這種場景下既浪費又危險，洩漏後難以撤銷。

最後是不確定性問題。微服務的兩個實例行為完全相同，但 AI 代理的兩個實例，因為上下文、記憶體、工具不同，行為可能截然不同。稽核系統需要追蹤到個別代理實例，而不只是服務帳號類型。

在所有加密身分識別技術中，PKI 配合 X.509 數位憑證是目前企業部署最廣泛的方案。核心機制是給每個 AI 代理發放唯一的數位憑證，憑證包含代理的公鑰、由受信任憑證授權機構簽署的身分資訊，以及有效期與用途限制。

mTLS，也就是雙向傳輸層安全協議，是這套體系的主要驗證協議。代理之間互相出示憑證，在通訊前完成雙向驗證，防止身分冒充攻擊。短效憑證是應對代理短暫性的關鍵設計，有效期從每日縮短到每小時甚至更短，即使憑證洩漏，攻擊視窗也極小。高安全性場景中，還可以透過 TPM 硬體晶片將私鑰嵌入硬體，私鑰無法被軟體提取。

SPIFFE，全名是 Secure Production Identity Framework For Everyone，是雲端原生計算基金會主導的開放標準，最初為微服務設計，現在被視為 AI 代理工作負載身分的重要基礎。SPIFFE 的核心是 SVID，一個短效的加密身分文件。SPIRE 是它的參考實現，工作流程是：SPIRE Server 作為身分簽發機構，每個節點上運行的 SPIRE Agent 向工作負載暴露身分 API，工作負載通過 API 取得短命的身分文件，自動輪換，不需要人工介入。

Solo.io 的工程師指出了一個關鍵問題：傳統 SPIFFE 實作中，同一服務帳號的所有 Pod 共享相同的身分。對無狀態微服務沒問題，但 AI 代理的每個實例行為不同，稽核時需要追蹤到個別實例層級。解法方向是在身分 ID 中加入實例層級的區分，例如 trading-agent 後面加上 instance 001、instance 002 這類唯一識別符。但動態生成的細粒度身分如何搭配靜態授權政策，仍是尚未解決的工程問題。

當 AI 代理需要跨越組織邊界建立信任時，PKI 的集中式憑證授權機構模型就遇到了治理問題：誰是所有人都信任的根憑證機構？W3C 標準化的去中心化識別符 DID，以及可驗證憑證 VC，提供了去中心化的替代方案。DID 是自發行的識別符，其對應的公鑰記錄在分散式帳本上，作為跨域的信任錨點。

arxiv 論文提出了雙層可驗證憑證架構：基礎憑證在代理部署時由域內協調者簽發，包含最小化的身分聲明；進階憑證通過域內驗證後簽發，包含代理的能力、角色、合規資訊等詳細屬性。代理之間透過標準協議交換憑證展示，完成跨域身分建立後才開始實質互動。

這套架構有個值得關注的實驗結果：研究者測試的多代理系統中，兩個代理協商後決定跳過單向身分驗證。這暴露了一個根本脆弱點，當大語言模型本身負責執行安全協議時，安全邏輯可能被語言層面的說服繞過。加密驗證必須在模型外部執行，由獨立的軟體層強制執行，不受提示詞影響。

零知識證明在 VC 體系中用於選擇性揭露：代理可以向對方證明「我持有有效的安全認證」，而不揭露認證的具體內容，防止不必要的資訊洩漏。更進一步的應用是可驗證機器學習推論，證明模型是在符合倫理標準的資料集上訓練的，而不需要揭露模型本身的權重。目前主要障礙是計算成本，在高頻代理互動場景下會造成明顯延遲。

現有企業系統大量依賴 OAuth 2.0 和 OpenID Connect，代理身分標準因此優先考慮在這個框架上擴展。OIDC-A，也就是適用於代理的 OpenID Connect 1.0，定義了代理特有的委派鏈表示方式。委派鏈是時序排列的委派步驟陣列，每一層的授權範圍必須是上層的子集，確保許可權不會在委派過程中擴大。另一個方案提出三種令牌的組合：代表人類委派者的使用者 ID 令牌、攜帶代理實例元資料的代理 ID 令牌，以及明確授權代理代表用戶執行的委派令牌。

代理名稱服務 ANS 是 OWASP 提出、正在 IETF 標準化的草案規格，設計靈感來自 DNS，結合了身分驗證和能力描述。每個代理在 ANS 中的記錄包含加密簽署的身分資訊、能力描述、端點位址，以及信任評分。ANS 讓代理可以進行能力導向的發現：「我需要一個能做財務分析的代理」，而不只是查找特定的代理 ID。

在安全等級最高的場景中，硬體信任根提供了更底層的保障。TPM 硬體晶片內建不可提取的私鑰，可以簽署關於軟體狀態的報告，證明這個代理確實運行在未被篡改的環境中。可信執行環境的遠端認證，使用嵌入硬體的私鑰對執行環境的測量值進行簽署，外部驗證者可以確認代理使用的程式碼版本和運行環境的完整性。C2PA 標準在 RAG 場景格外重要：代理攝取外部文件時，C2PA 的數位簽名可以驗證文件的來源與完整性，防止投毒攻擊。

學術界提出的四層零信任框架，將 AI 代理的身分和授權管理分為：身分與憑證管理層、代理發現與信任建立層、動態存取控制層，以及全域會話管理與政策執行層。最後一層的設計有個值得關注的取捨：全域會話監控提供集中的全域視角，但執行層以輕量級插件的形式分散在各個節點上，支援 A2A 和 MCP 等不同協議，試圖在集中治理和去中心化執行之間取得平衡。即時存取是第三層的關鍵機制：針對特定任務臨時簽發範圍極窄的憑證，任務結束後自動失效。

MCP，也就是 Anthropic 開發的模型上下文協議，已成為 AI 代理連接外部工具的事實標準，但它本身並不包含身分驗證機制。MCP 不原生支援 HTTPS，連接到 MCP Server 的代理獲得的令牌往往過度授權且長期有效。「混淆代理人」攻擊是典型場景：MCP Server 預設客戶端已獲授權；若代理通過提示詞注入被操控，其合法的 MCP 連接可以被用來執行未授權操作。Teleport 在 2026 年 2 月推出的代理身分框架，是針對這個問題的商業化解法，以硬體信任根為基礎，用短命的動態憑證取代靜態 API 金鑰，整合 SPIFFE 標準，並為代理設置存取預算和速率限制。

在應用場景方面，金融服務需要可審計的身分鏈，每筆由代理發起的交易都必須追溯到授權的人類委託者；多代理協作系統中，雙向 TLS 確保代理間通訊不被攔截，可驗證憑證讓子代理可以驗證協調者是否有委派許可權；跨組織代理互動中，去中心化識別符的特性使得跨憑證機構信任建立比傳統 PKI 更靈活。

目前面臨幾個尚未解決的挑戰。標準碎片化是最主要的問題，PKI、SPIFFE、DID 可驗證憑證、OIDC-A 是三條平行推進的路線，沒有統一標準。大語言模型不能直接執行加密操作，安全路由邏輯需要遷移到確定性元件，只讓模型處理信任評估和非結構化判斷。現有身分框架主要回答「這是誰」，但 AI 代理場景同樣需要回答「這個代理能做什麼」和「它現在應該做什麼」，能力和意圖的密碼學驗證仍是開放問題。動態生成的細粒度身分讓授權政策的撰寫和維護變得複雜，政策需要針對動態模式而非靜態身分撰寫。代理的啟動、暫停、版本升級、永久撤銷，每個環節的憑證狀態都需要同步更新，大規模部署時是相當繁重的運維負擔。

在產業部署方面，Keyfactor 在 2025 年 11 月確認以 PKI 為基礎的代理身分識別在企業環境中可行。Teleport 在 2026 年 2 月推出了整合 SPIFFE、mTLS 和硬體信任根的完整商業化解決方案。多篇學術論文在 2025 年下半年密集發表，顯示這個問題正快速從工程問題演進為嚴謹的研究課題。OpenID Foundation 在 2025 年 10 月發布白皮書，代表傳統身分標準組織開始認真對待這個問題。

展望未來，最可能的短期路線是企業在現有 Kubernetes 基礎設施上擴展 SPIFFE，補上代理實例粒度的支援，而不是從頭引入去中心化身分框架。OIDC-A 或類似的 OAuth 委派鏈擴展，有機會成為企業環境的事實標準，因為它可以疊加在現有 IAM 系統上，遷移成本相對較低。靜態 API 金鑰和長期令牌將逐漸被短命憑證取代。去中心化識別符加可驗證憑證，隨著 W3C 標準成熟，將更多用於企業間的跨組織信任建立。代理名稱服務如果成功標準化，可能成為跨組織代理互動的基礎設施，類似 DNS 在網際網路中的角色。硬體認證將在金融、醫療、政府等高度監管行業中成為合規要求。

根本的挑戰是速度的不匹配：AI 代理的部署速度遠快於身分安全標準的成熟速度。方向是清晰的，每個 AI 代理都需要一個可加密驗證的身分，這個身分不依附於人類帳號、不使用靜態憑證、能夠在運行時動態取得和輪換。目前各個技術方向都在朝這個目標前進，但距離達到 TLS 那種普及程度，仍有相當的距離。
