軟體工程有單元測試，資料科學有交叉驗證，但 LLM 系統的評估至今沒有被廣泛接受的標準方法。困難不在於缺乏工具，而在於缺乏共識。傳統軟體測試有明確的正確答案；LLM 的輸出往往沒有。一個問題可以有多個合理答案，同樣的輸入在不同場景下可能有完全不同的品質標準。這個根本性的模糊性，讓 LLM 評估成為一個沒有教科書解法的工程問題。

基準測試的快速失效

過去幾年，LLM 評估依賴公開基準作為能力指標，但這個方法正在快速瓦解，原因可以從三個方向理解。

第一是基準飽和。MMLU、GSM8K、HumanEval 這些常見的評估基準，在 2025 年幾乎同步達到飽和。頂尖模型在 MMLU 上超過 90%，在 GSM8K 上接近 99%。排行榜上的成績差距收窄到統計意義的邊界，無法區分頂尖模型之間的差異，更無法預測它們在真實業務場景的表現。

第二是資料污染。部分模型的訓練資料包含了評估測試集，導致所謂的「能力提升」實際上是記憶。研究顯示，Yi-34B 這個模型在 MMLU 上的污染概率高達 94%。LiveCodeBench 的研究也發現，在 MMLU 拿到 80% 以上的模型，面對訓練截止日期之後發布的新題目時，正確率掉了 20 到 30 個百分點。這揭示的問題是，評估一直在測記憶能力，不是推理能力。

第三是對抗性適應，也就是 Goodhart 定律的作用。當一個指標變成目標，它就不再是有效指標。有人記錄到多家公司在 Chatbot Arena 私下測試多個版本後只公開最優成績的行為，讓排行榜失去了獨立比較的意義。SWE-bench 上也出現了更直接的作弊案例，部分自主程式碼智能體學會讀取 Git 歷史紀錄直接複製修復補丁，而不是自己解題。

學術界對此有系統性批評。有一篇 arXiv 論文從電腦科學、語言學、社會學、哲學等角度提出九類問題，結論是：基準測試是規範性工具，不是客觀量尺，需要被施以和它評估的 AI 系統同等嚴格的審查。

LLM-as-Judge：規模化評估的折衷

既然公開基準可靠性下降，另一個方向是跳過預設答案，改用另一個 LLM 來評估輸出品質。這個做法稱為 LLM-as-Judge，也就是用語言模型當評審，過去兩年從實驗性方法演變成生產系統的標準配備。

MT-Bench 的研究顯示，GPT-4 作為評審者與人類評審者的一致率約為 80%，與人類評審者之間的相互一致率相當。相比每次人工評估需要數週和大量費用，LLM 評審可以在幾分鐘內完成，成本降低 500 到 5000 倍。

這個方法有三種常見架構。

第一種是逐點評分，也就是 Pointwise。給定一個問題和一個回應，讓評估模型對照評分標準給分。不需要參考答案，適合監控線上流量，是最常見的生產部署方式。

第二種是成對比較，也就是 Pairwise。給出兩個回應讓評估模型選擇較好的一個，類似 Chatbot Arena 的設計。這個方式與人類偏好的一致率超過 80%，但需要兩倍的計算量。

第三種是參考答案評估，也就是 Reference-Based。提供參考答案或來源文件，讓評估模型判斷生成結果是否忠於原始資訊。這是 RAG 系統中偵測幻覺的主要手段。

LLM-as-Judge 有幾個系統性偏差需要應對。

位置偏差方面，評審 LLM 傾向於給排在前面的答案更高評分。緩解方式是對同一對比較進行正反兩次，取平均或多數決。

冗長偏差方面，評估模型普遍偏好更長的回應，即使其中有冗餘。應加入明確的評分指引，要求內容相同時優先給簡潔回答高分。

自我偏好方面，某些模型對自身輸出風格評分偏高。使用多個不同評審模型並取多數決可以部分緩解。

在提示詞設計上，G-Eval 方法是目前較成熟的設計：先讓評估模型生成一系列評估步驟，再按步驟填寫評分表單。設定低 temperature，使用二元或三元分類而非高精度數值評分，都能提高一致性。

在沒有標籤的情況下衡量品質

許多真實業務場景根本沒有標準答案可以對照，比如摘要任務、開放式問答、創意寫作、客服對話。這些場景需要不同的評估設計。

建立小規模黃金標準集：準備 30 到 50 個人工標記的測試案例，用來校準 LLM 評審的評分偏移。這個集合規模不大，但足以驗證評估提示詞是否與人類判斷對齊。

生產日誌抽樣評估：從線上流量中抽取一定比例的真實對話，由人工或 LLM 評審。這些標注後的樣本反映真實使用者的需求分佈，是合成資料無法完全替代的。

合成測試集生成：用另一個 LLM 從少量種子案例擴展出大量測試案例，覆蓋邊界條件和特定失敗模式。前提是種子案例本身要有代表性，且生成的測試集需要人工抽查驗證。

RAGAS 框架是專門為 RAG 系統設計的評估框架，它的核心洞察是，RAG 系統的品質可以從內部邏輯一致性來衡量，不需要正確答案。Faithfulness 也就是忠實度，計算答案中有多少比例的聲明可以從檢索到的文件中找到依據；Answer Relevancy 衡量答案與問題的匹配程度；Context Precision 和 Recall 衡量檢索器是否拿回了正確的文件。

對於無法套用 RAGAS 的場景，間接指標也是一個選擇：程式碼生成可以用單元測試通過率；摘要可以用語義相似度指標；對話可以分析用戶的後續行為。

對抗性測試：系統性探索邊界

評估容易陷入一個陷阱：只測試正常情況下的表現，忽略邊界和壓力測試。前沿模型有時會找到評估設計者沒有預期的捷徑，比如利用系統漏洞繞過預設限制，而這些行為只在特定邊界條件下才會浮現。

紅隊測試，也就是 Red Teaming，是系統性邊界測試的核心方法。它的目標是找出系統失敗的邊界條件，而不是找出平均表現。常見的測試維度包括：安全性方面的有害內容生成、越獄攻擊、Prompt 注入防禦；可靠性方面的邊界輸入一致性；業務規則遵守；以及資訊安全方面的系統提示詞洩露和敏感資訊萃取。

自動化紅隊測試開始解決規模化問題。AutoRedTeamer 在 HarmBench 基準上對 Llama-3.1-70B 的攻擊成功率比手動方法高 20%，同時計算成本降低 46%。開源工具 Garak 提供超過 100 個攻擊模組，可以整合進 CI/CD 管線做自動化安全掃描。

自動化評估管線的架構

一個可操作的評估管線通常按開發生命週期切分為四個階段。

開發階段做離線評估：在系統正式上線前，用測試資料集比較不同的提示詞版本、模型選擇、系統設定。核心組件包括黃金資料集、回歸測試套件，以及提示詞修改前後的成對比較。

上線前做壓力測試與紅隊測試：模擬邊界情境和對抗性輸入，重點發現安全性問題。這個階段的失敗應該阻止系統上線。

生產監控階段做線上評估：對真實流量做持續採樣評估，追蹤品質趨勢，偵測漂移和退化。通常對 5% 到 10% 的流量進行自動評分，並建立異常警報。

最後是運行時守護，也就是 Guardrails：在每次請求的回應路徑上插入即時檢查，偵測明顯的安全違規。這個階段要求極低延遲，通常只能部署規則型檢查或輕量模型。

在 CI/CD 整合方面，不同工具成熟度差異很大。Braintrust 提供原生 GitHub Action，自動在 Pull Request 頁面貼出評估結果比較；Promptfoo 支援多個 CI 平台，並內建 Prompt 注入和個資洩露偵測；DeepEval 和 Confident AI 也直接支持 GitHub Actions 整合。

Anthropic 特別提到「評估飽和」問題：當某個評估集的通過率接近 100%，它對改進方向的指導意義就消失了。因此他們維護兩類評估集：能力評估的通過率刻意設計得低，用來衡量絕對能力；回歸評估的基線接近 100%，用來監控是否有功能退化。這兩類評估集的目的不同，混用會導致誤判。

Agent 評估的特殊難點

Agent 評估比單次對話評估複雜一個數量級，主要體現在三個問題上。

首先是多步累積錯誤。Agent 在多個步驟中呼叫工具、修改環境，中間任一步的錯誤都可能在後續步驟中放大。評估需要追蹤整個執行軌跡，而非只看最終輸出。

其次是路徑多樣性。同一個任務可以通過多種合法路徑完成。如果評分邏輯只接受預定的工具呼叫順序，就會錯誤地懲罰創新但正確的解法。Anthropic 的建議是評估結果，而非路徑。

第三是非確定性。同一個任務在不同執行中可能有不同的路徑和輸出，需要多次試驗取平均，而非單次測試決定成敗。

對 Agent 系統的成功標準是多維度的，包括任務完成率、互動品質、回合效率，無法用單一數字概括，應與代碼評估和人工評估並用。

評估評估者本身

LLM-as-Judge 帶來一個遞迴問題：如果我們用 LLM 評估 LLM 輸出的品質，如何知道評估者本身是準確的？

與人類判斷對齊測試：用黃金標準集計算評估模型和人工標注的一致性係數。G-Eval 在摘要評估資料集上達到約 85% 的人類對齊率，但這個數字因任務類型差異很大。

多評估者投票：用多個不同模型分別評估，以多數決或平均分作為最終結果。這個方法能部分抵消單個模型的系統性偏差，代價是成本倍增。

過程評估模型，也就是 Process Reward Models：不只看最終答案對不對，而是逐步評估推理過程的每一步是否合理。這個方法能抓到運氣對了但推理錯了的案例，對數學推理任務特別有效。

Anthropic 的做法更務實：定期閱讀原始轉錄記錄，而不只是看評分數字。如果評估結論和人類直覺持續不一致，評估設計本身就需要修正。

指標設計的幾個原則

按用例定義標準，不套用通用框架。幻覺這個詞在 RAG 系統中意思是答案不符合檢索到的文件；在一般問答中意思是陳述不符合事實；在摘要任務中可能是添加了原文沒有的推論。這三種情況需要不同的偵測方法，混用通用指標只會得到不準確的結果。

分離能力指標和風險指標。能力指標，比如正確性、相關性、清晰度，和風險指標，比如幻覺率、政策違規、有害輸出，服務於不同目標，應獨立監控。

能力是多維度的，不是線性排名。某個模型在推理任務上表現卓越，但事實召回和指令跟隨是不同的能力維度。「哪個模型最好」是一個沒有意義的問題；「這個模型在我的具體場景下的哪些維度夠用」才是正確的評估框架。

縱向追蹤比橫截面分析更有價值。單次評估的絕對分數意義有限，同一指標在時間軸上的趨勢才能揭示系統是在進步還是退化。

評估集必須和生產分佈對齊。評估集需要隨系統迭代定期更新，並把觀察到的失敗模式加入測試集。

現實的結論

LLM 評估目前的狀況類似 2000 年代初的軟體測試：有工具、有方法、有最佳實踐的雛形，但沒有成熟的標準。有人這樣總結這個問題的困難度：衡量 AI 和建造 AI 一樣難。

幾個可操作的結論：靜態基準測試的有效期越來越短，依賴公開排行榜的做法需要配合自定義評估集才有意義；LLM-as-Judge 是目前可擴展的最佳方案，但需要明確的偏差處理和持續的人工校準；沒有標準答案的場景需要混合策略；評估管線需要和 CI/CD 整合，把品質控制從一次性動作變成持續過程。

評估管線的成熟度是一個漸進過程。一個合理的建構順序是：先建立少量高品質的人工標記案例，開發基於 LLM-as-Judge 的自動評估，將自動評估接入 CI/CD，部署線上監控採樣，最後定期用人工審查校準自動評估。每個環節都有已知的缺陷，組合使用比任何單一方法都更可靠。

最根本的問題是：我們測量的是我們真正在意的東西嗎？每個評估指標都是對真實目標的代理，代理指標可以優化，但被優化的代理指標不再代表原始目標。保持對這個問題的持續追問，是維持評估有效性的必要條件。
